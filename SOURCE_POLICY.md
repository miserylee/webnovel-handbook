# 来源与版权政策

本仓库会长期吸收公开资料、写作经验和项目实践。开源后，来源处理必须更谨慎：我们要沉淀规则，不搬运原文；要保护私人信息，不公开未授权材料。

## 来源优先级

处理资料冲突时，按以下顺序判断：

1. 用户当前明确指令。
2. 本仓库已经确认的文档和 `docs/99-decision-log.md`。
3. 官方法律法规、主管部门政策、平台公开规则和权威标准。
4. 本仓库已经抽象化的通用写作方法与已公开可复用经验。
5. 旧项目、编辑部、读者群中已经匿名化和抽象化的经验。
6. 临时笔记、个人观点、未核实材料。

如果来源之间冲突，要写明冲突，不要强行合并。

## 可接受来源

优先使用：

- 官方网站、法律法规数据库、政府部门公开文件。
- 平台公开规则、作者帮助中心、投稿规范。
- 权威行业报告、公开访谈、研究资料。
- 已公开且允许引用的写作经验。
- 经过匿名化处理的项目复盘和读者反馈。

谨慎使用：

- 自媒体观点。
- 论坛帖子和社群讨论。
- 未标明发布时间的资料。
- 可能已经过时的平台规则和政策解读。
- 个人经验总结。

不接受：

- 未授权的付费课程原文。
- 未公开稿件、编辑内部意见原文、匿名读者反馈原文。
- 含真实姓名、联系方式、账号、群名、截图等可识别信息的材料。
- 大段复制文章、书籍、规则或课程内容。
- 诱导违法、规避监管、侵犯隐私或伤害他人的资料。

## 写入方式

资料写入仓库时，不要只粘贴原文。优先整理为：

- 原则：解决什么问题。
- 用法：写作或审稿时怎么执行。
- 风险：违反后会怎样影响读者体验或安全边界。
- 检查问题：AI agent 如何自查。
- 案例：抽象化、匿名化、可复用。
- 来源：链接、标题、发布日期或访问日期。

## 来源登记

新增稳定来源时，必须更新 `docs/01-source-inventory.md`，至少写明：

- 来源标题。
- 来源链接或出处。
- 使用目的。
- 吸收方式。
- 使用限制。

不要把来源清单变成链接仓库。每条来源都要说明它如何帮助小说创作或 agent 工作流。

## 版权边界

- 不大段复制受版权保护文本。
- 不复制付费课程、内部资料或未授权稿件。
- 引用短句时必须服务于说明，并尽量改为概括和转述。
- 对平台规则、政策文件和法律法规，也优先做摘要、流程化和写作化处理。
- 如果来源不允许再分发，不要写入仓库。

## 隐私边界
## 去身份化要求

私有来源进入公开仓库前，必须先按 `DEIDENTIFICATION.md` 处理。旧项目、编辑反馈、读者反馈和作者实践只能使用匿名标签；不得保留真实项目名、本机路径、群名、账号、昵称、私有工具链或可拼接识别的来源结构。


读者群、编辑部、旧项目和真实案例必须匿名化：

- 删除姓名、昵称、账号、联系方式、群名、公司名、学校名等可识别信息。
- 不提交截图原图。
- 不保留完整聊天记录。
- 只提炼可迁移的读者体感、审稿标准、流程经验和风险判断。

## 时效性

涉及法律、平台规则、模型能力、商业政策、金融、医疗、教育、未成年人、公共安全等内容时，必须注意时效性。后续 agent 使用这些资料前，应按任务风险重新核对最新来源。

本仓库提供写作参考，不构成法律、医疗、金融、平台运营或商业成功保证。
