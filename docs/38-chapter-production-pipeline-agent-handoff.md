# 38. 章节生产流水线与多 agent 交接规范

写手、试读、审稿与改稿一体化入口：

- 如果用户希望写稿时自动完成内部试读、商业审稿、反馈仲裁和改稿，而不是在读者群、编辑部、写手项目之间手动切换，先看 `docs/58-integrated-drafting-beta-review-revision-workflow.md`。该专题把本流水线中的“正文起草 -> 风格与连续性初检 -> 商业审稿 -> 修订执行”扩展为“起草 agent -> 试读 subagents -> 审稿 subagents -> 反馈仲裁 -> 修订 agent -> 二次轻检 -> 台账回写”。

长篇时间跳跃与状态压缩入口：

- 如果需要写“几天后、几个月后、几年后”、压缩训练/赶路/调查/备赛/项目推进/伤后恢复，或卷间存在未写时间，先看 `docs/57-long-timeskip-state-compression.md`。该专题负责判断哪些内容能跳、哪些必须写成场景，并用状态差额、关系债、伏笔回响、对手变化和台账回写防止追读断裂。

卷间转场与新阶段启动入口：

- 如果卷末结算完成后需要开启下一卷、新地图、新副本、新项目、新案件、新组织层级或新关系阶段，需要判断旧卷资产是否带入、旧代价是否仍影响行动、新规则是否通过冲突展示、新卷前五章是否有第一单元回报，先看 `docs/56-volume-transition-new-stage-launch.md`。该专题负责防止下一卷重新开书、换地图灌设定、旧卷资产断线和新敌人空降。

卷末结算与下一卷牵引入口：

- 如果一卷进入卷末、阶段高潮、结案、项目交付、关系质变或下一卷切换，需要判断本卷目标是否回答、支线和伏笔是否清账、余波是否落地、下一卷目标是否从本卷结果自然生长，先看 `docs/55-volume-ending-settlement-next-hook.md`。该专题负责防止卷末只爆不结算、悬崖替债、余波太软和下一卷空降。

卷级支线与多线程管理入口：

- 如果 30 章后人物、支线、伏笔、关系线、悬念线、机制线或世界线同时推进，需要判断哪些线该开、该并入、该回收、该延期或该删除，先看 `docs/54-volume-subplot-thread-management.md`。该专题负责把支线从“有趣插曲”变成有功能、交叉点、回报、退出点和卷末清账的可管理线程。

30 章后第一卷推进入口：

- 如果前 30 章已经完成留存闭环，需要设计第 31 章以后、判断第 30 章结果能否转成卷目标、小单元链是否有回报、卷中转折是否改变推进方式、支线是否服务卷目标、卷末是否能结算并牵引下一卷，先看 `docs/53-after-30-first-volume-structure.md`。该专题负责把 30 章后从“继续发生事件”切换成“卷级故事”。

第 21-30 章稳定追读入口：

- 如果第 16-20 章已经完成小高潮或入轨确认，需要设计第 21-30 章、判断第 20 章结果能否转成十章目标、第 25 章是否有单元小结、第 26-30 章是否形成阶段高潮、第 30 章是否完成前 30 章闭环并交接 30 章后卷级推进，先看 `docs/52-chapters-21-30-stable-retention-stage-climax.md`。该专题负责防止第 21 章重新开书、稳定水、只堆危机和第 30 章只爆不交接。

第 16-20 章第三轮推进入口：

- 如果第 11-15 章已经完成第二轮目标和小回报，需要设计第 16-20 章、判断第 15 章钩子能否转成第三轮目标、循环引擎是否开始运转、第 18 章是否形成压迫升级、第 20 章是否有小高潮或入轨确认并牵引第 21-30 章，先看 `docs/51-chapters-16-20-third-cycle-mini-climax.md`。该专题负责防止空降任务、纯设定章和只赢不入轨。

第 11-15 章第二轮展开入口：

- 如果第 9-10 章已经完成第一轮落袋，需要设计第 11-15 章、判断第 10 章余波能否转成第二轮目标、旧资产能否试新规则、对手是否形成新压迫点、第 15 章是否有小回报并牵引第 16-20 章，先看 `docs/50-chapters-11-15-second-cycle-expansion.md`。该专题负责防止二次开篇、重复爽点和无结果过渡章。

第 9-10 章落袋入口：

- 如果第 6-8 章已经完成换招，需要设计第 9-10 章、判断第一轮回报是否落袋、硬收益/硬代价是否成立、余波和更大问题是否能牵引第 11-15 章，先看 `docs/49-chapters-9-10-first-payoff-next-hook.md`。该专题负责把前 10 章的承诺兑现、公开结果、状态变化和下一阶段牵引做成结算闸门。

第 6-8 章换招入口：

- 如果第 4-5 章已经入轨、需要设计第 6-8 章、避免同一爽点横向重复、安排对手反制、暴露机制限制，或判断能否进入第 9-10 章落袋，先看 `docs/48-chapters-6-8-variation-escalation.md`。该专题负责让旧招受限、主角换方法、阻力升级、回报币种具体化，并设置第 8 章后闸门。


第 4-5 章入轨入口：

- 如果前三章已经完成、需要设计第 4-5 章、判断主线是否入轨、公开阶段目标、安排阻力升级，或决定能否进入第 6-8 章换招，先看 `docs/47-chapters-4-5-track-stage-goal.md`。该专题负责把前三章承诺验证转成阶段目标、行动路线、阻力升级、回报币种和前 10 章落袋方向。


前三章承诺兑现入口：

- 如果第 1 章已经完成、需要生成第 2-3 章任务、审稿前三章、处理“首章强但后续掉速”、或判断能否进入第 4-5 章入轨，先看 `docs/46-first-three-chapters-promise-fulfillment.md`。该专题负责检查第 2 章是否兑现第 1 章钩子、第 3 章是否证明卖点可重复，以及前三章是否形成前 10 章落袋方向。


第一章生产许可入口：

- 如果任务涉及第一章起草、第一章重写、`ch001.md` 是否可以开写、开篇任务单是否齐备或第一章写后回写，先看 `docs/45-first-chapter-task-brief-production-gate.md`。该专题负责检查项目许可、任务许可、输出许可、第一章输入包、任务单字段和写后回写边界。


项目初始化入口：

- 如果任务涉及通过立项后创建单本小说项目、生成目录、建立最小文档包、准备第一章任务或多 agent 接力前的项目事实来源，先看 `docs/44-single-novel-project-initialization-package.md`。该专题负责规定项目目录、必填文件、状态标签、台账、风格手册和第一章生产许可。


反馈冲突入口：

- 如果编辑反馈、读者反馈、后台数据、作者直觉或项目主承诺互相冲突，先看 `docs/42-editor-reader-data-feedback-conflict-resolution.md`。该专题负责判断来源权重、证据同向性、主承诺保护、修改成本和观察窗口，避免简单投票或盲从单一反馈。


评论语料入口：

- 如果任务涉及读者评论、章评、书评、试读反馈、匿名读者反馈或编辑反馈的整理与取舍，先看 `docs/41-reader-comment-coding-feedback-clustering.md`。该专题负责把评论原文拆成观察点、打标签、聚类、判断证据强度，并转成可观察的修改动作。


相关入口：

- 连载期数据已经出现异常、推荐期需要判断要不要改、上架前后需要设置观察窗口，先看 `docs/40-serial-data-tuning-loop.md`。

## 1. 核心原则：章节生产不是“一次生成”，而是带闸门的流水线

AI agents 写网文最容易犯的错，是把“生成正文”当成唯一任务。商业级连载实际需要多层动作：

- 判断本章在连载中的功能。
- 打包最小必要上下文。
- 设计场景目标、阻力、转折和章末牵引。
- 起草读者可见正文。
- 审稿检查承诺、节奏、人物、连续性、风格和合规。
- 修订时保护有效资产，不把文本改平。
- 更新台账、风格手册和下一章任务。

一句话原则：章节不是被 AI “写出来”的，而是从项目事实、读者承诺、场景设计、正文执行、质量闸门和台账回写中生产出来的。

## 2. 什么时候必须使用本流水线

必须使用：

- 用户要求直接写第一章，但项目资料、章节任务单或风格台账不完整。

- 批量写 2 章以上。
- 多个 agents 接力写作、审稿或修订。
- 已经有台账、风格手册或前文连续性的项目。
- 需要从大纲推进到正文。
- 用户要求“商业级”“可连载”“高质量”“降低 AI 味”。
- 审稿后要执行改稿。
- 每 5 章、卷末、推荐前、上架前或发布前质检。

可以简化：

- 只写一个短片段测试声音。
- 只做一句话设定 brainstorm。
- 只对单句进行语言解释。

简化不等于跳过来源状态。即使只写试写段，也要标明“候选”，不能写成已确认正文。
简化也不等于跳过开书准备。新项目、第一章、前三章试写必须先确认是否已有最小项目文档包；缺包时先补任务单和文档，不进入正式章节生产。


## 3. 推荐 agent 分工

多 agent 系统不需要每个角色都由独立模型承担；同一个 AI 也可以按顺序切换角色。但不能在同一步里混用职责。

| 角色 | 主要职责 | 输入 | 输出 | 禁止事项 |
| --- | --- | --- | --- | --- |
| 总控 agent | 判断阶段、调用专题、分配任务、检查门槛 | 用户指令、索引、项目状态 | 本轮任务单、必读材料、交接对象 | 不直接写正文 |
| 上下文封包 agent | 压缩项目事实、台账、规则、风格和禁区 | 项目文档、台账、专题 | 最小上下文包 | 不把整库塞给起草 agent |
| 章节设计 agent | 把大纲转成场景链 | 本章目标、上一章状态、人物/对手状态 | 场景表、章末牵引、禁止提前解决项 | 不写完整正文 |
| 起草 agent | 写读者可见正文 | 上下文包、场景表、风格约束 | 章节草稿 | 不输出自检、解释和台账复述 |
| 风格守门 agent | 检查旁白距离、人物声音、称呼术语 | 草稿、风格手册 | 风格问题和保留建议 | 不把有效粗糙改平 |
| 连续性 agent | 检查事实、时间线、人物知道什么、伏笔 | 草稿、故事台账 | 连续性问题、台账更新项 | 不做全文审美润色 |
| 审稿 agent | 判断商业效果和问题层级 | 草稿、定位、承诺、审稿标准 | 审稿报告、修改优先级 | 不边审边乱改 |
| 修订 agent | 按问题层级执行改稿 | 草稿、审稿报告、有效资产、边界 | 修订正文、变更说明 | 不擅自改已确认设定 |
| 发布/复盘 agent | 检查发布质量、更新节点、读者数据 | 定稿、发布计划、评论数据 | 发布检查、复盘记录 | 不把数据现象当病因 |

## 4. 一章的标准生产链路

```text
任务识别
-> 上下文封包
-> 章节设计
-> 正文起草
-> 风格与连续性初检
-> 商业审稿
-> 修订执行
-> 台账/风格手册回写
-> 下一章任务生成
```

每一步都必须有明确退出物。如果退出物缺失，不能进入下一步。

新项目或第一章必须先加一层前置检查：

```text
项目初始化检查 -> 第一章/本章任务单检查 -> 再进入章节生产链路
```

前置检查不通过时，流水线的输出不是正文，而是缺口清单、补资料问题、候选开场方案或章节任务单。


### 4.1 任务识别

输出：

- 当前阶段：开篇验证 / 前 10 章 / 中段 / 卷末 / 终局 / 改稿 / 发布前。
- 本章主功能：证明卖点、兑现回报、升级阻力、推进关系、回收伏笔、制造转场、卷末结算。
- 必读专题：只列本章需要的专题，不泛读全部知识库。

不通过门槛：

- 新项目没有项目简报、前 10 章方向、风格手册或台账。
- 第一章没有通过 `45` 的生产许可。

- 不知道本章服务哪个读者承诺。
- 不知道上一章留下什么未完成动作。
- 不知道本章结束后局面要改变什么。

### 4.2 上下文封包

输出：

- 本章最小事实包。
- 活跃台账摘要。
- 风格手册摘要。
- 本章硬约束和禁区。
- 输出契约。

不通过门槛：

- 候选设定和已确认事实混在一起。
- 风格样例没有标注“只学规则，不复制句子”。
- 输入材料过多，起草 agent 无法判断本章重点。

### 4.3 章节设计

输出：

```text
【本章主任务】
【开场承接】
【场景 1】目标 / 阻力 / 转折 / 微回报
【场景 2】目标 / 阻力 / 转折 / 微回报
【章末】落袋 / 后果反弹 / 下一步必须行动
【禁止】不能提前揭示、不能解决、不能改动
```

不通过门槛：

- 场景只有事件，没有目标、阻力和转折。
- 章末只有口号或预告，没有具体未完成动作。
- 连续两章使用同一种阻力和同一种回报。

### 4.4 正文起草

输出：只输出读者可见正文。

硬约束：

- 不写“本章要体现”“这里埋伏笔”“下面进入高潮”。
- 不把台账、审稿意见、作者说明混入正文。
- 不用系统、旁白或便利证据自动替主角赢。
- 每个重要人物带着自己的目的和压力进入场景。

### 4.5 风格与连续性初检

输出：

- 人物声音问题。
- 旁白距离问题。
- 称呼、术语、特殊格式问题。
- 事实、时间、人物知识、伏笔问题。
- 必须改 / 可保留 / 待确认。

不通过门槛：

- 人物台词不看名字无法区分。
- 旁白知道角色不该知道的信息。
- 本章改变事实但没有台账更新项。
- 修订会破坏已建立风格或有效粗糙。

### 4.6 商业审稿

输出：

- 总体结论。
- 有效资产。
- 风险点。
- 问题层级：承诺 / 结构 / 场景 / 人物 / 信息 / 句段 / 格式。
- 修改优先级。

审稿不是挑错比赛。审稿 agent 必须先识别什么有效，避免修订 agent 把有价值的东西删掉。

### 4.7 修订执行

输出：

- 修订后正文。
- 变更说明。
- 未采纳问题及理由。
- 台账和风格手册更新项。

不通过门槛：

- 只润句子，没有解决审稿指出的结构问题。
- 改稿引入新事实，却没有标注台账更新。
- 把人物非完美反应全部改成清楚完整的说明句。

### 4.8 回写与下一章任务

输出：

- 新增或改变的事实。
- 人物状态变化。
- 关系变化。
- 规则、资源、伤势、时间线变化。
- 伏笔埋入、激活、延后或回收。
- 风格手册更新项。
- 下一章必须承接的问题。

不通过门槛：没有回写，就不能批量生成下一章。

## 5. 交接包规范

任何 agent 交给下一个 agent 时，都必须提供交接包。

```markdown
# Agent 交接包

## 任务
- 当前阶段：
- 本轮目标：
- 下一 agent：

## 权威来源
- 用户当前指令：
- 项目已确认文档：
- 本知识库专题：
- 候选/待确认：

## 输入摘要
- 已确认事实：
- 活跃台账：
- 风格/声音约束：
- 本章任务：
- 禁止事项：

## 本 agent 产出
- 产物：
- 有效资产：
- 风险点：
- 待确认：

## 下游要求
- 必须保留：
- 必须检查：
- 必须输出：
- 不能做：
```

交接包的目标不是写得详细，而是让下游 agent 知道“什么可以信、什么要做、什么不能碰”。

## 6. 单 agent 也要分角色

如果只有一个 AI 处理全部工作，也要按顺序切换帽子：

1. 总控：判断任务和读取路线。
2. 封包：压缩上下文。
3. 设计：列场景链。
4. 起草：只写正文。
5. 审稿：只诊断，不改。
6. 修订：按诊断改。
7. 台账：只记录变化。

禁止在同一轮输出里同时写正文、夸自己、审稿、改稿和台账复述。这样看似省事，实际上最容易把正文和工作说明混在一起。

## 7. 多 agent 协作的三种模式

### 7.1 串行流水线

适合：稳定章节生产。

```text
总控 -> 封包 -> 设计 -> 起草 -> 审稿 -> 修订 -> 台账
```

优点：职责清楚，适合长篇连载。

风险：如果交接包太长，会拖慢生产；如果交接包太短，会丢事实。

### 7.2 总控加专家

适合：章节涉及专业难点。

```text
总控 agent
-> 机制专家 / 情绪线专家 / 悬疑专家 / 合规专家 / 风格专家
-> 总控合并
```

优点：能处理复杂章节。

风险：专家意见可能互相冲突，必须由总控按用户当前指令和项目已确认事实裁决。

### 7.3 审稿对照组

适合：开篇、上架前、卷末、争议改稿。

```text
起草稿 -> 审稿 agent A / 审稿 agent B -> 总控聚类 -> 修订任务卡
```

优点：能发现盲点。

风险：不能把多份意见简单相加。必须聚类成问题层级和优先级，否则会改成四不像。

## 8. 质量闸门

### 起草前闸门

- 本章目标是否清楚？
- 上章未完成动作是什么？
- 主角本章是否有主动选择？
- 对手或环境阻力是什么？
- 风格手册是否给到？
- 活跃台账是否给到？

### 起草后闸门

- 本章是否只包含读者可见正文？
- 每个场景是否有目标、阻力、转折？
- 本章是否给了至少一个回报？
- 章末是否有具体未完成动作？
- 是否出现提纲式正文、复述剧情或空洞水文？

### 修订前闸门

- 审稿是否先列有效资产？
- 问题层级是否明确？
- 修改边界是否明确？
- 已发布内容是否存在风险？

### 交付前闸门

- 台账更新项是否列出？
- 风格手册更新项是否列出？
- 待确认事项是否标注？
- 下一章任务是否能直接承接？

## 9. 常见失败模式

### 9.1 责任混乱

表现：起草 agent 顺手改设定，审稿 agent 顺手重写正文，台账 agent 顺手发表审美意见。

修法：每个 agent 只做自己的输出契约，越权内容放“建议”，不直接执行。

### 9.2 上下文接力污染

表现：上游把候选脑洞、读者猜测、竞品桥段和已确认事实混在一起，下游直接写进正文。

修法：所有交接包必须标“已确认 / 候选 / 待确认”。

### 9.3 审稿意见堆叠

表现：多个审稿 agent 都有道理，修订 agent 全部采纳，文本变成平均值。

修法：总控先聚类，再按主承诺、读者体验和已确认风格排优先级。

### 9.4 修订改平

表现：正文更通顺，但人物声音、节奏和情绪毛刺消失。

修法：修订前标出有效资产和可保留粗糙；风格守门 agent 有一票提醒权。

### 9.5 台账欠债

表现：连续几章写完才补台账，结果事实、称呼、资源和伏笔对不上。

修法：每章只回写变化，不复述整章；少而准才可持续。

## 10. 对 AI agents 的硬约束

- 不允许没有任务阶段就直接写正文。
- 不允许没有交接包就让下游 agent 接手。
- 不允许审稿 agent 在诊断前直接重写。
- 不允许修订 agent 擅自修改已确认事实、称呼、规则和风格。
- 不允许台账 agent 复述整章代替记录变化。
- 不允许多个审稿意见未经聚类就全部采纳。
- 不允许把候选资料、竞品桥段或读者猜测写成项目事实。
- 不允许为了批量生产跳过风格、连续性、台账和合规检查。

## 11. 快速自查问题

- 当前是流水线哪一步？
- 这一步的唯一输出是什么？
- 下游 agent 需要什么，哪些不需要？
- 本章失败会失败在承诺、结构、场景、人物、信息、风格、连续性还是句段？
- 有效资产是否先被保护？
- 修改是否需要同步台账、风格手册或决策记录？
- 下一章能否根据交接包直接开工？

## 长篇训练、修炼与成长段落防水入口

当章节或小单元涉及训练、修炼、闭关、学习、备赛、技能升级、职业成长、系统任务或团队磨合时，需要调用 `docs/58-training-growth-anti-padding.md`。该专题负责判断训练段是否应该展开、压缩或跳过，并要求训练段具备目标、瓶颈、方法、代价、反馈、验证和下一阶段限制，防止训练流水账、闭关后直接无敌、数值升级无证明和对手原地等待。

## 反派、竞争对手与外部世界同步升级入口

当主角胜利、训练突破、暴露新能力、夺取资源、改变规则、经过时间跳跃、进入新卷或开启新地图后，需要调用 `docs/59-antagonist-world-synchronous-escalation.md`。该专题负责判断外部世界是否同步变化：谁看见了主角变化、谁产生误判、对手换了什么策略、组织调了什么资源、规则或舆论怎样反馈、下一轮压力如何针对主角旧优势，防止世界原地等待、对手重复送、新敌人空降和升级失真。

## 高潮战、大场面与群体冲突调度入口

当章节或卷级单元涉及高潮战、决战、群战、团队战、公开审判、朝堂对峙、比赛决胜、灾难救援、发布会、直播爆点、群体谈判，或任何三条以上行动线同时推进的大场面时，需要调用 `docs/60-climax-battle-large-scene-coordination.md`。该专题负责确认目标、胜负条件、空间锚点、视角锚点、群体角色分工、关键转折、动作压缩、战后结算和台账回写，防止只堆人数、招式、爆炸、喊话和震惊。

## 信息战、舆论战与公开场合话语权攻防入口

当章节涉及证据链反杀、公开审判、庭审、朝堂对峙、寿宴摊牌、直播翻车、热搜风波、危机公关、发布会、项目验收、商战舆论、娱乐圈黑料、职场甩锅、宗门审判、弹幕评论区风向，或任何“胜负取决于谁被相信”的场景时，需要调用 `docs/61-information-public-opinion-discourse-battle.md`。该专题负责确认争点、关键受众、证据来源与边界、对手叙事、抢解释权方式、主角反制节奏、公开行动请求、落袋余波和合规风险，防止万能录音、主角独白、众人集体倒戈、热搜空转和现实舆论风险。

## 日常章、过渡章与低强度章节防水入口

当章节涉及日常、过渡、休整、赶路、聊天、吃饭、生活流、低冲突推进、战后余波、关系缓冲、情绪沉淀、信息整理、准备章、轻松章或主线暂时低烈度推进时，需要调用 `docs/62-everyday-transition-low-intensity-anti-padding.md`。该专题负责判断低强度章节是否值得独立成章，是否承接旧结果、产生微变化、交付小回报、保留人味、生成下一步并回写台账，防止日常流水账、过渡无增量、休整清零、赶路只写到了和聊天无目的。


## 与 63 号专题的衔接：交接时必须交代伏笔记忆温度

章节生产交接时，若存在未回收伏笔、旧物、旧账、关系暗线、规则例外、情绪承诺或反派后手，应调用 `docs/63-foreshadowing-callback-reader-memory-management.md`，在交接信息中补充：伏笔编号、最近回响、记忆温度、下一次提醒、禁止剧透点、阶段回收计划和最终回收计划。

下一位 agent 不应只知道“这条线以后有用”，还要知道读者现在是否还记得、下一章适合轻触还是再激活、哪些信息不能提前说破。


## 与 64 号专题的衔接：交接包要记录场景资产

章节生产中如果出现角色高光、爽点落袋、情绪峰值、关系爆点、公开反转、梗点变体或卷级名场面，应联动 `docs/64-iconic-scene-character-highlight-shareable-moment.md`，在交接包中记录读者复述句、可记忆载体、见证者行动变化、状态翻转、后文余波和可回响元素。

下一位 agent 不应只知道“上一章很燃”或“上一章打赢了”，而要知道读者可能记住哪一个动作、哪一句话、哪件物品、哪次站位变化，以及这些资产后文如何回响或被对手、配角、评论区误读利用。


## 与 65 号专题的衔接：上架节点必须进入章节流水线

当章节生产接近上架、收费、会员章节、订阅、活动爆更或付费后 3-5 章时，应联动 `docs/65-paid-conversion-subscription-retention.md`。

交接包需要额外记录：最后免费章的阶段回报和合法期待、首个付费章前 25% 要兑现什么、付费价值单位是什么、付费第 2-5 章如何承接后果并建立新阶段目标、哪些平台规则仍待确认、哪些后台数据不能编造。下一位 agent 不应只知道“这里上架”，还要知道读者为什么愿意继续付费。

## 与 69 号专题的衔接：交接包必须写视角契约

章节交接时，必须把 `docs/69-pov-narrative-distance-information-focus.md` 的视角契约纳入交接包。至少记录：当前视角人物、人称/视角类型、叙述距离、视角人物已知、读者已知、本章禁止泄露、下一次允许切视角的位置、切视角必须带来的收益。

如果交接包没有这些字段，下一位 AI 很容易把悬疑信息、角色声音和情绪连续性写散；审稿代理应将其标记为“视角交接不完整”。
