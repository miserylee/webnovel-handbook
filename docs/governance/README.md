# governance

状态：已确认  
用途：活跃决策、维护治理和历史决策归档入口。

## 先读结论

默认只读 `99-decision-log.md`。它是活跃决策短索引，用来判断当前规则是否已有确认结论。

不要默认整读 `../decisions/`。历史归档只在需要追溯旧决策原因、处理规则冲突或复盘演进时按日期和关键词检索。

## 文件分工

| 文件或目录 | 什么时候读 | 作用 |
|---|---|---|
| `99-decision-log.md` | 知识库维护、规则冲突、用户要求改变维护方式、新增关键流程前 | 查看已确认决策和新增决策写入格式 |
| `../decisions/` | 活跃索引无法解释旧规则来源，或需要查看某月历史决策详情时 | 历史决策归档，默认不整读 |
| `../../CHANGELOG.md` | 面向开源用户说明结构变化、入口变化、重要新增或移除时 | 记录可见变更，不替代决策记录 |

## 常见路线

### 新增或调整长期规则

```text
99-decision-log.md -> 必要时新建 ../decisions/YYYY-MM-*.md -> 更新相关专题/入口 -> CHANGELOG.md
```

适合：目录结构、索引策略、skill 分发、开源边界、项目文档粒度、写稿硬闸门等长期规则变化。

### 处理来源冲突或规则冲突

```text
99-decision-log.md -> 命中专题 -> ../sources/README.md -> 必要时 ../decisions/ 历史归档
```

先确认冲突来自用户最新指令、既有文档、来源材料还是旧项目经验，再按 `AGENTS.md` 的来源优先级处理。

### 判断是否需要写决策

需要写入 `99-decision-log.md` 的情况：

- 影响后续 agent 默认行为。
- 改变索引、目录、skill、开源维护或项目初始化规则。
- 用户明确纠正了长期流程。
- 同一问题可能在后续反复出现，需要稳定裁决。

不需要写决策的情况：

- 单次文档润色。
- 临时方案或未确认想法。
- 只影响某个专题内部措辞的小修。
- 已经能被现有规则覆盖的执行细节。

## 维护规则

- `99-decision-log.md` 保持短索引，不继续膨胀成长文。
- 长说明放进 `../decisions/` 的单条决策文件，再从 `99-decision-log.md` 链接。
- 决策只记录稳定结论和理由，不记录可识别个人信息、私有项目名、聊天身份、群名或本机路径。
- 如果决策改变了 agent 读取路径，必须同步更新 `README.md`、`../00-index.md`、相关目录 README 或 skill 入口中的至少一个。
