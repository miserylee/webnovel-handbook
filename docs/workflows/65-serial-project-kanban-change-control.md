# 65. 连载项目看板、变更控制与完成定义

状态：已确认
用途：给单本小说项目的总控 agent、章节设计 agent、写手 agent、审稿 agent 和台账 agent 使用，用来管理连载过程中的任务流、变更、阻塞、验收和回写。

## 1. 核心判断

连载项目不是一次性写完的瀑布项目，而是持续交付、持续反馈、持续修正的长线系统。

因此，项目管理重点不是把大纲锁死，而是让每一项工作都能回答：

- 当前为什么要做？
- 谁或哪个 agent 负责下一步？
- 完成标准是什么？
- 会改动哪些正稿事实、台账、人物、设定或后续细纲？
- 如果读者反馈或作者改稿改变方向，哪些文档要同步更新？

## 2. 从项目管理方法迁移来的规则

本篇借用 Kanban 和 Scrum 的通用项目管理思想，但不照搬软件开发术语。

可迁移部分：

- 可视化工作流：把工作拆成可移动的卡片，而不是藏在聊天记录里。
- 限制进行中任务：不要同时开太多章节、设定、修订和包装任务，否则上下文会散。
- 拉动式生产：只有当前置资料、任务单、验收标准和台账入口准备好，才进入起草。
- 完成定义：不是“写完字数”就完成，而是达到质量、连续性、索引和回写标准。
- 短周期复盘：每 3-5 章或重要节点后检查流程本身，而不只检查故事好不好看。

不可照搬部分：

- 不要把小说章节当成纯工单；情绪、人物状态和读者期待无法只靠任务列表解决。
- 不要用看板替代大纲、台账、风格手册和正稿核对。
- 不要为了流程整齐，强行压制作者临场灵感；临场变化可以进“变更候选”，再走确认链。

## 3. 推荐连载看板列

单本小说项目可以在 `<novel-project-docs>/00-index.md` 或项目管理文件里维护一个轻量看板。

推荐列：

| 列 | 放什么 | 进入条件 | 移出条件 |
|---|---|---|---|
| 候选池 | 灵感、读者反馈、编辑意见、作者临场想法 | 有一句清楚描述 | 判断是否进入近期计划 |
| 待澄清 | 方向可用但缺资料的问题 | 影响后续章节或项目方向 | 明确成任务、否决或归档 |
| 已准备 | 可执行任务卡 | 有目标、输入文件、验收标准 | 被写手/审稿/台账 agent 拉走 |
| 起草中 | 正在写的章节或片段 | 任务单通过 | 初稿交付试读/审稿 |
| 审稿中 | 等待试读、逻辑审查、风格审查 | 有初稿和任务单 | 出修订意见或通过 |
| 修订中 | 正在改稿、补台账、改细纲 | 有明确修改目标 | 达到完成定义 |
| 待发布/待交付 | 可给用户或平台的结果 | 正文、台账、索引均合格 | 用户确认、发布或进入观察 |
| 已发布/已确认 | 正稿事实和稳定决策 | 用户确认或正文发布 | 进入复盘或触发变更 |
| 阻塞 | 缺资料、冲突、无法判断的任务 | 无法安全继续 | 用户决策、资料补齐或降级处理 |

## 4. WIP 限制：防止 agent 多线失忆

WIP 是“进行中工作”。对 AI 写作项目来说，WIP 过多会导致：

- 角色状态混淆。
- 未发生事件被当成已发生。
- 大纲、细纲、正稿和台账互相覆盖。
- 审稿意见堆积但没有进入改稿。
- agent 看似忙碌，实际没有完成可发布成果。

建议限制：

- 同一时间只允许 1 个“正式章节起草中”。
- 同一时间最多 2 个“修订中”任务；超过时先合并或排序。
- 同一时间最多 1 个“影响主线的大变更候选”进入讨论。
- 正稿发布前，不再新增无关设定专题或大段世界观资料。
- 如果出现 3 个以上阻塞任务，暂停起草，先做项目整理。

例外：

- 纯资料检索、短小包装文案、错别字修复可以并行，但不得改变正稿事实。
- 多 agent 并行时，必须由总控 agent 维护一个统一看板和台账入口。

## 5. 任务卡最小字段

每张卡不要写成长报告。建议保持 8 个字段：

```text
【任务】一句话说明
【类型】章节 / 修订 / 台账 / 大纲 / 包装 / 反馈 / 资料 / 决策
【状态】候选 / 待澄清 / 已准备 / 起草中 / 审稿中 / 修订中 / 待交付 / 已确认 / 阻塞
【输入】必须读取的项目文件或正稿
【目标】本轮要改变什么
【完成定义】什么情况才算完成
【影响范围】会改哪些人物、设定、台账、细纲或发布材料
【下一步】谁或哪个 agent 接手
```

任务卡必须能指向具体文件；不能只写“优化一下”“增强爽点”“人物更自然”。

## 6. 完成定义

### 6.1 章节完成定义

一章进入“待发布/待交付”前，至少满足：

- 字数达到当前项目约定；普通中文网文章节默认不低于可发布长度门槛。
- 本章目标、阻力、尝试、代价、回报和章末压力清楚。
- 没有明显逻辑漏洞、信息空降、动作顺序错位或无功能句。
- 人物对白和行动符合当前关系、压力、知识限制和风格手册。
- 写手、试读、审稿、修订闭环至少走过一轮。
- 正稿事实、人物状态、规则变化、伏笔状态和下一章任务已回写。

### 6.2 文档变更完成定义

项目文档变更完成前，至少满足：

- 新增文件有入口，入口说明后续 agent 什么时候读。
- 修改过的事实没有和正稿冲突；如冲突，以正稿和人工确认版本为准。
- 候选、待确认、已确认状态标清楚。
- 影响大纲、细纲、台账或风格手册的变更已经同步。
- 旧信息不再准确时，写明废止或替代关系，不靠删掉上下文制造假一致。

### 6.3 反馈处理完成定义

读者反馈或编辑意见进入“完成”前，至少满足：

- 已记录反馈来源类型和版本，不保留可识别隐私。
- 已判断是单人偏好、多人共识、数据信号还是编辑判断。
- 已拆成可执行修改项或明确不采纳理由。
- 已标出观察窗口：改完后看哪几章、哪些数据或哪些读者体感。

## 7. 变更控制

连载中所有内容都可以动态调整，但不能无痕漂移。

### 7.1 变更分级

| 等级 | 示例 | 处理方式 |
|---|---|---|
| 微调 | 台词、段落顺序、错字、局部动作 | 可直接改，必要时回写章节台账 |
| 小变更 | 单章钩子、支线出场顺序、伏笔提前/延后 | 更新章节任务单、细纲和台账 |
| 中变更 | 角色关系转向、卷目标调整、机制限制变化 | 进入待澄清，用户或总控确认后再改 |
| 大变更 | 主角目标、核心卖点、CP 归属、世界底层规则、结局方向 | 必须写决策记录，评估已发布正稿和读者承诺成本 |

### 7.2 变更卡字段

```text
【变更提议】
【触发原因】作者决定 / 正稿事实 / 读者反馈 / 编辑意见 / 数据异常 / agent 发现漏洞
【涉及范围】章节、人物、设定、伏笔、包装、发布材料
【与正稿冲突】无 / 有，冲突点是：
【读者承诺影响】不影响 / 轻微影响 / 重大影响
【推荐处理】接受 / 暂缓 / 拆小 / 否决 / 需要用户确认
【同步文件】
【回滚方案】如果后续证明不合适，怎么退回
```

## 8. 阻塞处理

阻塞不是失败，而是防止 agent 胡编。

常见阻塞：

- 正稿版本不明。
- 细纲过薄，无法支撑完整章节。
- 人物动机缺失，导致剧情只能靠作者推。
- 设定规则冲突。
- 用户尚未确认关键方向。
- 涉及平台规则、法律、医疗、金融等需要最新核查的信息。

处理顺序：

1. 标记阻塞卡，不继续硬写。
2. 写明缺口和风险。
3. 给 1-3 个可选解决方案，或补一张项目任务单。
4. 如果能安全降级，改成候选小样、资料整理或任务单输出。
5. 阻塞解除后，回到“已准备”，不要直接跳到“完成”。

## 9. 短周期复盘

建议复盘节点：

- 第 1 章完成后。
- 第 3 章或第 5 章后。
- 第 10 章后。
- 每卷中点和卷末。
- 重大读者反馈、编辑意见、数据异常或作者大改后。

复盘只回答五件事：

1. 本周期原目标是什么？
2. 实际交付了什么？
3. 哪些读者承诺兑现了，哪些还欠债？
4. 哪个流程问题最影响质量：资料缺、任务单薄、WIP 过多、审稿弱、台账没回写，还是变更无痕？
5. 下一周期只改一个最重要流程问题是什么？

## 11. Agent 自查

交付前自问：

- 我是否同时打开太多起草、修订和变更任务？
- 当前任务是否有明确完成定义？
- 本轮输出是否已经回写正稿事实、台账或索引？
- 是否把候选想法误写成已确认设定？
- 是否因为想继续写而跳过了阻塞？
- 如果下一个 agent 接手，它能从入口找到当前任务状态吗？

## 12. 来源与迁移边界

本篇吸收了 Kanban 与 Scrum 的项目管理思想，主要取其“可视化工作流、限制进行中任务、显式工作策略、完成定义、短周期复盘”的通用原则。迁移到小说项目时，只保留能降低 agent 失忆、减少上下文切换、提高交付质量和保证台账回写的部分。
