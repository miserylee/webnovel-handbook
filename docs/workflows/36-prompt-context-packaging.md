# 36. 提示词与上下文打包规范

专题路由与最小阅读包入口：

- 如果提示词需要引用多个知识库专题，或任务容易把大量县域/行业/部门专题全塞进上下文，先看 `docs/workflows/57-knowledge-base-routing-consolidation-guide.md`。上下文包应只保留任务阶段核心、人物/节奏/风格规则、主专题和必要相邻专题；看似相关但本轮不用的专题要明确排除。

第一章生产许可入口：

- 如果任务涉及第一章起草、第一章重写、`ch001.md` 是否可以开写、开篇任务单是否齐备或第一章写后回写，先看 `docs/workflows/45-first-chapter-task-brief-production-gate.md`。该专题负责检查项目许可、任务许可、输出许可、第一章输入包、任务单字段和写后回写边界。


项目初始化入口：

- 如果任务涉及通过立项后创建单本小说项目、生成目录、建立最小文档包、准备第一章任务或多 agent 接力前的项目事实来源，先看 `docs/workflows/44-single-novel-project-initialization-package.md`。该专题负责规定项目目录、必填文件、状态标签、台账、风格手册和第一章生产许可。


反馈冲突入口：

- 如果编辑反馈、读者反馈、后台数据、作者直觉或项目主承诺互相冲突，先看 `docs/feedback-revision/42-editor-reader-data-feedback-conflict-resolution.md`。该专题负责判断来源权重、证据同向性、主承诺保护、修改成本和观察窗口，避免简单投票或盲从单一反馈。


## 相关入口：开篇试写与 A/B 版本评估

打包开篇候选方案给下游 agent 时，调用 `docs/feedback-revision/39-opening-variant-ab-evaluation.md`。上下文包必须标注每个版本的状态、主变量、假设和不变量，候选内容不能污染已确认设定。

## 相关入口：章节生产流水线与多 agent 交接规范

当提示词用于章节生产、审稿后修订或多 agent 接力时，应先调用 [38. 章节生产流水线与多 agent 交接规范](./38-chapter-production-pipeline-agent-handoff.md)。该专题负责定义当前流水线步骤、当前 agent 职责、下游交接包、质量闸门和台账回写要求，避免起草、审稿、修订和记账混成一团。

## 相关入口：项目风格手册与角色声音规范

提示词上下文中涉及起草、续写、修订、审稿或多 agent 交接时，应按需加入 [37. 项目风格手册与角色声音规范](../core-writing/37-project-style-bible-character-voice.md)。风格样例必须标注功能和可迁移规则，不能复制原句；角色声音、称呼术语、旁白距离和可保留粗糙要作为上下文约束，而不是模糊的“文风像某某”。

状态：已确认。用途：给 AI agents 在使用本知识库起草、续写、审稿、修订、数据复盘或维护文档前，打包项目简报、台账、专题规则、章节目标、风格样例和输出格式，避免上下文遗漏、上下文污染、规则冲突、事实漂移和提示词口号化。

本专题联动 `docs/workflows/35-ai-agent-novel-creation-workflow.md`、`docs/workflows/38-chapter-production-pipeline-agent-handoff.md`、`docs/core-writing/37-project-style-bible-character-voice.md`、`docs/workflows/03-project-workflow.md`、`docs/core-writing/06-ai-writing-guidelines.md`、`docs/core-writing/07-continuity-ledger.md`、`docs/templates/10-templates-and-checklists.md`、`docs/feedback-revision/33-revision-rewrite-version-workflow.md`。

## 1. 核心原则：提示词不是愿望，是任务封包

有效提示词必须让 AI 明白：当前任务是什么、哪些事实已确认、必须遵守哪些规则、可用哪些参考、输出什么格式、哪些内容禁止做。

小说任务上下文包使用这条链路：

```text
任务阶段 -> 来源优先级 -> 已确认事实 -> 相关专题规则 -> 当前章节/修订目标 -> 风格与禁区 -> 输出契约 -> 后续台账动作
```

如果提示词只写“写得商业一点”“降低 AI 味”“人物鲜活一点”，它没有给出可执行依据，只是在许愿。

## 2. 上下文包的八个层级

| 层级 | 内容 | 作用 |
|---|---|---|
| 任务层 | 开新书、起草、续写、审稿、修订、数据复盘、维护知识库 | 决定读取路线和输出形式 |
| 权威层 | 用户当前指令、项目文档、本知识库、`novel` skill、旧项目经验 | 解决冲突时按优先级处理 |
| 事实层 | 已确认设定、人物状态、时间线、伏笔、上一章结果 | 防止编造和连续性错误 |
| 规则层 | 本次任务相关专题的核心规则 | 防止泛泛调用整个知识库 |
| 目标层 | 本章主任务、修订目标、审稿维度、数据问题 | 让输出有方向和验收标准 |
| 风格层 | 叙述视角、语气、人物声音、可模仿的结构特征 | 保持项目声音，不复制样例文本 |
| 禁区层 | 不许改动、待确认、合规风险、抄袭风险、已发布风险 | 防止越权和误伤 |
| 输出层 | 要输出正文、诊断、表格、台账更新还是修改说明 | 降低返工和格式漂移 |

## 3. 上下文选择：只放相关，不堆全库

上下文窗口不是仓库。把整个知识库、全部大纲、全部旧章都塞进提示词，会制造噪音、冲突和成本。

选择顺序：

1. 用户当前任务和最新决定。
2. 当前项目的已确认事实、台账和当前章节目标。
3. 与任务直接相关的 2-6 个专题规则。
4. 最近一章或最近一段需要承接的正文。
5. 风格样例和禁区。
6. 低相关背景、旧讨论、完整资料库和候选想法不放，除非任务明确需要。

硬规则：候选信息必须标“候选”，待确认信息必须标“待确认”；不能混进已确认事实。

## 4. 上下文排序

推荐排序：

```text
# 任务
# 权威来源与冲突规则
# 输出要求
# 已确认项目事实
# 当前状态/台账摘要
# 相关专题规则摘要
# 当前任务目标
# 风格样例/禁区
# 待确认问题
# 输入正文或材料
```

复杂长上下文任务，可以在末尾再次压缩提醒：本次必须完成什么、禁止什么、输出什么。

原因：模型需要先知道任务和规则，再读材料；长材料之后需要被重新拉回本次目标。

## 5. 使用 Markdown 与标签分隔材料

用清晰标题和标签标出材料边界，避免模型把正文、台账、评论、规则和输出格式混在一起。

推荐格式：

```markdown
# 任务
请根据下列资料起草第 12 章。

<context type="project_brief" status="已确认">
...
</context>

<context type="ledger" status="已确认">
...
</context>

<context type="rules" source="docs/32" status="已确认">
...
</context>

<input type="previous_chapter">
...
</input>

# 输出要求
只输出正文；不要输出章纲、自检或解释。
```

标签不用复杂，关键是让每块材料有类型、状态和来源。

## 6. 任务类型对应上下文包

### 6.1 开新书

必放：市场/竞品结论、目标平台、目标读者、主承诺、题材定位、核心卖点、禁抄边界、项目简报模板。

禁止：直接要求写第一章，除非第一章目标、主角压力和开篇场景已确认。

### 6.2 起草新章

必放：本章编号、上一章结果、活跃台账、本章主任务、场景目标/阻力/转折、相关人物状态、相关专题规则、输出只要正文。

禁止：把整本设定集和全部旧章塞进去；不要在正文后附作者说明。

### 6.3 续写下一章

必放：上一章末状态、未完成动作、读者等待的问题、主角当前目标、活跃伏笔、必须回收/不能提前揭示的信息。

禁止：只根据上一章最后一段续写，忽略台账和阶段目标。

### 6.4 审稿

必放：作品定位、目标读者、包装承诺、待审文本、审稿维度、评分或结论格式、是否需要给修改优先级。

禁止：让 AI 只凭“好不好看”输出笼统意见。

### 6.5 修订/重写

必放：用户目标、不许动的内容、问题层级、有效资产、待修文本、修订模式、小修/中修/大修边界、输出是否需要说明。

禁止：未做有效资产盘点就整段重写。

### 6.6 数据复盘

必放：数据来源、观察窗口、样本量、评论聚类、平台节点、更新情况、包装和开篇承诺。

禁止：编造后台指标；样本不足时不能下大修结论。

### 6.7 知识库维护

必放：新增资料主题、来源链接、时效性、可迁移结论、要写入的专题、要更新的入口和决策记录。

禁止：只把链接堆进来源清单，不沉淀专题规则。

## 7. 风格样例打包规则

风格样例用于保持声音，不用于复制文本。

规则：

1. 一次放 1-3 个短样例即可。
2. 每个样例标注用途：人物声音、叙述节奏、喜剧口吻、情绪浓度、动作密度。
3. 标明“学习节奏和声音，不复制句子、桥段、比喻和专有表达”。
4. 样例太长时只摘最有代表性的段落，并总结可迁移特征。
5. 风格样例不能覆盖项目事实和章节目标。

## 8. 规则摘要不要口号化

从专题文档提取规则时，不要只写标题。

低效：

```text
参考人物、节奏、降 AI 味。
```

有效：

```text
本章必须满足：
- 主角有可见目标和主动选择。
- 对手/环境提供具体阻力。
- 每场至少有目标、阻力、转折。
- 设定只通过动作、物件、后果或对话投放。
- 章末必须“本章落袋 -> 后果反弹 -> 下一步必须行动”。
```

## 9. 输出契约必须明确

提示词最后写清输出内容：

- 只输出正文。
- 输出诊断表，不改正文。
- 先给问题层级，再给修改方案。
- 输出修订后正文和变更说明。
- 输出台账更新项，不复述整章。
- 输出知识库新增段落，不写聊天总结。

输出契约越清楚，越不容易出现正文夹杂自检、审稿夹杂改写、台账变成剧情复述。

## 10. 上下文压缩规则

当材料太多时，按功能压缩：

| 原材料 | 压缩成 |
|---|---|
| 多章正文 | 上一章结果、关键动作、人物状态、未完成钩子 |
| 设定集 | 本章会用的规则、限制、代价、已知/未知状态 |
| 人物卡 | 当前目标、压力、关系状态、声音禁区 |
| 评论区 | 高频正反馈、负反馈、争议点、样本不足说明 |
| 竞品资料 | 可迁移结构规律、不可迁移内容、抄袭风险 |
| 专题文档 | 本任务必须遵守的 5-10 条硬规则 |

压缩不等于改写事实。事实不确定时，标“待确认”。

## 11. 防上下文污染

上下文污染常见来源：

- 旧设定和新设定混在一起。
- 候选想法没标候选。
- 读者猜测被写成真实伏笔。
- 审稿意见被当成剧情事实。
- 竞品桥段被当成项目灵感。
- 平台旧规则被当成最新规则。
- 风格样例覆盖人物真实声音。

处理方式：每块材料标注状态、来源、用途和是否可写进正文。

## 15. AI 写作硬性禁令

1. 禁止没有任务阶段就直接打包上下文。
2. 禁止把整个知识库、整本设定集或全部旧章无差别塞进提示词。
3. 禁止把候选、读者猜测、竞品桥段写成已确认事实。
4. 禁止只写“更商业、更爽、更有情绪”这类不可执行要求。
5. 禁止上下文里规则互相冲突却不标注优先级。
6. 禁止风格样例变成可复制文本。
7. 禁止输出契约不清，导致正文、自检、台账、审稿混在一起。
8. 禁止在数据不足或资料过期时要求 AI 下确定结论。
9. 禁止把平台规则、法律、医疗、金融、审核结果等不稳定信息写成永久事实。
10. 禁止改稿后不要求台账更新项。

## 16. 快速自查问题

- 当前任务阶段是什么？
- 本次真正需要哪些专题，不需要哪些？
- 已确认、候选、待确认是否分开？
- 是否给了本章/本次任务的主目标和验收标准？
- 是否给了必要台账，而不是只给旧正文？
- 是否给了输出格式和禁止输出内容？
- 是否有过期资料、竞品内容或读者猜测污染上下文？
- 是否给了风格样例的使用边界？
- 是否要求输出台账更新项或改稿说明？
