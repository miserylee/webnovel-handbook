# 57. 知识库专题路由、最小阅读包与去发散收束规范

状态：已确认  
最近整理：2026-06-09  
适用对象：AI 写作 agent、审稿 agent、项目初始化 agent、知识库维护 agent  
核心用途：在专题数量已经很多的情况下，帮助 agent 少读、读准、组合使用，而不是继续横向发散或把所有专题都塞进上下文。

---

## 0. 渐进式披露索引协议

本知识库不再鼓励 agent 通过“全量索引 + 全量来源 + 全量模板”理解项目。默认路径改为：

```text
启动索引 -> 任务阶段 -> 最小阅读包 -> 目标专题 -> 必要来源
```

执行规则：

1. `docs/00-index.md` 只保留启动路由、最小阅读包和专题簇，不承载完整专题长表。
2. 旧版长索引移动到 `docs/navigation/00-expanded-topic-catalog.md`，只用于关键词检索和精确定位，不作为默认阅读文件。
3. `docs/sources/01-source-inventory.md` 只在来源登记、来源冲突、版权/合规边界和引用核查时定向读取。
4. `docs/templates/10-templates-and-checklists.md` 只在需要具体模板时按标题检索，不得整篇塞入写稿上下文。
5. 写稿、续写、审稿、改稿任务优先读取质量闭环：`58`、`06`、`11`、`37`、`59`、`07`、`32`，再按需要补题材或事实专题。
6. 如果某个专题体积过大，agent 应先检索标题、适用场景、硬规则、检查问题和模板，不要从头读到尾。
7. 活跃文档必须短小。历史长文可以归档冻结；新内容不要继续追加到超大文档。若内容膨胀，允许索引链路增加一跳或多跳，用更薄的入口指向更小的目标文档。

## 1. 本规范解决什么问题

这个知识库已经覆盖大量写作流程、商业网文技法、县域生活、公共服务、司法行政、市场经营、教育医疗、灾害安全和社会治理细节。继续无节制新增专题，会带来三个风险：

- agent 拿到任务后不知道先读哪几篇，只能机械扩大阅读范围。
- 相邻专题重复解释同一种链路，比如“投诉、证据、分流、处罚、救济”反复出现。
- 新资料越来越细，但对小说生产的帮助递减，反而增加上下文噪音。

因此，后续维护原则从“继续补空白”转为“先路由、再压缩、再必要补充”。

一句话规则：

```text
先判断任务阶段，再判断场景主轴，只读取最小必要阅读包；除非出现独立主体、独立流程、独立证据链和高频写作价值，否则不新增专题。
```

## 2. 三层阅读模型

### 2.1 第一层：任务阶段层

先判断用户要做什么，而不是先判断知识库里有什么。

| 任务 | 默认读取 |
|---|---|
| 开新书、重定方向 | `43`、`35`、`34`、`29`、`13`、`03` |
| 初始化单本项目 | `43`、`44`、`03`、`36`、`37`、`10` |
| 写第一章 | `44`、`45`、`22`、`29`、`32`、`02`、`06` |
| 写第 2-3 章 | `46`、`38`、`07`、`37`、`32`、`06` |
| 写第 4-10 章 | `47`、`48`、`49`、`38`、`05`、`07` |
| 写第 11-30 章 | `50`、`51`、`52`、`38`、`05`、`07` |
| 写 30 章后卷结构 | `53`、`54`、`55`、`56`、`23`、`07` |
| 审稿/试读 | `38`、`08`、`18`、`33`、`11` |
| 评论/数据复盘 | `40`、`41`、`42`、`18`、`33` |
| 修订/重写 | `33`、`11`、`32`、`06`、`37` |
| 知识库维护 | `00`、`01`、`57`、任务对应专题、`99` |

原则：任务阶段层优先于行业专题。比如用户说“这一章写县城烟酒店被查”，不能只读烟草专题，还要看这是起草、审稿、修订还是素材核查。

新手开书保护：如果用户说“开书”“写第一章”“先试写”，但没有项目简报、前 10 章方向、第一卷细纲、第一章任务单、风格手册或台账，路由结果不得直接进入起草专题。先读 `43`、`44`、`45` 和 `10` 的开书准备缺口诊断卡。

### 2.2 第二层：叙事功能层

同一个现实专题，在小说里可能承担不同功能。先判断它在故事中的作用。

| 叙事功能 | 读取方向 |
|---|---|
| 增强生活质感 | 读职业/地域/日常细节类专题，提取物件、流程、人物口吻 |
| 制造冲突 | 读对应制度链，确认主体、规则、证据、后果 |
| 推动案件/悬疑 | 读证据链、部门分流、时间线、信息差专题 |
| 塑造人物 | 读人物、关系、情绪线、人味增强，再读现实专题 |
| 做爽点/反杀 | 读类型承诺、阻力系统、机制设计、合规边界 |
| 做审稿纠错 | 读审稿、反馈、修订，再读相关事实专题 |

原则：现实资料不是越多越好。只把能服务“人物选择、冲突升级、证据推进、情绪回报”的部分带入上下文。

### 2.3 第三层：专业事实层

只有当具体情节真的触及专业流程时，才读取细分专题。

判断问题：

- 这场戏是否需要知道具体主管部门？
- 这场戏是否需要证据材料、许可证、申请、审批、处罚、救济或台账？
- 这场戏是否涉及未成年人、隐私、公共安全、刑事风险、行政处罚或民生资格？
- 如果不读专题，会不会写出明显错误或危险细节？

如果四个问题都是否定，通常不用读取细分专题，只用通用生活质感和人物规则即可。

## 3. 最小阅读包规则

### 3.1 轻任务：2-4 篇

适用：一句话设定建议、短段落润色、轻量审稿、桥段补真实感。

默认包：

```text
任务阶段核心 1 篇 + 当前文本直接相关专题 1 篇 + 人味/节奏专题 1 篇 + 必要时合规专题 1 篇
```

例子：改一段“小卖部老板拒绝学生买烟”的桥段：

- `11` 人味增强
- `254` 烟草与未成年人保护
- 必要时 `32` 移动端段落体验

### 3.2 中任务：4-8 篇

适用：起草一章、审稿一章、设计 3-5 个场景、修订一个剧情单元。

默认包：

```text
任务阶段核心 2-3 篇 + 人物/节奏/风格 2-3 篇 + 直接专业专题 1-2 篇
```

例子：写一章“县城体育馆群众赛事出事故，主角协助处理”：

- `38` 章节生产流水线
- `02` 网文核心原则
- `32` 章法与移动端阅读
- `37` 风格与角色声音
- `247` 体育赛事与场馆
- `235` 消防或 `236` 安全生产，按事故性质二选一

### 3.3 重任务：8-12 篇

适用：开新书、搭项目、改大纲、重写前 10 章、跨部门大型事件、复杂审稿。

默认包：

```text
总流程/项目层 2-4 篇 + 类型/读者/结构 3-4 篇 + 角色/风格 1-2 篇 + 专业专题 1-3 篇
```

硬上限：除非用户明确要求做全库整理或法规研究，一次任务不要主动读取超过 12 篇专题。

### 3.4 禁止包

不要这样打包：

- “所有县域公共事务专题都读一遍。”
- “把 200 篇专题全文塞进提示词。”
- “因为有一个学生角色，就把所有未成年人相关专题都读完。”
- “因为有投诉，就把信访、市场监管、法院、检察、公安、纪检全读完。”

正确做法：先找主轴，再找相邻接口。

## 4. 主轴判定法

当多个专题都像是相关时，用下面五问选主专题。

### 4.1 谁在做决定

谁拥有最关键的许可、处罚、审批、救助、裁判或服务决定权，谁就是主专题。

例子：

- 小卖部卖烟：主专题是 `254`，不是普通市场监管。
- 药店卖处方药：主专题是 `231`，不是普通消费纠纷。
- 公租房资格：主专题是 `250`，不是普通租房。

### 4.2 主要证据是什么

证据链决定专题归属。

例子：

- 合同、发票、税票、账簿：优先 `232` 税务或 `222` 采购。
- 病历、诊断、转诊、医保结算：优先 `169`、`193`、`199`。
- 监控、报警记录、伤情、笔录：优先 `166`、`228`、`229`，按刑事/民事/检察分流。

### 4.3 伤害结果是什么

如果情节的核心后果不同，专题也不同。

例子：

- 学生进网吧：`245` 文化市场与未成年人保护。
- 学生买烟：`254` 烟草与未成年人保护。
- 学生被欺凌：`192` 校园欺凌。
- 学生心理危机：`205` 心理健康。

### 4.4 叙事回报是什么

如果专业事实只是背景，就不必过度读取。

例子：

- 主角在图书馆偶遇，只需要公共空间质感，不需要完整公共文化资金链。
- 主角因假烟投诉触发调查，则需要 `254` 的证据链和部门分流。

### 4.5 用户要的输出是什么

输出类型决定阅读深度。

例子：

- “帮我补几处真实细节”：读少量专题摘要。
- “审这章有没有制度错误”：读主专题和相邻接口。
- “写一份完整办案线大纲”：读主专题、证据链、司法救济和合规边界。

## 5. 专题簇，而不是专题海

后续 agent 面对大量编号时，先按专题簇归类。

### 5.1 小说生产簇

核心用途：开书、包装、起草、续写、审稿、修订、复盘。

默认入口：

- `02`、`03`、`04`、`05`、`06`、`07`、`08`、`10`、`11`
- `18`、`19`、`20`、`22`、`23`、`24`
- `32`、`33`、`35`、`36`、`37`、`38`
- `39` 到 `56` 的开篇、前 30 章和卷结构专项

调用规则：这组永远先于生活细节专题，因为它决定“这个资料要服务什么故事功能”。

### 5.2 市场与包装簇

核心用途：判断一本书值不值得写、卖给谁、怎么包装。

默认入口：

- `12` 平台差异
- `13` 题材类型
- `16` 书名简介标签
- `17` 封面视觉
- `29` 读者预期与卖点
- `34` 竞品拆解
- `39` A/B 开篇评估
- `43` 立项评分

调用规则：开书前读；章节中途一般不读，除非包装与正文错位或数据反馈异常。

### 5.3 人物关系与情绪簇

核心用途：让人物活、关系有张力、情绪能黏住读者。

默认入口：

- `04` 人物与对话
- `11` 人味增强
- `15` 关系类型
- `21` 对手阻力
- `25` 情绪线
- `26` 悬疑信息差
- `27` 喜剧梗点
- `37` 风格与角色声音

调用规则：任何“人物不活、剧情生硬、没有梗、情绪淡、读者不黏”的任务，优先读这一簇，再读事实专题。

### 5.4 现实质感与公共事务簇

核心用途：县城、乡村、行业、部门、公共服务和生活制度真实感。

默认入口：

- `30` 职业行业生活细节
- `31` 地域时代社会风俗
- `57` 本路由规范
- 再按场景主轴选 `160` 以后大量县域专题

调用规则：只在具体情节触及现实流程时读。不要因为作品背景是县城，就把所有县域专题打包。

### 5.5 安全合规与边界簇

核心用途：避免违法教程化、隐私泄露、未成年人伤害、平台审核风险。

默认入口：

- `20` 内容合规
- `105` 污名和伤害类表达边界
- `163` 隐私与个人信息
- `166` 公安警务
- `227` 行政救济
- `228` 法院
- `229` 检察

调用规则：出现未成年人、犯罪、公共安全、隐私、行政处罚、医疗、金融、灾害、武器、诈骗、走私等内容时，必须至少读一个安全边界相关专题。

## 6. 相邻专题分流规则

### 6.1 投诉不等于信访

先判断投诉对象：

- 消费纠纷：市场监管、消费者权益、具体行业专题。
- 政府不作为或公共服务：政务服务、12345、信访调处。
- 行政处罚不服：行政复议、行政诉讼。
- 犯罪线索：公安、检察、法院链路。
- 干部违纪：纪检监察。

不要把所有“群众反映问题”都写成上访。

### 6.2 未成年人不是一个专题能包完

先判断发生场景：

- 学校日常、学籍、乡村学校：教育专题。
- 欺凌：校园欺凌专题。
- 心理危机：心理健康专题。
- 网吧、KTV、剧本杀：文化市场专题。
- 买烟、电子烟：烟草专题。
- 家暴、监护困境：妇联、民政、公安、法院衔接。

不要把“未成年人保护”写成一个万能部门。

### 6.3 钱款纠纷先看钱的性质

先判断资金来源：

- 工资、社保、工伤：劳动人社。
- 财政补贴、项目资金：财政、审计、纪检。
- 采购合同、工程款：采购交易、审计、法院。
- 税票账簿：税务。
- 贷款、银行卡、诈骗：金融、电诈、公安。

不要看到“钱没到账”就直接写成欠债讨债。

### 6.4 事故先看事故类型

先判断事故主因：

- 火灾：消防救援。
- 生产作业：应急管理和安全生产。
- 自然灾害：防汛、气象、自然灾害救助。
- 交通运输：道路交通、客运、出租网约车。
- 食品药品：市场监管、药监、疾控。
- 校园或活动：教育、体育、文旅、公安消防按场地分流。

不要让一个部门包办所有事故调查。

### 6.5 “官方发布”先看发布目的

先判断信息性质：

- 政府信息公开申请：政务公开。
- 突发事件通报：应急、公安、卫健等主管部门加融媒体。
- 舆情回应：融媒体和主管部门会商。
- 新闻采访：媒体采编和事实核实。
- 谣言治理：网信、公安、主管部门分流。

不要把公众号推文写成万能公文。

## 7. 专题提纯格式

后续整理旧专题或新增必要专题时，统一压缩为六个可执行块：

1. 入口判断：什么时候读这篇，什么时候不要读。
2. 行动链：人物或机构从问题出现到结果回写的步骤。
3. 证据链：能推动事实判断的材料。
4. 常见误写：AI 最容易写错的地方。
5. 人物冲突：制度如何压到人物选择、面子、利益和情绪。
6. 安全边界：哪些内容只能写诱因、线索和后果，不能写操作方法。

不推荐继续堆这些内容：

- 大段法规原文。
- 和小说无关的机构简介。
- 与其他专题重复的通用投诉流程。
- 只可用于论文或政策研究、不能转成情节的资料。

## 8. 新专题准入门槛

后续除非满足以下至少四项，否则不要新增专题：

- 高频：中文网文或县域生活中经常出现。
- 易错：AI 不查资料就容易写假。
- 独立：有独立主管主体、独立程序或独立证据链。
- 有戏：能制造人物选择、冲突、悬念、情绪或回报。
- 有风险：涉及未成年人、隐私、行政处罚、刑事风险、公共安全、医疗金融等边界。
- 现有专题无法承接：不能通过给旧专题增加小节或交叉引用解决。

如果只是一个旧专题的子场景，优先做三种处理：

- 在旧专题补一个“场景变体”。
- 在 `00-index.md` 增加交叉调用说明。
- 在 `docs/templates/10-templates-and-checklists.md` 增加字段，而不是新建专题。

---

## 9. 旧专题整理优先级

当用户要求“整理和提纯”时，优先做这些事：

1. 把入口文档的阅读路线压缩成“任务阶段 -> 最小阅读包 -> 可选专题”。
2. 把相邻专题合并成专题簇索引。
3. 把重复的证据链、投诉链、部门分流链抽成模板。
4. 给过长专题加“十秒定位卡”。
5. 给容易误读的专题加“不读本篇的情况”。
6. 标出可被旧专题承接的候选新增主题，避免继续开新坑。

暂缓做这些事：

- 无明确目标地继续联网搜资料。
- 为冷门低频场景单独开专题。
- 大规模重排编号。
- 删除旧专题正文。
- 把所有专题做成同一种模板而牺牲已有细节。

## 10. 最小阅读包模板


## 11. 给 agent 的执行口令

在开始复杂任务前，可以直接使用这段：

```text
请先按 `docs/workflows/57-knowledge-base-routing-consolidation-guide.md` 做专题路由：
1. 判断任务阶段。
2. 判断叙事功能。
3. 判断本轮是否允许写正式正文；新书和第一章缺项目包时，先输出缺口诊断或任务单。
4. 选出主专题和最多 2 个相邻专题。
5. 明确本轮不读哪些看似相关专题以及原因。
6. 只提取入口判断、行动链、证据链、常见误写、人物冲突和安全边界。
7. 完成本轮任务后停止，不继续新增专题或横向扩展。
```

## 12. 维护口径

从 2026-06-09 起，知识库默认进入“收束提纯优先”阶段。

维护排序：

1. 先修入口和路由。
2. 再合并重复规则。
3. 再补模板和检查清单。
4. 再按必要性更新旧专题。
5. 最后才考虑新增专题。

新增资料仍然可以做，但必须证明它比“加入旧专题小节”更合适。否则，宁可让知识库更薄、更准，也不要让它更散。

## 13. 新增文件的索引门槛

状态：已确认。

知识库维护 agent 新建任何文件前，必须先判断它是否有明确索引路径。专题已经很多时，新增文件的成本不只是文件本身，还包括后续 agent 能否找到、何时读取、是否会和旧专题重复。

### 13.1 新文件准入问题

创建前先回答：

1. 这个内容能否并入已有专题、小节、模板、案例参考书或决策记录？
2. 如果必须新建，它属于哪个任务路线：开书、起草、审稿、修订、包装、运营、数据复盘、知识库维护？
3. 它的入口挂在哪里：`README.md`、`docs/00-index.md`、专题路由文档、某个目录 README，还是单本项目的文档地图？
4. 后续 agent 在什么情况下必须读它？如果答案只是“可能有用”，优先不要新建。
5. 是否需要同步 `docs/sources/01-source-inventory.md`、`docs/governance/99-decision-log.md`、`CHANGELOG.md` 或模板文档？

### 13.2 硬规则

- 新增正式文档和更新索引必须同一轮完成。
- 新增来源材料必须有来源登记或明确说明“用户当前指令/项目内经验”。
- 新增关键规则必须有决策记录。
- 新增模板必须挂到模板文档或相关任务入口。
- 新增案例参考必须挂到对应案例集或专题入口。
- 如果只创建文件、不更新入口，视为未完成任务。

### 13.3 收束原则

不要为了索引而继续开更多索引文件。优先复用现有入口：`README.md` 面向快速接续，`docs/00-index.md` 面向全库导航，`docs/workflows/57-knowledge-base-routing-consolidation-guide.md` 面向路由和收束，`docs/governance/99-decision-log.md` 面向规则决策。只有当一个文件承担独立高频任务时，才允许新增独立参考书或专题。
