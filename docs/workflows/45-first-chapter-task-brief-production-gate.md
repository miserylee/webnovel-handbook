## 1. 核心原则：第一章不是“开写”，而是一次生产放行

第一章是读者从外包装进入正文后的第一次验证。对 AI agents 来说，第一章不能靠“文笔好一点”“开头炸一点”解决，它必须把项目主承诺转成可见场景：

- 谁在行动。
- 他或她为什么现在必须行动。
- 面前有什么阻力。
- 读者在第一章看见了什么类型承诺。
- 第一章末为什么要点下一章。

一句话原则：第一章任务单不是大纲摘要，而是正文生产许可；任务单不齐，不能写 `ch001.md`。

如果用户是第一次开书，或项目里还没有大纲、细纲、人物卡、设定台账和风格手册，agent 必须先回到 `docs/workflows/44-single-novel-project-initialization-package.md` 补最小文档包。第一章许可不能替代项目初始化。

## 2. 第一章生产许可的三个层级

### 2.1 项目许可

确认项目是否有资格进入第一章：

- 已通过或候选通过立项评分。
- 单本项目最小文档包已经建立。
- 目标读者、平台、主承诺、主角、核心卖点有清晰记录。
- 风格手册和初始台账至少有可用版本。
- 前 10 章方向和第一卷阶段目标至少有可用版本。
- 没有未处理的高风险合规、版权或资料问题。

不满足时，只能补项目文档或做小样，不能写完整第一章。

### 2.2 任务许可

确认第一章要写什么：

- 本章读者承诺清楚。
- 开场场景清楚。
- 主角第一章压力清楚。
- 主角主动选择清楚。
- 阻力或对手清楚。
- 卖点证明清楚。
- 章末钩子清楚。

不满足时，只能完善第一章任务单，不能进入正文。

### 2.3 输出许可

确认正文输出边界：

- `ch001.md` 只放读者可见正文。
- 不在正文里写任务单、自检、解释、作者说明、台账或审稿意见。
- 写完正文后，另行输出台账更新项、风格手册更新项和下一章任务建议。

不满足时，说明 agent 没有区分创作层和工作层，不能进入正式章节生产。

## 4. 第一章任务单标准字段

第一章任务单必须放在大纲或卷纲里，不放进正文文件。

```markdown
## 第 1 章任务单

状态：已确认 / 候选 / 待确认

【本章作用】
第一章负责让读者确认什么？

【读者承诺】
读者从书名、简介、标签、封面进入正文后，期待看到什么？

【开场场景】
地点、时间、人物在做什么；不能只写背景介绍。

【主角可见压力】
主角现在面对什么损失、限制、羞辱、危险、机会或倒计时？

【主角欲望】
主角此刻想要什么？长线想要什么？

【主角主动选择】
本章主角必须做出的选择是什么？

【阻力/对手】
谁或什么阻止主角轻易得到想要的东西？

【卖点证明】
本章用哪个场景证明本书核心卖点？

【信息投放】
必须让读者知道什么？哪些信息不能提前说？

【情绪/爽点/悬念回报】
本章读完，读者拿到什么短回报？

【章末钩子】
读者为什么要点下一章？下一章必须推进什么？

【风格约束】
旁白距离、视角、语言质感、角色声音注意事项。

【禁止事项】
本章不能写什么，不能提前解决什么，不能泄露什么。

【写后回写】
台账、风格手册、人物卡、世界规则、下一章任务需要更新什么？
```

## 5. 第一章必备七件事

### 5.1 主角或核心视角人物必须可见

第一章可以用反派、案件、灾难、旁观者开场，但读者必须很快知道核心视角是谁，或知道这段开场如何服务主角承诺。

风险：

- 开篇很热闹，但读者不知道该跟谁。
- 群像人物太多，读者记不住。
- 反派序章抢走主角期待。

### 5.2 本章必须有现在进行的压力

压力可以是外部危机，也可以是关系、身份、资源、名誉、时间、秘密、规则或心理损失。

风险：

- 只有设定介绍，没有当下问题。
- 主角像导游一样带读者看世界。
- 冲突发生在过去或未来，当下没有动作。

### 5.3 主角必须做出选择

选择不一定大，但必须让读者看见主角不是被剧情推着走。

风险：

- 系统自动发奖。
- 配角替主角解决。
- 巧合把主角送到下一关。
- 主角只震惊、吐槽、解释，不行动。

### 5.4 类型承诺必须可见

第一章要证明这本书是哪类体验：

- 爽文：让读者看见反制、落袋或即将反制的明确路径。
- 情绪线：让读者看见关系特殊性、情绪债或立场冲突。
- 悬疑：让读者知道核心问题是什么，并拿到可追线索。
- 喜剧：让读者看见人物化错位，而不是硬塞段子。
- 机制文：让读者看见机制如何改变选择，而不是只看规则说明。

### 5.5 信息投放必须服务场景

第一章只给读者读懂当前场景所需的信息。作者写作时需要知道的背景，不等于读者第一章需要知道。

风险：

- 世界观百科。
- 家族关系表。
- 长篇前世回忆。
- 金手指说明书。
- 平台标签和设定名词密集堆叠。

### 5.6 本章必须有短回报

第一章可以不解决大问题，但必须给读者一个短回报：

- 知道主角有本事。
- 看见对手压迫。
- 拿到一个反杀伏笔。
- 明白一个关系欠账。
- 得到一个信息差。
- 看到一个情绪触点。

### 5.7 章末必须有下一章欲望

章末钩子不是一句“更大的危机来了”。章末要让读者知道下一章必须看什么：

- 主角刚做选择，后果马上反弹。
- 对手出招，主角必须应对。
- 机制启动，但代价未明。
- 关系状态变化，但答案未落地。
- 悬念给出线索，但真相还差一步。

## 6. 第一章禁止事项

默认禁止：

- 用大段设定替代场景。
- 用主角自我介绍替代行动。
- 用梦境、预言、旁白史诗当作唯一开场。
- 第一章塞太多人物。
- 第一章提前解释完整世界规则。
- 第一章只写主角被动遭遇，没有选择。
- 第一章卖点和书名简介不一致。
- 第一章末只喊口号，不给具体下一步。
- 正文里夹作者说明、创作意图、任务单、自检或台账。

允许例外，但必须写明理由：

- 慢热文学向。
- 群像开场。
- 反派或案件开场。
- 极强氛围开场。
- 先声夺人的非主角序章。

例外仍要满足：

- 读者知道该期待什么。
- 场景有压力。
- 章节有短回报。
- 下一章有具体欲望。

## 7. 第一章生产流程

### 7.1 总控放行

总控 agent 检查项目许可、任务许可和输出许可。

输出：

- 可进入第一章。
- 只能补任务单。
- 只能做第一屏小样。
- 必须回到立项或项目初始化。

### 7.2 上下文封包

封包 agent 按 `docs/workflows/36-prompt-context-packaging.md` 提供最小上下文。

禁止：

- 把整个知识库塞给起草 agent。
- 把候选设定和已确认事实混在一起。
- 把竞品桥段当成本书事实。

### 7.3 场景设计

章节设计 agent 把任务单转成场景链。

输出：

- 场景目标。
- 阻力。
- 转折。
- 读者回报。
- 章末牵引。

不写完整正文。

### 7.4 正文起草

起草 agent 只输出读者可见正文。

正文要求：

- 按项目风格手册执行。
- 用具体场景承载信息。
- 让主角有动作、选择和后果。
- 不输出作者说明和自检。

### 7.5 写后回写

写完第一章后，另行输出：

- 台账更新项。
- 风格手册更新项。
- 世界规则更新项。
- 人物状态变化。
- 下一章任务建议。
- 待确认事项。

## 11. 自查清单

开写前：

- 目标读者是否明确？
- 主承诺是否能在第一章看见？
- 开场是否是场景，不是设定说明？
- 主角是否在第一章做选择？
- 阻力是否具体？
- 卖点证明是否可见？
- 信息投放是否只够读懂本章？
- 章末是否有具体下一步欲望？
- 风格手册和台账是否给到？

写完后：

- 正文里是否混入任务单或作者说明？
- 第一章是否兑现了包装承诺？
- 主角是否比开场多了新处境、新风险或新目标？
- 读者是否拿到短回报？
- 下一章必须推进什么是否清楚？
- 台账和风格手册更新项是否列出？

## 12. 来源与迁移说明

本规范综合以下来源形成：

- 本知识库已有的开篇留存、章法移动端体验、章节生产流水线、开篇 A/B、项目初始化、上下文封包、风格手册和连续性台账规则。
- Reedsy 关于小说开篇和 book hook 的建议：开篇要尽早建立人物欲望、冲突、风险和触发情节的事件；一句话 hook 可作为写作过程中的北极星。
- Writer's Digest 关于小说开篇目标的建议：开篇需要引入值得讲述的问题、吸引读者、建立故事规则，并为故事方向提供预告。
- Jane Friedman 关于第一场景的建议：开头常因试图过早讲太多背景、描述和内心独白而拖慢，作者需要删掉遮蔽行动的准备性文字。

迁移到单本小说项目时，不要整段复制本专题。应复制“第一章生产许可卡”“第 1 章任务单”和“第一章写后回写卡”，并填入该项目自己的已确认事实、候选设定、待确认问题和正文生产边界。
