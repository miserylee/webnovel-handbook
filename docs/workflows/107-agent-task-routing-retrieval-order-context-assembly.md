# 107. AI agent 任务路由、检索顺序与上下文装配专项

状态：已确认

适用范围：任何 AI agent 使用本知识库起草、续写、审稿、修订、立项、拆大纲、设计人物、处理反馈、维护台账、联网查资料、生成提示词、分配多 agent 任务或更新知识库时使用。尤其适用于“我该先读哪几篇”“这个反馈应该调用哪个专题”“上下文太多塞不下”“agent 总是直接写正文”“读了很多但没按任务输出”的情况。

核心结论：知识库越大，越不能靠 AI 记忆和全文扫描工作。AI agent 必须先判任务类型，再按最小必读链检索；先读总流程，再读专题；先装配上下文，再执行；先输出契约，再生成正文或审稿结论；写后把变化回写到正确位置。检索不是越多越好，而是“读对、读少、读到可执行”。

最小链路：用户任务 -> 任务类型 -> 阶段判断 -> 必读入口 -> 专题路由 -> 项目事实包 -> 风格/禁区/正典 -> 输出契约 -> 执行 -> 回写/复盘。

---

## 1. 为什么需要独立专题

本知识库已经有大量专题。没有路由规则时，AI agent 容易出现三种坏情况：

- 读得太少：直接起草正文，漏掉目标读者、台账、风格、禁区和章节目标。
- 读得太多：把整库塞进上下文，导致重点稀释、执行混乱、输出像综述。
- 读错顺序：先读技巧文，后补项目事实，导致正文漂亮但违背 canon 或主承诺。

已有第 35 篇负责 AI 创作总流程，第 36 篇负责上下文封包，第 38 篇负责章节生产交接，第 95 篇负责反馈症状路由。本篇负责更高一层的“查库导航”：任何任务先如何分诊、读哪些入口、如何控制上下文、输出什么、回写到哪。

## 2. 来源吸收与迁移边界

### 2.1 Diataxis：按用户需求组织文档

Diataxis 将文档按用户需求分成教程、操作指南、参考和解释四类。迁移到本知识库：AI agent 不应把所有文档都当同一种材料。流程文是操作指南，专题文是参考和检查表，案例文是解释和经验，模板文是执行表单。

### 2.2 Microsoft SharePoint IA：从用户任务规划导航

Microsoft Learn 的信息架构原则强调导航要从用户角度规划，让用户找到信息、完成任务并降低认知负担。迁移到本知识库：索引和路由不应只按文件编号，而要按“开书、起草、审稿、修订、反馈、完本、知识库维护”等任务组织。

### 2.3 NN/g 识别优先于回忆

NN/g 关于 recognition vs recall 的资料指出，识别比从零回忆更容易。迁移到 agent 使用：路由表应让 agent 在任务标签中选择，而不是靠模型想起全部专题名；输出模板也应给字段，减少自由发挥。

### 2.4 NIST RAG 定义

NIST 将 RAG 描述为模型结合独立检索系统或知识库，并把相关信息放入上下文中供模型回答。迁移到本项目：AI agent 使用知识库时，应把本地文档视为可检索权威资料，而不是依赖模型内存；同时要控制进入上下文的材料，只放与任务相关、状态清晰、可执行的信息。

## 3. 文档类型分层

| 类型 | 作用 | 典型文档 | 使用方式 |
| --- | --- | --- | --- |
| 总入口 | 判断任务和知识库范围 | `README.md`、`docs/00-index.md`、`docs/sources/01-source-inventory.md` | 新会话、上下文丢失或维护知识库时先读 |
| 流程文 | 告诉 agent 怎么工作 | `35`、`36`、`38`、`44`、`45`、`104` | 执行任务前读，决定步骤和输出 |
| 专题文 | 针对具体写作问题给规则 | `29`、`64`、`78`、`95` 等 | 按任务或症状调用，不全文堆叠 |
| 模板文 | 把规则变成表单 | `10` | 输出检查卡、任务单和审稿表时调用 |
| 来源文 | 记录资料来源和迁移边界 | `docs/sources/01-source-inventory.md` | 联网资料入库、追溯依据时调用 |
| 决策文 | 记录已确认规则和关键变更 | `docs/governance/99-decision-log.md` | 冲突处理、规则升级、知识库维护时调用 |
| 项目文档 | 单本小说的 canon 和当前状态 | 单本项目 README、台账、大纲、正文 | 起草和修订时优先于通用技巧 |

原则：通用知识不能覆盖项目 canon；专题规则不能替代用户当前指令；来源材料不能直接写成正文。

## 4. 任务类型总路由

### 4.1 新书立项

先读：`README.md`、`docs/00-index.md`、`docs/workflows/03-project-workflow.md`、`docs/workflows/43-commercial-project-greenlight-scorecard.md`。

按需读：`29`、`34`、`16`、`17`、`22`、`39`、`44`、`98`、`102`。

输出：目标读者、主承诺、市场位置、竞品边界、风险评分、是否立项、下一步验证样本。

禁止：只凭一个新鲜设定直接写第一章。

### 4.2 初始化单本小说项目

先读：`03`、`35`、`44`、项目 `AGENTS.md` 或用户指定结构。

按需读：`07`、`36`、`37`、`100`、`106`。

输出：项目目录、项目简报、市场策略、核心概念、风格手册、台账初版、待确认清单。

禁止：把通用知识库整段复制进单本项目。

### 4.3 第一章生产

先读：`45`、`22`、`29`、`37`、`38`。

按需读：`32`、`39`、`64`、`72`、`80`、`87`、`101`、`105`。

输出：第一章任务单、开篇场景、章末钩子、正文草稿或审稿结论。

禁止：任务单不齐、主承诺不清、风格手册缺失时直接起草。

### 4.4 续写下一章

先读：项目台账、上一章结果、本章任务、风格手册、`38`、`07`、`37`。

按需读：`05`、`50-56`、`63`、`68`、`72`、`79`、`91`、`100`。

输出：本章场景目标、冲突、转折、回报、正文、台账变化项。

禁止：复述上一章、重写 canon、跳过写后回写。

### 4.5 章节审稿

先读：用户审稿要求、正文片段、项目目标、`08`、`95`。

按需读：按问题调用：水读 `62/73/85`，不爽读 `79/86/68`，假读 `72/90/100`，人物不活读 `11/74/77/80/87`，对话尬读 `75`，看不懂读 `78/88/91/106`，AI 味读 `76/11/70`。

输出：发现优先、问题层级、证据、修订建议、保留资产。

禁止：先润色句子，后判断结构问题。

### 4.6 修订或重写

先读：`33`、审稿结论、项目 canon、有效资产清单。

按需读：对应专题和 `38`。

输出：修订范围、不可改项、改动方案、修订正文或差异说明、台账回写项。

禁止：未确认范围就大改已发布剧情。

### 4.7 读者/编辑/数据反馈处理

先读：反馈原文、数据位置、`41`、`42`、`95`。

按需读：`18`、`40`、`65`、`101`、具体症状专题。

输出：反馈证据强度、目标读者匹配度、问题层级、优先调用专题、观察指标。

禁止：把读者处方直接当修订命令。

### 4.8 完本、卷末或项目复盘

先读：`24`、`55`、`104`、项目数据、评论、台账。

按需读：`18`、`41`、`65`、`103`。

输出：原始目标、实际结果、有效资产、失败模式、可迁移经验、下一本闸门。

禁止：把剧情总结当复盘。

### 4.9 知识库维护

先读：`README.md`、`00`、`01`、`99`、本篇。

按需读：相关专题、外部来源、`102`。

输出：新增/更新文档、来源登记、模板卡、决策记录、冲突说明。

禁止：只在聊天里留下稳定规则，不写入专题文档。

## 6. 上下文装配规则

### 6.1 最小上下文包

任何执行型 agent 接收的上下文应包括：

```text
任务目标：
任务阶段：立项/起草/审稿/修订/复盘/维护
用户当前指令：
项目 canon：只放与本任务相关的已确认事实
候选/待确认：明确标记，不混入 canon
风格约束：旁白、角色声音、称谓术语、禁用表达
必读专题：列文件名和调用理由
禁区：不能改、不能写、合规风险、原创风险、敏感内容风险
输出契约：必须输出什么，不输出什么
回写要求：写到哪里，哪些变化需要登记
```

### 6.2 不该塞进上下文的内容

- 整个知识库全文。
- 无关章节和旧设定。
- 未标状态的脑洞。
- 长段竞品原文。
- 读者猜测和 canon 混写。
- 只为“保险”加入的十几个专题。
- 与本任务无关的模板。

### 6.3 文档调用上限

普通单章任务：3 到 6 篇专题通常足够。

复杂大纲/复盘/知识库维护：可以更多，但必须说明每篇文档的调用理由。

如果超过 8 篇仍无法执行，先压缩为上下文包，不要让起草 agent 直接读全部。

## 7. 输出契约

不同任务的输出必须不同。

| 任务 | 应输出 | 不应输出 |
| --- | --- | --- |
| 起草 | 读者可见正文；必要时附台账回写项 | 自夸、自检长篇、审稿报告混入正文 |
| 审稿 | 问题、证据、风险、优先级、保留资产 | 直接重写全文，除非用户要求 |
| 修订 | 修订后的正文或差异说明 | 未说明范围的大改 |
| 立项 | 评分、风险、验证方案 | 直接批量正文 |
| 反馈处理 | 症状路由、证据强度、修订策略 | 按评论原话立刻改 |
| 知识库维护 | 专题文档、来源登记、模板、决策记录 | 只聊天总结 |
| 复盘 | 目标对照、证据、经验、下一本闸门 | 剧情复述 |

## 8. 冲突处理顺序

当多个文档或资料冲突，按以下顺序处理：

1. 用户当前明确指令。
2. 单本项目已确认 canon 和已发布正文。
3. 本知识库已确认规则。
4. `docs/governance/99-decision-log.md` 中的最新决策。
5. 来源清单中的已登记资料。
6. 外部资料的最新官方信息。
7. 候选设定、临时脑洞、读者猜测和模型推断。

如果冲突影响正文事实、合规、主承诺或已发布内容，先停下来标注冲突，不静默选择。

## 9. 维护知识库时的新增专题判断

新增专题前问：

- 是否已有专题能覆盖？
- 新问题是否跨多个既有专题，且需要统一入口？
- 是否有联网来源或旧项目经验支持？
- 是否能输出原则、用法、风险、检查问题和模板？
- 是否会让 agent 更容易执行，而不是只增加文档数量？

如果只是某专题中的小节，不新增独立文档，改为补充现有文档或模板。

## 11. 最小规则

时间很少时，至少执行：

1. 先判任务类型，再读专题。
2. 新会话或上下文丢失后先读 README、00、01。
3. 单本项目任务中，项目 canon 高于通用技巧。
4. 起草前必须有任务单、风格约束和相关台账。
5. 审稿前必须先判问题层级，不先润色。
6. 反馈先当症状，不当处方。
7. 上下文包只放与任务相关、状态清楚、可执行的信息。
8. 超过 8 篇专题时先压缩，不直接塞给起草 agent。
9. 写作变化要回写台账、风格手册、来源或决策记录。
10. 稳定结论不能只留在聊天里。
