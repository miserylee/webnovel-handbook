# 64. 单本小说项目文档粒度与短文档维护规范

状态：已确认
用途：规定单本小说项目如何维护资料、台账、大纲和索引，避免把项目养成难以更新的巨型文档。

## 1. 核心原则

单本小说项目优先采用短文档、多索引、可替换的维护方式。

允许索引链路变长，不鼓励把全部设定、大纲、章节台账和讨论记录塞进少数巨型文件。

执行含义：

- 一个文件只解决一个主要问题。
- 一个实体尽量一个文件。
- 总览文件只保留结构、状态、入口和读取顺序。
- 细节文件下沉到对应目录，由索引串联。
- 新增结论优先写入对应短文档，不默认追加到总览末尾。
- 如果正文和大纲冲突，以正文和章节台账为准，再更新索引和相关短文档。

## 2. 推荐文件粒度

| 文件类型 | 建议职责 | 建议长度 |
|---|---|---|
| 项目入口 | 项目状态、读取顺序、重要路径 | 500-1500 字 |
| 大纲总览 | 四阶段、卷结构、主线索引 | 1000-3000 字 |
| 单卷纲 | 本卷目标、阶段节点、卷末状态 | 1500-3000 字 |
| 前十章拍表 | 章节功能和承接点 | 1000-2500 字 |
| 人物卡 | 人设、关系、声音、当前状态 | 800-2000 字 |
| 势力/地区/体系卡 | 单一实体规则和剧情功能 | 800-2000 字 |
| 单章台账 | 本章事实、包袱、承接点 | 300-800 字 |
| 包袱卡 | 单个或同类伏笔的埋设和回收 | 300-1000 字 |
| 决策记录 | 一次稳定结论和影响范围 | 300-1000 字 |

超过 3000 字的文档应考虑拆分。

超过 6000 字的文档默认需要拆分，或至少建立子索引并停止继续追加细节。

## 3. 推荐目录方式

单本小说项目可以采用类似结构：

```text
reference/
  资料库入口.md
  项目/
    项目状态.md
    文档维护规则.md
    全书结构索引.md
  人物/
    主角.md
    队友A.md
  势力/
  地区/
  体系/
  连载台账/
    台账使用规则.md
    章节/
      第001章.md
      第002章.md
    包袱/
      F001-某伏笔.md
    主线/
      主线-明线.md
      主线-暗线.md
```

目录不必完全照抄，但必须满足：入口能找到、文件职责单一、后续 agent 知道什么时候读哪个文件。

## 4. 索引链路规则

允许入口链路变长，但链路必须清楚。

推荐链路：

```text
资料库入口 -> 项目状态/总索引 -> 主题索引 -> 实体或台账短文件
```

禁止：

- 新增文件没有任何入口引用。
- 总览文件复制所有细节，导致细节和子文件互相冲突。
- 同一事实在多个大文件里反复维护。
- 旧讨论、废稿复盘、正文事实混在同一文件里。

## 5. 台账拆分规则

连载台账尤其要短。

优先拆分为：

- 章节节奏台账：可以按卷或按章节拆。
- 关键节点台账：可以按节点编号拆。
- 包袱与回收台账：长期项目建议一伏笔一卡，或同类伏笔一文件。
- 主线支线推进台账：一条线一个文件。
- 版本规则台账：正稿、草稿、废稿、作者改稿单独说明。

如果一个台账需要频繁追加，优先拆成目录，不要一直往一个总表追加。

## 6. 何时拆分

出现以下情况，应拆文档：

- 文件已经超过 3000 字，而且包含多个主题。
- agent 需要在文件里搜索很久才能找到当前事实。
- 新旧结论混在一起，导致后续续写误判。
- 同一文件同时承担大纲、台账、讨论记录、设定卡职责。
- 作者频繁改稿，旧章节事实和新章节事实混在同一段落里。
- 文件末尾不断追加“补充”，前文已经不再准确。

## 7. 新建文件必须有入口

每次为单本小说项目新建文件，必须同步完成至少一项：

- 更新项目资料入口。
- 更新对应目录索引。
- 更新卷目录或章节目录。
- 更新台账使用规则。
- 在相关实体文件里加反向引用。

交付时必须说明新文件从哪里进入、什么场景需要读取。

## 8. 与正稿事实优先规则的关系

短文档策略服务于正稿事实优先。

正文事实、设定规划、讨论过程、废稿复盘、章节台账必须分开放，避免 agent 把未发生内容当成已发生事实。

当作者改稿导致节奏变化时，只更新相关短文件和索引，不需要重写巨型总文档。

## 9. Agent 执行检查

维护单本小说项目时，agent 自查：

- 我是否正在把多个主题塞进一个文件？
- 这个文件以后会不会频繁追加？如果会，是否应该拆目录？
- 入口文件是否只做索引，而不是承载全部细节？
- 新增文件有没有被索引？
- 旧事实、计划、废稿和正稿事实有没有分开？
- 这次修改是否会让后续 agent 更容易找到当前事实？

## 10. 与连载看板的关系

文档粒度只解决“资料怎么拆”，不解决“任务怎么流转”。

当单本小说项目进入持续连载、多人/多 agent 协作、频繁改稿或读者反馈阶段时，应联动 [65. 连载项目看板、变更控制与完成定义](./65-serial-project-kanban-change-control.md)。

第 65 篇负责规定任务卡、WIP 限制、完成定义、变更分级和阻塞处理；本篇继续负责文件粒度、索引链路和台账拆分。
