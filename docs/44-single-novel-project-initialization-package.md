## 1. 核心原则：初始化不是建空壳，而是建立事实来源

单本小说项目初始化的目的不是“有一堆文件”，而是让后续 AI agents 知道：

- 这本书写给谁。
- 这本书承诺什么。
- 哪些事实已经确认。
- 哪些想法只是候选。
- 哪些问题还待确认。
- 正文应该从哪里开始。
- 写完一章后要回写哪里。

一句话原则：初始化项目时，目录结构是骨架，项目简报是方向，台账是连续性，风格手册是声音，章节任务是进入正文的闸门。

高危错误：

- 创建目录后立刻写正文。
- 把通用写作方法复制到单本项目里，导致项目文档膨胀但没有项目事实。
- 把 brainstorm、用户随口想法、竞品桥段、读者猜测都写成已确认设定。
- 正文文件夹里混入大纲、自检、作者说明和修订笔记。
- 没有台账和风格手册就让多个 agent 接力写长篇。

## 2. 进入初始化的前置条件

满足以下条件，才能创建正式单本小说项目：

- 已完成或暂定 `docs/43-commercial-project-greenlight-scorecard.md` 的立项判断。
- 至少明确目标平台或候选平台。
- 至少明确目标读者或候选目标读者。
- 至少明确主承诺。
- 至少有一个可验证的第一章入口。
- 没有未处理的高风险合规或版权问题。

如果以上条件不足，只能创建“候选项目夹”或 notes，不应建立完整长篇项目，也不应进入章节生产。

初始化前必须写明：

- `立项状态：通过立项 / 候选立项 / 暂缓测试 / 重构 / 淘汰`
- `可进入范围：项目框架 / 小样测试 / 第一章试写 / 前三章试写 / 完整章节生产`
- `禁止动作：不得批量起草 / 不得发布 / 不得把候选设定写成正文事实`

## 3. 标准目录结构

单本小说项目默认使用以下结构。文件名保持稳定，正文和文档内容使用中文。

```text
project/
├── AGENTS.md
├── .gitattributes
├── .gitignore
├── README.md
├── novel-notes.md
├── docs/
│   ├── 00-index.md
│   ├── 01-market-strategy.md
│   ├── 02-core-concept.md
│   ├── 03-world-rules.md
│   ├── 04-characters.md
│   ├── 05-plot-architecture.md
│   ├── 06-volume-outlines.md
│   ├── 07-writing-guidelines.md
│   ├── 08-style-bible.md
│   ├── 11-story-ledger.md
│   └── 99-decision-log.md
└── manuscript/
    ├── README.md
    └── vol-01-opening/
        ├── README.md
        └── outline.md
```

可选扩展：

- `docs/09-feedback-log.md`：有读者试读、编辑反馈或评论复盘时再建。
- `docs/10-publication-ops.md`：进入发布、推荐、上架或连载运营后再建。
- `docs/12-risk-review.md`：高合规、高版权、高专业资料项目可提前建。
- `assets/`：只在确实需要封面参考、地图、时间线图或素材授权记录时建立。

不要默认创建：

- `final/`
- `v2/`
- `old/`
- `backup/`
- `随便记.md`
- `章节修正版.md`

版本脉络写进决策记录或修订记录，不靠文件名堆版本。

## 5. 状态标签规则

项目初始化时必须使用三种状态：

- `已确认`：用户明确确认、立项卡确认、项目决策记录确认，或已经写入正文成为事实。
- `候选`：可用想法、待比较方案、尚未验证的设定、可能的角色关系。
- `待确认`：缺少用户决定、平台规则、资料来源、数据证据或上下文材料。

硬规则：

- `候选` 不得进入正文成为事实。
- `待确认` 不得作为章节任务的前提。
- `已确认` 变更必须写入 `docs/99-decision-log.md`。
- 正文一旦使用某个设定，必须同步进入台账或相关项目文档。
- 多个 agent 接力时，上下文包必须明确区分三种状态。

危险写法：

- `女主可能是重生，也可能不是。` 放进正文任务。
- `平台应该适合番茄。` 写成项目已确认平台。
- `读者大概喜欢这个。` 写成市场结论。

正确写法：

- `候选：女主重生设定，尚未决定是否进入正文。`
- `待确认：平台为番茄或起点，需根据目标读者和更新计划选择。`
- `已确认：主承诺是“女主凭借前世账本反杀宗门债务局”。`

## 6. 初始化阶段流程

### 6.1 阶段 A：立项资料转入项目

输入：

- 商业级开书立项评分卡。
- 市场调研和竞品拆解。
- 读者承诺与卖点设计。
- 平台频道判断。
- 合规风险初筛。

输出：

- `docs/01-market-strategy.md`
- `docs/02-core-concept.md`
- `docs/99-decision-log.md`

门槛：

- 主承诺、目标读者、候选平台、第一章验证点必须写入项目。
- 低分项和风险项必须保留，不得在初始化时抹掉。

### 6.2 阶段 B：核心资产成型

输入：

- 主角、对手、核心关系、机制、世界规则、长线压力。

输出：

- `docs/03-world-rules.md`
- `docs/04-characters.md`
- `docs/05-plot-architecture.md`

门槛：

- 主角必须有欲望、压力、主动选择和第一章行动。
- 机制必须有触发、限制、代价、失败方式和升级路径。
- 对手或阻力必须能持续制造章节冲突。

### 6.3 阶段 C：声音与连续性起步

输入：

- 风格方向、视角、旁白距离、角色声音、称呼术语、初始伏笔。

输出：

- `docs/07-writing-guidelines.md`
- `docs/08-style-bible.md`
- `docs/11-story-ledger.md`

门槛：

- 至少确定旁白距离、视角、主角声音底线和不能改平的文本资产。
- 台账至少记录初始事实、角色起点、世界规则和活跃伏笔。

### 6.4 阶段 D：第一卷和前 10 章任务

输入：

- 第一卷目标、前 10 章落袋、开篇留存模型。

输出：

- `docs/06-volume-outlines.md`
- `manuscript/vol-01-opening/README.md`
- `manuscript/vol-01-opening/outline.md`

门槛：

- 前 10 章至少有“入场、证明、入轨、换招、回报”。
- 第一章必须有场景、目标、阻力、转折、卖点证明和章末钩子。

### 6.5 阶段 E：第一章生产许可

只有以下内容齐备，才能进入第一章正文：

- 当前阶段和下一步写在 `docs/00-index.md`。
- 主承诺写在 `docs/02-core-concept.md`。
- 目标读者和平台写在 `docs/01-market-strategy.md`。
- 主角第一章行动写在 `docs/04-characters.md` 或章节任务中。
- 世界/机制第一章所需规则写在 `docs/03-world-rules.md`。
- 第一章任务写在 `manuscript/vol-01-opening/outline.md`。
- 风格底线写在 `docs/08-style-bible.md`。
- 初始台账写在 `docs/11-story-ledger.md`。

缺一项时，只能补项目文档，不得进入正文。

## 8. 第一章任务单最小字段

第一章任务单必须放在 `manuscript/vol-01-opening/outline.md` 或对应卷纲中，不放进 `ch001.md`。

```markdown
## 第 1 章任务单

状态：候选 / 已确认 / 待确认

【读者承诺】
【本章目标】
【开场场景】
【主角压力】
【主角主动选择】
【阻力/对手】
【卖点证明】
【信息投放】
【情绪或爽点回报】
【章末钩子】
【不能写】
【需要台账记录】
【需要风格手册记录】
```

进入正文前，第一章任务单必须至少有：

- 读者承诺。
- 开场场景。
- 主角主动选择。
- 阻力或对手。
- 卖点证明。
- 章末钩子。

## 10. 红线

- 不把项目初始化等同于创建空文件。
- 不把通用写作方法整段复制进单本项目。
- 不把候选想法写成已确认事实。
- 不把竞品桥段、读者猜测或临时脑洞写进正文任务。
- 不在没有第一章任务单时写 `ch001.md`。
- 不在正文文件里放大纲、自检、审稿意见或台账。
- 不创建 `final`、`v2`、日期版章节文件。
- 不让多个 agent 在没有台账和风格手册的情况下接力续写。
- 不跳过合规、版权和资料风险。

## 11. 自查清单

初始化完成后检查：

- 是否写明项目当前阶段？
- 是否写明下一步？
- 是否区分已确认、候选和待确认？
- 是否有目标读者和主承诺？
- 是否有第一章验证点？
- 是否有台账初始状态？
- 是否有风格手册初始状态？
- 是否有第一章任务单？
- 是否写明不能进入正文的情况？
- 是否把关键决策写入 `docs/99-decision-log.md`？

进入第一章前检查：

- 第一章是否有场景而不是设定说明？
- 主角是否在第一章主动选择？
- 类型卖点是否在第一章可见？
- 章末是否有下一步欲望？
- 需要的世界规则是否已确认？
- 需要的角色事实是否已确认？
- 风格手册是否给到起草 agent？
- 台账是否给到起草 agent？

## 12. 来源与迁移说明

本规范综合以下来源形成：

- 本地 `novel` skill 的项目结构、文件职责、正文文件规则、状态标签和 Git 管理原则。
- 本知识库已有的项目流程、总控路线图、提示词封包、风格手册、章节流水线、连续性台账和立项评分规则。
- Atlassian Project Kickoff 关于项目启动时明确愿景、任务、成功测试、角色、风险和下一步的项目管理方法。
- Reedsy 关于小说规划中主角、冲突、设定、研究和大纲的基础作用。
- Writer's Digest 关于小说蓝图中标题、类型、视角、主角、冲突、目标、动机、设定和故事火花的检查清单。

迁移到单本小说项目时，只复制该项目需要的最小文档结构、初始化报告和第一章任务单。不要把本知识库的通用方法整段搬进项目文档；项目文档只记录项目专属事实、确认决定、候选方向、待确认问题和连续性台账。
