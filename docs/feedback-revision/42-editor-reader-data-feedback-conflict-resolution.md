# 42. 编辑反馈、读者反馈与数据反馈冲突处理规范

状态：已确认。

用途：给 AI agents 在多方反馈互相矛盾时使用。适用于编辑建议、匿名读者试读、平台评论、后台数据、作者直觉、项目主承诺之间发生冲突的场景。目标不是简单判断“听谁的”，而是把反馈来源、证据强度、作品阶段、主承诺、商业风险和修改成本放在同一张决策表里，形成可解释、可复盘的取舍。

## 1. 核心原则：反馈冲突不是噪音，是决策材料

商业连载小说经常同时收到这些信号：

- 编辑说开篇不够直接，但读者群觉得铺垫有味道。
- 评论喜欢某个角色，但数据在该角色支线后下滑。
- 数据上涨，但编辑认为主线方向会透支后续。
- 目标读者催更，非目标读者骂雷。
- 作者觉得这一段很重要，读者觉得水。

AI agent 不能用“少数服从多数”解决这类问题。不同反馈来源回答的是不同问题：

- 编辑反馈常回答：这本书是否更适合商业投放、签约、频道、推荐和长期结构。
- 读者反馈常回答：读者当下是否爽、困惑、心疼、愤怒、想追或想弃。
- 数据反馈常回答：真实行为有没有继续发生，比如点击、读完、追读、收藏、订阅。
- 作者或项目主承诺回答：这本书不能牺牲什么核心资产。
- 平台合规回答：什么不能发布，什么必须降风险。

因此，反馈冲突要转成决策表，而不是争论谁更懂。

## 2. 反馈来源分工

### 2.1 编辑反馈

适合判断：

- 类型定位是否清楚。
- 开篇是否有商业入口。
- 主线是否可持续。
- 卖点是否能包装。
- 当前问题是否会影响签约、推荐、上架或完本。
- 修改成本是否值得。

局限：

- 编辑可能从平台、频道和商业效率出发，不一定保留所有个性化风格。
- 编辑建议可能是方向性，不一定是具体改法。
- 不同编辑的偏好和平台策略可能不同。

处理方式：

- 先提取编辑指出的真实问题。
- 再判断编辑给出的改法是否必须照做。
- 如果编辑反馈涉及签约、推荐、合规、平台规则，权重上调。
- 如果编辑反馈只涉及风格偏好，需要和目标读者、主承诺、数据交叉验证。

### 2.2 读者反馈

适合判断：

- 读者是否看懂。
- 读者是否被爽点、情绪、悬念、关系、梗点带动。
- 哪些角色、桥段、台词、关系是有效资产。
- 哪些毒点让读者退出。
- 哪些承诺没有兑现。

局限：

- 读者能准确描述感受，但不一定能提出正确改法。
- 评论区声音会被情绪、站队、玩梗、群体气氛影响。
- 非目标读者的反馈可能会把作品推向错误方向。

处理方式：

- 先用 `docs/feedback-revision/41-reader-comment-coding-feedback-clustering.md` 做标注和聚类。
- 区分问题感受和改法建议。
- 目标读者、付费读者、追更读者权重高于路人围观。
- 正反馈要同步进入有效资产库。

### 2.3 数据反馈

适合判断：

- 真实行为是否支持继续阅读。
- 哪个阶段发生掉点。
- 调整后是否有效。
- 读者嘴上和脚上是否一致。

局限：

- 数据告诉你发生了什么，不直接告诉你为什么。
- 数据可能受推荐、活动、更新时间、样本量、平台口径影响。
- 单日波动不能直接作为方向判断。

处理方式：

- 先用 `docs/feedback-revision/18-reader-data-feedback-diagnosis.md` 定位读者路径。
- 再用 `docs/feedback-revision/40-serial-data-tuning-loop.md` 设置观察窗口。
- 不把缺失数据编造成结论。
- 不在多个变量同时变化时强行归因。

### 2.4 作者直觉与项目主承诺

适合判断：

- 作品最想提供什么核心体验。
- 哪些人物、关系、机制、风格不能被改平。
- 哪些商业建议会伤害作品辨识度。
- 哪些读者建议其实违背目标类型。

局限：

- 作者可能过度保护个人偏好。
- AI agent 可能把“项目主承诺”误解成拒绝修改的借口。

处理方式：

- 主承诺必须写在项目简报、风格手册或承诺链里。
- 不能只说“作者喜欢”，要说明这个资产服务哪个读者承诺。
- 如果主承诺与真实读者行为长期冲突，必须回到项目定位复盘。

### 2.5 合规与平台规则

适合判断：

- 是否能发布。
- 是否需要删改、降风险、暂停上线。
- 是否涉及版权、隐私、未成年人、仇恨、色情、暴力、现实伤害等风险。

处理方式：

- 合规风险优先级最高。
- 只要涉及正式发布、签约、商业化、授权或现实争议，必须重新确认最新平台规则。
- 合规问题不能用“读者喜欢”抵消。

## 3. 冲突类型表

| 冲突 | 常见表现 | 优先动作 |
|---|---|---|
| 编辑要快，读者喜欢慢铺垫 | 编辑说开篇不够商业，核心读者说有味道 | 保留有效氛围，前置主角目标和类型承诺 |
| 读者喜欢，数据下滑 | 评论热闹但追读掉 | 判断是少数核心读者狂热，还是支线讨论消耗主线 |
| 数据好，编辑提醒后患 | 当前涨，但结构透支或人设风险 | 不急改已发布内容，记录风险并在后文补约束 |
| 读者分裂 | 一派爽，一派雷 | 按目标读者、主承诺和合规风险分层 |
| 作者想保留，反馈说水 | 作者认为是重要铺垫 | 检查铺垫是否改变状态，必要时压缩为有冲突场景 |
| 编辑建议和主承诺冲突 | 建议改成更热门套路 | 判断是否变成另一本文；可做小样验证，不直接推翻 |
| 数据缺失但反馈尖锐 | 没后台，评论骂得狠 | 标为候选风险，先文本诊断和小范围试读 |
| 数据下滑但评论无感 | 没人骂，但追读掉 | 优先查节奏、更新、承诺兑现和章节末牵引 |
| 合规风险和读者喜好冲突 | 读者爱看但平台可能卡审 | 进入合规审核，不能用热度抵消风险 |

## 4. 决策优先级：先处理不可逆风险

按以下顺序判断：

1. 合规与发布风险：违法、违规、版权、隐私、平台审核、商业化风险。
2. 核心承诺风险：作品是否背离目标读者、类型承诺、主角能动性、关键关系或长线引擎。
3. 行为数据风险：点击、读完、追读、收藏、订阅、完读是否持续受损。
4. 高频目标读者反馈：目标读者是否持续表达同一痛点。
5. 编辑结构反馈：是否影响签约、推荐、上架、长线稳定。
6. 正反馈资产：哪些不能在修订中误伤。
7. 修改成本：小修、中修、大修、重写、暂停发布各自代价。
8. 可测试性：能否用 A/B、盲评、观察窗口或后文补偿验证。

注意：优先级不是永远听第一个来源，而是先处理最不可逆的风险。合规风险必须先处理；商业结构风险要严肃处理；风格偏好和口味分歧要回到目标读者与主承诺。

## 5. 决策动作库

### 5.1 采纳

适用：

- 多个证据同向。
- 问题触及核心承诺、合规、留存或付费。
- 修改成本可控。

写法：

- `采纳原因：编辑反馈、目标读者反馈和数据同向。`
- `执行范围：第 X 章前段微修 / 后文补偿 / 包装调整。`
- `观察窗口：后续 X 章或 X 天。`

### 5.2 转译采纳

适用：

- 反馈指出的问题成立，但给出的改法会伤害作品。

例子：

- 读者说“把男主删了”，实际问题可能是男主缺少补偿动作。
- 编辑说“开篇砍设定”，实际问题可能是设定没有场景化。
- 数据掉在支线后，实际问题可能不是删支线，而是支线没有改变主线状态。

写法：

- `不直接采用建议，但采纳其背后的问题。`
- `读者问题：`
- `转译后的写作动作：`

### 5.3 暂缓观察

适用：

- 数据样本不足。
- 推荐、包装、更新节奏刚变化。
- 反馈强烈但尚未聚类。
- 修改会影响大量已发布章节。

写法：

- `状态：观察中。`
- `本轮不新增变量。`
- `观察到第 X 章或第 X 天。`

### 5.4 小样测试

适用：

- 两种方向都有道理。
- 冲突发生在开篇、简介、标签、章节顺序、角色出场方式。
- 还没大规模发布，或可以用盲评验证。

写法：

- `测试目标：验证哪种版本更能兑现同一个主承诺。`
- `控制变量：`
- `评估标准：`
- `不得比较无关方向。`

### 5.5 分层保留

适用：

- 核心读者喜欢，泛读者不适。
- 反馈不是要删除资产，而是要降低误读成本。

动作：

- 保留核心资产。
- 调整表达入口。
- 增加动机、边界、后果或补偿。
- 不把锋芒改成平均口味。

### 5.6 不采纳

适用：

- 反馈来自非目标读者且违背主承诺。
- 反馈只表达个人口味，没有文本证据。
- 建议会破坏有效资产。
- 单条反馈没有数据或聚类支持。

不采纳必须写原因，不能只写“忽略”。

### 5.7 进入重写或暂停

适用：

- 核心承诺错误。
- 开篇与包装完全错位。
- 主角不可追随。
- 合规风险无法局部解决。
- 已发布内容与后续结构发生不可修复矛盾。

动作：

- 进入 `docs/feedback-revision/33-revision-rewrite-version-workflow.md`。
- 涉及合规则先进入 `docs/safety-research/20-content-compliance-risk-review.md`。

## 6. 冲突反馈决策流程

### 6.1 稳定现场

先暂停“边看边改”。把反馈、数据、编辑意见、更新时间、包装变化、章节位置收集到同一表里。

输出：

- 发生了什么。
- 哪些来源互相冲突。
- 当前是否有合规或发布风险。
- 哪些章节或资产不能误伤。

### 6.2 拆分来源

把每条反馈拆成：

- 问题感受。
- 具体证据。
- 建议动作。
- 反馈来源。
- 读者或编辑的可能立场。

不要把“问题感受”和“建议动作”混在一起。

### 6.3 回到主承诺

判断这个反馈是否与项目主承诺相关：

- 支持主承诺。
- 伤害主承诺。
- 与主承诺无关。
- 要求作品改成另一个主承诺。

要求改成另一个主承诺的反馈，不直接采纳。

### 6.4 查证据同向性

判断编辑、读者、数据、文本诊断是否同向：

- 四者同向：高优先级修改。
- 三者同向：优先处理，保留观察窗口。
- 两者同向：候选判断，优先做低成本动作。
- 单一来源：不直接大改，除非涉及合规或核心毒点。

### 6.5 选动作

从动作库中选择：

- 采纳。
- 转译采纳。
- 暂缓观察。
- 小样测试。
- 分层保留。
- 不采纳。
- 重写或暂停。

### 6.6 写入记录

所有冲突反馈决策都要写：

- 冲突点。
- 采纳或不采纳理由。
- 本轮动作。
- 不改清单。
- 观察窗口。
- 复盘日期或章节。

## 7. 判断矩阵

| 编辑 | 目标读者 | 数据 | 文本诊断 | 建议 |
|---|---|---|---|---|
| 负 | 负 | 负 | 负 | 立即处理，可能需要中修或重写 |
| 负 | 正 | 正 | 中性 | 保留有效资产，检查是否只是编辑风格偏好 |
| 负 | 正 | 负 | 负 | 读者正反馈可能是少数资产，转译采纳编辑问题 |
| 正 | 负 | 负 | 负 | 优先处理读者和数据，编辑反馈可能漏掉阅读痛点 |
| 正 | 正 | 负 | 负 | 查更新、推荐、包装和样本；不立刻推翻正文 |
| 正 | 负 | 正 | 中性 | 判断是否非目标读者负评，不急改 |
| 中性 | 正 | 负 | 负 | 评论热但追读弱，防止支线或争议消耗主线 |
| 中性 | 负 | 正 | 正 | 负评可能是少数口味，不为消除噪音大改 |
| 负 | 中性 | 缺失 | 负 | 可做小修或小样测试，不直接大改 |
| 冲突 | 冲突 | 缺失 | 中性 | 回到主承诺，先补数据或盲评 |

说明：

- `正` 表示该来源支持当前文本或方向。
- `负` 表示该来源指出问题。
- `中性` 表示没有明确证据。
- `缺失` 表示没有可靠数据。

## 8. 常见场景处理

### 8.1 编辑让砍铺垫，读者说氛围好

不要直接砍掉氛围。先检查铺垫是否同时承担目标、冲突、信息、关系或情绪任务。

动作：

- 保留最有辨识度的氛围资产。
- 删除纯说明。
- 提前主角选择和类型承诺。
- 把静态铺垫改成有阻力的场景。

### 8.2 读者说角色可爱，数据在角色支线后掉

不要直接删角色。先判断掉点是角色本身，还是支线没有改变主线状态。

动作：

- 保留角色高记忆点。
- 缩短无状态变化的支线。
- 让角色支线带来资源、关系、信息、风险或主线选择。

### 8.3 编辑说主角不够主动，评论区还在夸爽

短期爽不等于长线稳。检查主角是否靠系统、巧合、配角替他赢。

动作：

- 保留爽点结果。
- 改赢法，让主角承担判断、代价和风险。
- 后文增加主动选择，而不是只加旁白解释。

### 8.4 数据好，但少数读者骂毒

不要因为数据好就忽略。先判断毒点是否涉及合规、价值观、核心人设或未来口碑。

动作：

- 若是合规或价值观风险，优先处理。
- 若是非目标读者口味，记录但不大改。
- 若是目标读者少数高风险反馈，用后文补偿或边界澄清。

### 8.5 读者要求按 CP 线走，编辑建议强化主线

判断本书主承诺是情绪关系还是主线升级。如果是双承诺，不能让两条线互相抢戏。

动作：

- 让 CP 互动改变主线资源、风险、选择或信息。
- 让主线事件改变关系状态。
- 避免把感情线写成番外感支线。

### 8.6 数据缺失，编辑和读者意见相反

不要假装有数据。先做低成本验证。

动作：

- 标记 `数据缺失：待确认`。
- 做目标读者盲评或小样测试。
- 若涉及已发布章节，先后文补偿，不直接大改。

## 9. Agent 分工

总控 agent：

- 判断是否存在反馈冲突。
- 指定必读文档。
- 输出决策表。

反馈整理 agent：

- 使用 `docs/feedback-revision/41-reader-comment-coding-feedback-clustering.md` 标注评论。
- 提取编辑反馈中的问题和建议。

数据诊断 agent：

- 使用 `docs/feedback-revision/18-reader-data-feedback-diagnosis.md` 定位读者路径。
- 使用 `docs/feedback-revision/40-serial-data-tuning-loop.md` 设置观察窗口。

文本诊断 agent：

- 判断反馈指向的文本问题是否存在。
- 标出有效资产和不能误伤项。

决策 agent：

- 用本专题的优先级、矩阵和动作库给出取舍。
- 明确采纳、转译采纳、暂缓、测试、不采纳或重写。

执行 agent：

- 按决策范围修订，不扩大修改。
- 修改后回写台账、风格手册或复盘记录。

## 13. 来源与迁移说明

本规范综合以下来源形成：

- 本知识库已有的审稿、读者反馈、数据诊断、修订流程、评论聚类和连载调参规则。
- Reedsy 关于 beta readers 和专业编辑的区分：读者反馈可以帮助发现阅读体验问题，但不能替代编辑的结构、市场和修订判断。
- Jane Friedman 关于 beta readers 的建议：反馈要基于目标读者和具体问题，作者需要选择合适读者并判断反馈适用性。
- Writer's Digest 关于处理编辑反馈的建议：面对编辑意见要先理解目的和问题，再决定采纳方式。
- UX 与用户研究中的三角验证思想：定量行为、定性反馈和专家判断要互相校验，不能单一来源定案。

迁移到单本小说项目时，不要整段复制本专题。应复制“反馈冲突决策卡”和该项目已经确认的冲突结论，把具体编辑意见、读者反馈、数据窗口和不采纳理由写入项目自己的决策记录。


## 与 66 号专题的衔接：互动不能用简单投票决定主线

当读者评论、读者群投票、编辑建议、后台数据和作者主承诺发生冲突时，除本专题外，还应联动 `docs/advanced-craft/66-author-note-reader-community-interaction.md`。

读者互动可以提供体感证据，但不能简单投票决定核心主线、角色生死、CP 归属、结局和价值观底线。若互动只是收集偏好，应明确“作为参考”；若承诺投票影响结果，就必须限定在番外、小剧场、周边表达或不破坏主承诺的范围内。
