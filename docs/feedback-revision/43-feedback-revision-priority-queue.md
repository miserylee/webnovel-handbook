# 43. 反馈修订优先级队列

状态：已确认  
用途：把评论聚类、编辑建议、后台数据、作者直觉和审稿问题转成可排队、可取舍、可观察的改稿任务。适用于“问题很多、不知道先改什么”“读者和编辑意见都对，但不能全改”“连载中需要边写边修”的场景。

## 0. 什么时候读

读本篇：

- 已经用 `41-reader-comment-coding-feedback-clustering.md` 把评论拆成观察点和主题，但还没决定修什么。
- 已经用 `42-editor-reader-data-feedback-conflict-resolution.md` 判断了冲突来源，但需要进入执行队列。
- `33-revision-rewrite-version-workflow.md` 开始修订前，待修项超过 3 个。
- 连载中同时出现掉点、差评、编辑提醒、作者新想法和正文 bug，需要保护更新节奏。

不必读本篇：

- 只有一个明确错字、病句或连续性硬 bug，直接修。
- 用户明确要求按某一条意见改，不需要 agent 排序。
- 还没有文本、数据或反馈证据，只是在空想优化。

## 1. 核心原则：反馈先入队，不直接上手改

读者反馈、编辑建议和数据异常都不是“命令”，而是候选改稿项。agent 先把它们转成同一种任务卡，再判断优先级。

一条反馈进入队列前，必须写清楚：

- 它指向哪个文本位置、章节区间、包装材料或连载动作。
- 它伤害的是逻辑、人味、主承诺、爽点回报、情绪、可读性、合规，还是更新信任。
- 它的证据来自单条评论、高频聚类、后台数据、编辑判断、作者确认，还是多源同向。
- 修它会不会误伤正反馈资产，比如人物锋利感、作者味、CP 张力、类型承诺或已经形成的读者期待。

## 2. 网文版 RICE：用四项排队，而不是凭焦虑改稿

借用 RICE 的结构，但不要把小说改稿算成精确数学题。

| 字段 | 网文转译 | 评分提示 |
|---|---|---|
| 影响范围 | 有多少目标读者、章节、入口材料或后续剧情会受影响 | 单章局部 / 前 3 章 / 整卷 / 全书承诺 / 平台发布 |
| 影响强度 | 不修会造成什么读者体感 | 小别扭 / 看不懂 / 不爽 / 不信人物 / 退出阅读 / 合规阻断 |
| 证据置信度 | 反馈是否可靠、同向、可复现 | 单一体感 / 高频评论 / 数据同向 / 编辑与文本诊断同向 / 作者确认 |
| 修改成本 | 要动多少正文、台账、细纲和发布节奏 | 一句话 / 一场戏 / 一章 / 多章回收 / 重构主线 |

推荐用粗粒度评分：

```text
优先分 = 影响范围 × 影响强度 × 证据置信度 ÷ 修改成本
```

注意：这个分数只用于迫使 agent 说清理由，不用于机械排名。合规风险、重大逻辑漏洞、正稿事实错位和平台发布阻断可以越过评分，直接进入最高优先级。

## 3. MoSCoW 转译：本轮必须、应该、可以、暂不做

同一个问题在全书层面重要，不代表本轮一定要改。连载修订要分时间盒。

| 桶 | 网文含义 | 典型处理 |
|---|---|---|
| Must | 不处理就不能发布、不能续写或会造成明显读者流失 | 本轮修；必要时暂停发布或退回细纲 |
| Should | 对留存、主承诺、人味或逻辑有明显帮助，但有替代路径 | 排进最近 1-3 章修订或后文补偿 |
| Could | 有优化价值，但不影响当前可读性和更新节奏 | 放观察池，等同类问题累计 |
| Won't this time | 口味偏好、低证据、会误伤资产，或会把作品改成另一本 | 记录原因，不进入本轮改稿 |

判断 Must 时问一句：

```text
如果这项不修，本章/本轮更新是否仍然能作为合格商品交付？
```

如果答案是否，才是 Must。不要把所有“我也想优化”都塞进 Must，否则队列会失去弹性，agent 会越改越乱。

## 4. 优先级队列

| 队列 | 含义 | 动作 |
|---|---|---|
| P0 阻断 | 合规、事实、逻辑、正稿错位、不可发布问题 | 立即修；修完再继续 |
| P1 本轮改 | 高影响、高置信、成本可控，且指向主承诺或留存 | 本轮改稿直接执行 |
| P2 近期补 | 重要但不必立刻动已发布正文 | 写入后续 1-5 章任务单或台账 |
| P3 观察 | 信号存在但证据不足 | 设置观察窗口，不大改 |
| Park 暂存 | 有价值但当前阶段不合适 | 进入想法池，定期复盘 |
| Reject 不采纳 | 与目标读者、主承诺或正反馈资产冲突 | 记录拒绝理由 |

队列不是越多越好。一次修订最多处理 1 个 P0、2-3 个 P1；其余进入 P2/P3/Park。否则修订会从“救问题”变成“把文本揉没味”。

## 5. 任务卡模板

```text
【反馈项】
【来源】评论聚类 / 编辑建议 / 后台数据 / 审稿诊断 / 作者确认
【指向位置】章节、段落、包装材料或连载动作
【问题类型】逻辑 / 人味 / AI 味 / 爽点 / 情绪 / 节奏 / 合规 / 包装 / 更新
【影响范围】局部 / 前 3 章 / 整卷 / 全书承诺 / 发布流程
【影响强度】轻微别扭 / 看不懂 / 不爽 / 不信人物 / 退出阅读 / 发布阻断
【证据置信度】低 / 中 / 高；说明证据是否同向
【修改成本】小修 / 中修 / 大修 / 重写 / 后文补偿
【保护资产】哪些有效点不能改坏
【队列】P0 / P1 / P2 / P3 / Park / Reject
【动作】本轮怎么改，或为什么暂不改
【观察窗口】改后看哪几章、哪些评论或哪些数据
```

## 6. 典型反例

反例一：单条读者说“女主烦”，agent 立刻把女主所有锋利台词改温柔。  
问题：低证据反馈误伤人物辨识度。应先看是否高频、是否目标读者、是否影响主承诺，再判断是“女主讨厌”还是“女主行为缺动机”。

反例二：后台首章读完率掉，agent 只润色句子。  
问题：高影响数据被低成本动作糊弄。应排查开篇承诺、主角目标、冲突入口、信息密度和章末牵引。

反例三：编辑说“节奏快点”，agent 删掉所有生活动作。  
问题：把节奏问题误解成删人味。应保留能暴露关系、压力和人物选择的动作，只压缩无状态变化的铺垫。

反例四：作者突然想到一个更酷设定，agent 立刻重排大纲。  
问题：新灵感没有经过影响范围和修改成本评估。应先入 Park 或 P3，除非它能低成本强化主承诺。

## 7. 交接顺序

推荐链路：

```text
41 评论聚类 -> 42 冲突仲裁 -> 43 优先级队列 -> 33 修订执行 -> 40 数据观察 -> 65 看板回写
```

执行后必须回写：

- 哪些反馈已修、暂缓、拒绝。
- 哪些正反馈资产被保护。
- 哪些细纲、台账、包装材料或后续章节任务单被更新。
- 观察窗口到哪一章、看什么数据或评论。

## 8. 来源与迁移说明

本篇把产品管理里的优先级方法迁移为网文修订队列：

- Intercom 的 RICE 方法强调用影响范围、影响强度、置信度和成本比较候选项。
- Atlassian 对 RICE 的说明提醒：评分能帮助解释取舍，但也可能主观、耗时或误导，不能迷信数字。
- Agile Business Consortium 的 MoSCoW 说明强调 Must/Should/Could/Won't this time 的时间盒取舍；迁移到连载修订中，用来保护本轮更新节奏和队列弹性。

这些方法不是小说创作公式，只用于让 agent 把“很多意见”变成可执行、可复盘、不会误伤作品资产的改稿队列。
