## 1. 核心原则：先诊断层级，再决定改法

AI 改稿前不能直接重写正文。必须先判断问题在哪一层：

```text
承诺层 -> 结构层 -> 场景层 -> 人物层 -> 信息层 -> 句段层 -> 格式层
```

越上层的问题，越不能靠句子润色解决。开篇承诺错了，句子再漂亮也留不住读者；主角目标错了，补形容词只会让文本更粘；连续性错了，大段重写可能牵出更多矛盾。

## 2. 七层问题与对应改法

| 层级 | 问题表现 | 优先改法 | 禁止动作 |
|---|---|---|---|
| 承诺层 | 书名简介、题材、第一章给的体验和正文不一致 | 重定目标读者、主承诺、前十章回报 | 只改文案或只润第一章 |
| 结构层 | 主线偏题、目标断、阶段回报缺失 | 调整大纲、章节顺序、卷目标和回报点 | 先做逐句润色 |
| 场景层 | 场景没目标、没阻力、没转折 | 重写场景目标、冲突、结果 | 只加描写和心理 |
| 人物层 | 人物不主动、声音同质、关系没变化 | 重写欲望、选择、台词、关系后状态 | 给人物贴标签 |
| 信息层 | 设定堆砌、伏笔断、线索不公平 | 重排信息投放和台账 | 用旁白补课 |
| 句段层 | AI 味、重复、抽象总结、节奏拖 | 压缩、换动作、换物件、换对白 | 改成更工整的套话 |
| 格式层 | 段落密、错字、标点、排版问题 | 格式清理和校对 | 用格式修正掩盖剧情问题 |

## 3. 三种改稿模式

### 3.1 小修

适用：错字、称谓、数字、个别句段、局部节奏、段落切分、单个物件或一处轻微设定误差。

要求：不改变章节主任务、不改变人物选择、不改变后续因果。

### 3.2 中修

适用：一场戏目标不清、对话无效、情绪不到位、信息顺序错、章末牵引弱、人物反应不对。

要求：保留本章功能和有效资产，重排场景、对白、动作和章末。

### 3.3 大修/重写

适用：主承诺错位、开篇方向错、主角目标不存在、结构偏题、核心关系不成立、世界规则破坏后续、读者理解完全错位。

要求：先写重写简报，再动正文；先保留有效资产，再确定删除内容；先同步台账，再更新后续大纲。

## 4. 改稿前必须做“有效资产盘点”

不要因为一章整体有问题，就把所有内容推倒。AI 重写前先保留能用的资产：

```text
【必须保留的承诺】读者已经被什么吸引？
【必须保留的信息】后文依赖哪些设定、线索、物件、时间和人物状态？
【必须保留的人物动作】哪些选择体现了人物核心？
【必须保留的情绪点】哪一处读者可能会爽、笑、心疼、紧张或心动？
【必须保留的语言风格】有哪些句子或对话有项目声音？
【必须删除的无效内容】哪些只是解释、重复、拖延、AI 味或违约内容？
```

重写不是清空重来，而是“保留有效资产，重建失效链路”。

## 5. 改稿流程：诊断、方案、执行、台账

```text
1. 读任务：用户要小修、中修、大修、审稿后改稿，还是重写？
2. 定层级：问题属于承诺、结构、场景、人物、信息、句段还是格式？
3. 保资产：列出必须保留、可移动、可删除、待确认内容。
4. 给方案：说明改什么、不改什么、为什么。
5. 执行改稿：按层级从上往下改，先结构后句子。
6. 更新台账：事实、人物状态、伏笔、设定和后续大纲变化必须同步。
7. 输出变更说明：说明解决了什么风险，留下什么待观察点。
```

## 6. 已发布/连载中内容的特殊规则

网文和未发布手稿不同。连载内容已经被读者读过，随意大改会破坏阅读连贯和读者信任。

规则：

1. 已发布章节优先小修和补丁式修复。
2. 大范围 bug 优先在后文圆回，必要时只在前文悄悄补一两句铺垫。
3. 不在正文里公开解释 bug，不把作者和读者争论写进故事。
4. 影响范围小的问题，悄悄修；影响范围大的问题，在正文外简短说明和致谢。
5. 大修会影响更新节奏时，先评估读者流失风险和后续收益。
6. 连载中改设定、改关系、改时间线，必须同步台账和后续大纲。

## 7. Bug 修复决策树

```text
发现 bug
-> 是否影响读者理解当前剧情？
   -> 否：记录台账，择机小修。
   -> 是：是否已发布且影响范围大？
      -> 否：直接小修或中修。
      -> 是：是否能用后文合理圆回？
         -> 能：后文圆回，前文最多补轻铺垫。
         -> 不能：评估是否必须公开说明、重写或承认设定调整。
```

分类处理：

- 笔误/数字/称谓：直接小修。
- 时间线矛盾：优先改最少位置，同时同步时间线台账。
- 人物行为矛盾：补动机、压力、误判或代价，不能只改一句心理。
- 设定规则矛盾：优先限制例外条件，不要扩大外挂能力。
- 读者误解：先判断是作者表达不清还是读者预期错配；不要急着争辩。

## 8. 反馈处理：先聚类，不逐条服从

读者、编辑、用户和数据反馈都要处理，但不能一条一改。

AI 先做反馈聚类：

| 类型 | 例子 | 处理方式 |
|---|---|---|
| 承诺反馈 | “不是这个味”“标题党”“不像爽文” | 回查包装、第一章、前三章 |
| 节奏反馈 | “水”“拖”“看不下去” | 查每章主任务、场景转折、回报密度 |
| 人物反馈 | “主角窝囊”“女主工具人”“反派降智” | 查人物目标、选择、阻力和关系后状态 |
| 信息反馈 | “看不懂”“设定好多”“反转突兀” | 查信息投放、伏笔台账、解释顺序 |
| 情绪反馈 | “不甜”“不爽”“不心疼” | 查同币种回报和情绪债 |
| 文本反馈 | “AI 味”“像大纲”“句子尬” | 查抽象总结、重复句式、动作承载 |
| 合规反馈 | “擦边”“危险”“侵权” | 暂停发布，走合规专题 |

处理优先级：高频反馈 > 影响留存的反馈 > 与主承诺相关的反馈 > 可低成本修复的反馈 > 单条个人偏好。

## 9. 小修不要改坏风格

AI 常见问题是把有生命力的句子改成平滑、正确、无聊的句子。

小修原则：

- 保留人物声音，不把台词改成标准说明文。
- 保留节奏毛刺，不把紧张、喜剧、痛感全抹平。
- 保留项目风格，不把所有作品都润成同一种“高级感”。
- 只删除重复、空泛、错漏和失效内容，不删除有效个性。
- 不把口语、方言、停顿、沉默全改成书面语。

检查问题：这处修改让读者更清楚、更有感、更想看，还是只是更工整？

## 10. 重写不是扩写

重写的目标是改变失效结构，不是把原文扩成更长。

重写前写清：

```text
【原问题】哪条链路失效？
【重写目标】读者读完后应该感到什么变化？
【保留资产】哪些设定、人物、物件、台词、信息、情绪必须保留？
【删除内容】哪些解释、回忆、寒暄、空转场景必须去掉？
【新结构】场景目标 -> 阻力 -> 转折 -> 回报 -> 章末牵引
【后续影响】哪些台账、大纲、伏笔、人物状态需要同步？
```

如果重写后只是字数更多、描写更密、台词更顺，但主承诺、场景转折和人物选择没有变化，这不叫重写，叫扩写失败。

## 11. 多轮修订顺序

推荐顺序：

```text
第一轮：承诺和结构
第二轮：场景和人物
第三轮：信息、伏笔、连续性
第四轮：章法、段落、节奏
第五轮：句子、AI 味、错别字和格式
```

不要反过来。先校对再改结构，会浪费工作；先润句子再重写场景，会制造沉没成本。

## 12. AI 改稿硬性禁令

1. 禁止未诊断问题层级就直接重写。
2. 禁止把用户要求“小修”擅自扩大成大修。
3. 禁止把已发布正文大范围改动而不说明风险。
4. 禁止为了回应单条评论破坏主承诺。
5. 禁止只做句子润色却声称解决结构问题。
6. 禁止删除有效人物声音、梗点、情绪毛刺和项目风格。
7. 禁止改完后不更新台账和后续大纲。
8. 禁止用“AI 更流畅”的统一腔覆盖角色差异。
9. 禁止为了圆 bug 临时扩大神级设定，破坏长期风险。
10. 禁止把所有反馈都当命令，也禁止把所有反馈都当噪音。

## 13. 修订/重写任务卡

```text
【任务类型】小修 / 中修 / 大修 / 重写 / 审稿后改稿 / bug 修复 / 连载补丁
【文本状态】未发布 / 已发布 / 连载中 / 完结后修订
【用户目标】修什么？不希望动什么？
【问题层级】承诺 / 结构 / 场景 / 人物 / 信息 / 句段 / 格式
【证据】具体文本现象、读者反馈、数据或编辑意见。
【有效资产】必须保留的承诺、信息、人物动作、情绪点、语言风格。
【风险】牵连后文 / 破坏台账 / 改坏人设 / 更新受影响 / 平台合规 / 读者已读
【修改方案】先改什么，后改什么；不改什么。
【执行结果】解决了什么问题。
【台账同步】事实 / 人物状态 / 伏笔 / 设定 / 时间线 / 后续大纲
【待观察】改后数据、评论、读者理解是否改善。
```

## 14. 审稿后改稿流程

1. 把审稿意见拆成“必须改”“建议改”“可不改”“待确认”。
2. 对每条意见标问题层级。
3. 把同层级、同原因的问题合并处理。
4. 先解决会影响承诺、主线、人物能动性和留存的问题。
5. 再处理场景、信息、章法和句段。
6. 最后统一校对格式和错漏。
7. 输出改稿说明，说明哪些意见采纳、哪些暂不采纳、原因是什么。

## 15. 快速自查问题

- 这次修改是在修问题，还是只是在换表达？
- 问题属于哪一层？有没有用低层修改冒充高层修复？
- 有效资产保住了吗？
- 修改是否影响后续台账、大纲、伏笔和人物状态？
- 已发布章节是否需要控制改动范围？
- 读者反馈是高频共性，还是单条偏好？
- 改后是否更兑现主承诺，而不是更“漂亮”？
- 是否把有风格的文本改成了无个性的 AI 平滑文？

## 16. 来源与迁移说明

- 起点社区《bug处理技巧》强调网文连载即时性强、出错不可避免，读者认为错就会影响阅读体验；处理 bug 要端正态度、控制扩散，并尽量避免已发布内容大修。本文将其迁移为“已发布/连载中内容的特殊规则”和“Bug 修复决策树”。
- 起点社区《嘿，编辑，你跟我说说，为啥我的书没人看啊》强调新书早期应稳定更新、提高质量、检查书名简介开篇、分析同类热销文，并在后续章节边写边改进。本文将其迁移为“反馈聚类”和“连载中改进优先级”。
- 起点社区《开文前，拟好一份大纲，至少可以少走点弯路吧》强调大纲能减少偏题、思路混乱和流水账，写偏后也能帮助调整。本文将其迁移为“结构层优先”和“改稿后同步大纲”。
- Reedsy《How to Revise a Novel》把修订拆为通读记录、时间线和节奏、人物、世界观、逐场节奏对白、句子习惯等阶段。本文将其迁移为“多轮修订顺序”。
- Reedsy《How to Edit Your Own Book》区分结构、人物、视角、句子和 copy edit 等层级，并提醒每个句子应贡献故事。本文将其迁移为“七层问题与对应改法”。
- Writer's Digest《7 Strategies for Revising Your Novel》强调重写比初稿更难，不要过早 copyedit，要结构化审视章节、场景、人物和重复习惯。本文将其迁移为“先诊断层级，再决定改法”。
- Writer's Digest《Facing the Edits》强调处理编辑反馈要先调整心态、理解反馈目的，再决定采纳方式。本文将其迁移为“反馈处理：先聚类，不逐条服从”。
