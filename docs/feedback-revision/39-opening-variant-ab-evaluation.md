前三章承诺兑现入口：

状态：已确认。

用途：当 AI agent 需要为一本新书、重启项目、第一章重写或包装调整设计多个开篇方案时，用本规范把“我觉得哪个好”改造成“基于读者承诺、变量控制、盲评反馈和数据迹象的版本决策”。

## 1. 核心原则：A/B 不是多写几个开头让 AI 挑喜欢

开篇版本评估要解决的是“哪种入口更能兑现同一个商业承诺”，不是“哪种故事方向更像一本新书”。

有效 A/B 必须同时具备五个要素：

- 假设：本次想验证什么，例如“先给危机比先给设定更能提升下一章欲望”。
- 不变量：题材、人设底盘、主承诺、禁忌边界、目标读者不能乱变。
- 单一关键变量：每轮只主测一个或少数强相关变量，例如开场压力、信息揭示顺序、视角距离。
- 证据：用目标读者盲评、编辑反馈、点击、停留、追读、评论关键词等信号判断。
- 决策：选用、合并、重写、继续测试或放弃，并写入项目记录。

常见错误是把 A/B 做成“新书大乱斗”：A 版是重生爽文，B 版是悬疑权谋，C 版是恋爱群像。这样的比较无法判断开篇优劣，只是在比较不同项目。

## 2. 什么时候必须做开篇多版本

建议做多版本的情况：

- 开新书前：竞品和读者预期已经明确，但第一章入口还没有验证。
- 题材承诺强但开头平：设定、金手指、人设都有卖点，第一屏却没有压力或钩子。
- 包装和正文可能错位：书名、简介、标签吸引来的读者，与第一章实际体验不一致。
- 团队判断分歧：编辑、作者、AI agent 对开篇取向争议明显。
- 数据信号矛盾：点击不低但追读差，或试读好评多但真实转化弱。
- 重写第一章前：不要只修句子，要先确认入口策略是否错了。

不建议做多版本的情况：

- 题材、主角、卖点、目标读者还没定。
- 只是因为作者或 agent 不自信，反复开新坑逃避推进。
- 已连载较久，读者已经接受当前开篇认知。
- 样本极小且非目标读者，不足以代表商业判断。
- 版本差异太多，无法知道胜负来自哪个因素。

## 3. 版本层级：先决定测什么，不要一上来重写整本书

开篇 A/B 可以分成七个层级。agent 应选择最小可验证层级。

| 层级 | 测试对象 | 适用问题 | 输出 |
| --- | --- | --- | --- |
| L1 包装钩子 | 书名、简介、标签、首句卖点 | 点击低、题材感弱 | 包装候选表 |
| L2 第一屏 | 前 300 到 800 字 | 起读慢、读者不知道看点 | 第一屏候选 |
| L3 第一幕 | 第一个场景或第一个冲突单元 | 开头没有事件、压力不足 | 场景候选 |
| L4 第一章 | 完整第一章 | 读完后不想追下一章 | 章节候选 |
| L5 前三章 | 黄金三章承诺兑现 | 有开头钩子但留存弱 | 三章节奏表 |
| L6 前十章 | 升级、关系、矛盾闭环 | 前期追读不稳 | 十章反馈表 |
| L7 包装加正文联测 | 书名简介和开篇组合 | 点击和留存冲突 | 组合方案表 |

优先原则：能测第一屏就不先测前三章；能测入口顺序就不推翻人设；能调整包装就不要重写正文。

## 4. 试写前必须锁定的不变量

每轮试写前，agent 要先写清楚不变量。没有不变量，版本比较没有意义。

不变量清单：

- 目标平台：例如番茄、起点、晋江、红袖、七猫、短篇平台或自有渠道。
- 目标读者：年龄层、性别倾向、题材经验、爽点偏好、雷点。
- 类型承诺：读者点进来期待获得什么核心体验。
- 主角底盘：身份、欲望、缺陷、初始困境、可追随理由。
- 商业卖点：金手指、关系张力、逆袭路径、悬念机制或情绪爽点。
- 合规边界：不能碰的题材风险、平台限制、年龄分级风险。
- 叙事约束：人称、视角、时间线、语气、篇幅区间。
- 禁止行为：不得搬运竞品桥段，不得为了测试制造虚假数据，不得把候选版当已确认设定。

## 5. 常用开篇变量库

每个版本只改一个主变量，最多附带一到两个从变量。否则反馈无法归因。

| 变量 | A 方向 | B 方向 | 重点观察 |
| --- | --- | --- | --- |
| 入口时间 | 事发前铺垫 | 事发中切入 | 哪个更快形成压力 |
| 主角状态 | 已经受辱受困 | 正在做选择 | 哪个更可追随 |
| 金手指揭示 | 第一屏明示 | 第一章末揭示 | 爽点是否更清楚 |
| 关系张力 | 先给关系矛盾 | 先给外部危机 | 情绪黏性是否增强 |
| 信息密度 | 少设定快推进 | 多设定强世界观 | 是否降低阅读阻力 |
| 类型信号 | 直接打标签感 | 通过事件暗示 | 读者是否能秒懂题材 |
| 叙事距离 | 近距离贴身感 | 冷静叙述感 | 主角代入是否增强 |
| 第一章结尾 | 悬念钩子 | 爽点兑现后新压力 | 下一章欲望是否更强 |
| 反派登场 | 开局压迫 | 延迟露面 | 压力来源是否清晰 |
| 情绪基调 | 爽感先行 | 痛感先行 | 是否符合目标平台 |

## 6. 推荐版本设计方式

agent 设计版本时，不要用“版本一、版本二、版本三”这种空标签，要写清版本假设。

| 版本 | 假设 | 主要改动 | 适用风险 |
| --- | --- | --- | --- |
| A 熟悉入口版 | 读者需要快速识别类型和爽点 | 用高辨识度场景开局 | 容易套路、缺少新鲜感 |
| B 高压入口版 | 先把主角逼到选择点更能留人 | 第一屏给危机和损失 | 可能信息不足、读者懵 |
| C 人物入口版 | 先让读者喜欢主角更能支撑长线 | 第一幕强化欲望、缺陷、关系 | 节奏可能慢 |
| D 机制入口版 | 金手指或系统规则足够新，值得前置 | 第一章早揭示机制 | 可能像说明书 |
| E 关系入口版 | CP、师徒、家族、宿敌关系是最大卖点 | 先给关系张力 | 外部剧情压力可能弱 |

推荐做法：第一轮最多三版，通常选择 A 熟悉入口版、B 高压入口版、C 人物入口版。若核心卖点是机制或关系，再替换其中一版。

## 7. 评分维度：开篇不是文采比赛

建议使用 100 分制评分，但不要机械追求小数。重点看哪一版的短板最致命。

| 维度 | 权重 | 评分问题 |
| --- | --- | --- |
| 类型承诺清晰度 | 15 | 读者是否能快速知道这是哪类爽感、情绪或关系体验 |
| 第一屏钩子 | 15 | 前 300 到 800 字是否有压力、悬念、冲突或强烈反差 |
| 主角可追随性 | 15 | 主角是否有欲望、困境、行动和情绪入口 |
| 卖点前置程度 | 10 | 核心卖点是否足够早出现，且没有变成硬讲设定 |
| 信息负担 | 10 | 名词、背景、设定是否压垮读者 |
| 情绪强度 | 10 | 爽、痛、甜、惊、怒、期待是否有明确峰值 |
| 下一章欲望 | 15 | 读者是否想知道下一步、下一个选择或下一个反转 |
| 长线引擎 | 5 | 开篇是否能自然接出十章、三十章和主线升级 |
| 风险成本 | 5 | 是否存在合规、撞梗、过度套路或重写成本问题 |

决策底线：如果某版总分高但“类型承诺清晰度”或“下一章欲望”明显低，不要直接选用。商业开篇最怕漂亮但不卖。

## 8. 盲评方法：让读者说体验，不让读者当编辑

小样本盲评适合判断体验问题，不适合宣称统计结论。

盲评设置：

- 读者数量：5 到 8 名目标读者即可发现大量明显问题；若题材分层强，可分两组。
- 呈现方式：只标 A、B、C，不透露作者偏好和版本假设。
- 阅读材料：同等长度，尽量同等排版，不让格式影响判断。
- 反馈时机：读完第一屏问一次，读完第一章再问一次。
- 记录方式：保留原话关键词，但最终归纳成问题类型。

不要只问“你觉得哪个好”。优先问这些问题：

- 你看到前 300 字时，以为这本书会给你什么体验？
- 哪一句或哪一段让你最想继续看？
- 你第一次想跳读或退出的位置在哪里？为什么？
- 你现在最想知道的下一个问题是什么？
- 你能用一句话说出主角想要什么吗？
- 你觉得这本书像什么类型？和简介承诺一致吗？
- 哪个版本最让你愿意点下一章？不是哪个写得最好。

判读原则：读者给的“修改建议”可参考，但更重要的是他们的退出点、误解点、期待点和情绪词。

## 9. 数据 A/B：没有足够流量时，不要伪装成科学

真实平台数据有价值，但要注意边界。

可以观察的数据：

- 曝光到点击：包装、标题、简介、标签和首图问题。
- 点击到首章读完：第一屏、第一幕、信息负担问题。
- 首章到次章：第一章结尾、主角行动、下一章问题。
- 三章追读：黄金三章承诺兑现问题。
- 收藏、加入书架、评论：读者是否愿意建立长期关系。
- 评论关键词：读者实际记住的是人物、梗、设定、情绪还是雷点。

数据解释规则：

- 点击高、追读低：多半是包装承诺和正文体验错位，或第一章留不住。
- 点击低、读完率高：多半是包装弱，正文可能可用。
- 首章读完高、三章掉：第一章钩子有效，但承诺兑现或节奏引擎不足。
- 评论多但转化弱：可能争议强、噱头强，但目标读者不稳定。
- 收藏高但追读弱：读者觉得“以后可能看”，当下推动力不足。

严禁行为：不要伪造读者反馈，不要把非目标读者意见当市场结论，不要用一两个样本宣布胜出，不要为了 A/B 测试制造误导性包装。

## 10. 版本胜出不等于全文确认

开篇版本胜出，只说明该入口在当前证据下更值得进入下一阶段。agent 还要继续检查它是否能支撑后续连载。

胜出后必须追问：

- 这个版本能自然接出前三章的升级、反转或关系推进吗？
- 这个版本的主角目标能撑住前三十章吗？
- 这个版本有没有牺牲主承诺，只是制造了短期刺激？
- 这个版本的读者期待，是否和书名、简介、标签一致？
- 这个版本的高光能否重复生产，而不是只靠第一章一次性爆点？

如果答案是否定，允许选用该版本的局部资产，例如第一屏钩子、主角压力、结尾悬念，但不要直接确认全稿。

## 11. 合并版本的规则

合并不是把 A、B、C 好句子拼在一起。合并必须围绕一个主版本进行。

合并顺序：

1. 先选主版本：以商业承诺最清楚、下一章欲望最强的一版为骨架。
2. 再提取资产：从其他版本提取一个钩子、一个动作、一个情绪峰值或一个句式。
3. 删除冲突：凡是改变主角动机、类型信号或时间线的资产，不能直接拼入。
4. 重写过渡：合并后的文本必须像一个自然版本，不像剪贴稿。
5. 更新记录：写明保留了哪个版本的哪个优势，放弃了什么。

## 12. agent 执行流程

建议把开篇试写拆成七步，适合单 agent 或多 agent 协作。

| 步骤 | 负责 agent | 输入 | 输出 |
| --- | --- | --- | --- |
| 1 锁定承诺 | 总控 agent | 项目设定、市场调研、读者预期 | 不变量清单 |
| 2 设计假设 | 策划 agent | 不变量、竞品差异、开篇问题 | 版本假设表 |
| 3 试写版本 | 写作 agent | 假设表、风格圣经、上下文包 | A/B/C 候选稿 |
| 4 自检过滤 | 审稿 agent | 候选稿、评分表 | 初筛评分 |
| 5 盲评收集 | 读者反馈 agent | 候选稿、问题清单 | 反馈摘要 |
| 6 决策综合 | 总控 agent | 评分、反馈、数据 | 版本决策 |
| 7 沉淀同步 | 连续性 agent | 胜出版、取舍说明 | 项目圣经和交接包更新 |

每一步都要标注状态：“候选”“待确认”“已确认”。候选版本不能污染正式设定。

## 14. 审稿检查清单

agent 审稿时，可以用以下问题快速判断版本评估是否合格。

- 是否先写明了同一个主承诺，而不是比较不同书？
- 每个版本是否只有一个主变量？
- 版本差异是否能被读者感知，而不只是作者内部设定不同？
- 第一屏是否已经给出压力、悬念、反差、关系或情绪钩子？
- 读者能否在第一章内说清主角想要什么？
- 读者能否在第一章内看见这本书的类型承诺？
- 结尾是否制造了明确的下一章问题？
- 盲评是否使用目标读者，而不是随便找朋友？
- 数据解释是否区分点击问题、正文问题和承诺错位？
- 胜出版本是否已经同步到项目圣经、章节流水线和交接包？

## 16. 参考来源

- Google Search Central 关于网站测试的说明：A/B 或多变量测试应避免误导用户，实验结束后要保留最终版本；迁移到小说知识库时，对应“不要用误导性包装做测试，测试结束要确认唯一正式版本”。
- Nielsen Norman Group 关于小样本可用性测试和 A/B 测试定位的资料：小样本适合发现体验问题，但不能当作统计显著结论；迁移到小说知识库时，对应“盲评看退出点和误解点，不把 5 个读者当全市场”。
- Optimizely 关于 A/B testing 的基础资料：A/B 的核心是比较两个或多个版本对同一目标的影响；迁移到小说知识库时，对应“先定义变量、指标和决策”。
- Reedsy 与 Writer's Digest 关于开篇、第一章和 hook 的写作资料：开篇要尽快制造问题、张力、人物欲望和继续阅读理由；迁移到网文场景时，对应“第一屏钩子、第一章结尾和类型承诺”。
