# feedback-revision

状态：已确认

审稿、试读、反馈聚类、修订和掉点诊断。

本目录负责回答一个问题：作品收到文本问题、读者反馈、编辑意见和数据异常后，agent 应该怎样诊断、排序、改稿、观察，而不是凭感觉一条一条乱改。

## 1. 先选任务路线

| 任务 | 先读 | 作用 |
|---|---|---|
| 发布前审稿、试读、判断 AI 含量 | `08-review-and-reader-feedback.md` | 建立审稿维度、读者体感、问题层级和修改优先级 |
| 已有稿件要修订或重写 | `33-revision-rewrite-version-workflow.md` | 把问题转成改稿动作，处理小修、中修、重写和版本同步 |
| 有后台数据或掉点 | `18-reader-data-feedback-diagnosis.md`、`40-serial-data-tuning-loop.md` | 判断点击、读完、追读、评论、更新节奏和承诺兑现问题 |
| 有大量评论、章评、试读反馈 | `41-reader-comment-coding-feedback-clustering.md` | 把原始反馈拆成观察点、标签、主题和可验证发现 |
| 多方反馈互相打架 | `42-editor-reader-data-feedback-conflict-resolution.md` | 在编辑、读者、数据、作者直觉和主承诺之间做仲裁 |
| 问题很多，需要决定先改什么 | `43-feedback-revision-priority-queue.md` | 把反馈转成 P0/P1/P2/P3/Park/Reject 队列 |
| 开篇多版本选择 | `39-opening-variant-ab-evaluation.md` | 比较不同开篇入口、包装承诺和读者反应 |
| 目标读者毒点、弃书红线 | `101-reader-dropoff-poison-points-red-flags-retention-safety.md` | 识别会直接伤留存或劝退目标读者的高危问题 |

## 2. 推荐链路

### 2.1 发布前文本质检

```text
08 审稿标准 -> 33 修订执行 -> 43 优先级队列 -> 必要时回到正文重写
```

适合：新章发出前、投稿前、第一章生产许可前、用户说“这段有 AI 味/不自然/逻辑怪”时。

### 2.2 发布后反馈复盘

```text
18 数据诊断 -> 40 连载调参 -> 41 评论聚类 -> 42 冲突仲裁 -> 43 排队 -> 33 修订执行
```

适合：已经有点击、读完、追读、收藏、评论、试读意见或编辑建议时。

### 2.3 多版本开篇取舍

```text
39 开篇变体评估 -> 18 数据诊断 -> 41 反馈聚类 -> 43 修订队列
```

适合：第一章、前三章、简介/标题/开篇入口正在做 A/B 或盲评时。

## 3. 常见误用

- 不要只读 `08` 就开始逐句润色；审稿结论必须进入 `33` 或 `43` 才能变成可执行修订。
- 不要把 `41` 的评论聚类当成读者投票；评论要和文本位置、目标读者、后台数据交叉验证。
- 不要用 `42` 的仲裁结果替代改稿队列；仲裁解决“听谁的”，`43` 解决“本轮先改什么”。
- 不要把 `18` 和 `40` 当作数据玄学；没有真实数据时，只能标为假设或观察任务。
- 不要为了修毒点把有效资产也删掉；改稿前要写明哪些人物锋利感、作者味、CP 张力或类型承诺必须保护。

## 4. 最小读取建议

轻量审稿：`08` + `33`。  
评论很多：`41` + `43`。  
反馈冲突：`42` + `43`。  
数据掉点：`18` + `40` + `43`。  
完整复盘：`18` + `40` + `41` + `42` + `43` + `33`。

如果任务已经涉及正式正文改动，还要回到 `docs/workflows/58-integrated-drafting-beta-review-revision-workflow.md`，让写手、试读、审稿和修订闭环一起工作。
