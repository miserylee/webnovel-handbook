## 1. 核心原则：移动端阅读要让进展可见

网文读者大量在手机上读。手机屏幕小、翻页频繁、环境碎片化，读者需要更清楚地感到：每一屏、每一段、每一章都在推进某个承诺。

AI 写作时使用这条链路：

```text
读者主承诺 -> 本章叙事任务 -> 场景目标/阻力/转折 -> 段落功能 -> 句子节奏 -> 章末牵引
```

章法不是把句子写短，也不是满屏空行。章法的核心是：读者在手机上不用费力，就能看见目标、变化、情绪和下一步。

## 2. 七个单位不要混用

| 单位 | 作用 | AI 常见误用 |
|---|---|---|
| 章节 | 完成一次阶段任务，制造下一章期待 | 只是字数容器 |
| 场景 | 角色带目标遇阻并产生转折 | 只是人物聊天或地点描写 |
| 段落 | 管理一个动作、反应、信息或情绪单位 | 一段塞三四个功能，或全篇一行一段 |
| 句子 | 控制速度、语气和读者注意力 | 长句堆解释，短句乱制造紧张 |
| 转场 | 跳过无效过程，连接必要状态 | 把赶路、洗漱、开会流水账全写出 |
| 留白 | 让读者补足未明说的情绪或信息 | 该交代的不说，造成迷路 |
| 断章 | 在本章有进展后留下下一步压力 | 没有回报，只把答案扣住 |

## 3. 一章必须有一个主任务

每章可以承担多个功能，但必须有一个主任务。

常见主任务：

- 推进目标：主角离目标更近或更远。
- 完成回报：打脸、获得资源、关系确认、线索答案、经营改善。
- 扩大压力：敌人升级、规则变化、时间逼近、代价暴露。
- 改变关系：信任、误会、亏欠、偏爱、背叛、站队发生变化。
- 更新信息：读者、主角或对手获得新认知。
- 完成过渡：从一个阶段切到下一个阶段，但过渡也要有状态变化。

AI 起草前先写：

```text
本章主任务：
本章必须让读者看见的变化：
本章不能拖到下一章才给的回报：
本章末尾要牵引的下一步压力：
```

## 4. 场景最小结构：目标、阻力、转折

一个有效场景至少包括：

```text
视角人物想要什么 -> 谁/什么阻止他 -> 他怎样尝试 -> 结果如何改变局面
```

如果没有转折，它可能只是说明、氛围、过场或台词堆。

转折不一定要爆炸。安静戏也要有状态变化：认知变了、关系变了、情绪防线破了、证据位置变了、下一步选择变了。

## 5. 段落是手机阅读的呼吸单位

段落不是学校作文里的固定长度，而是读者眼睛和注意力的换气点。

移动端段落规则：

1. 一段优先只承载一个功能：动作、反应、观察、信息、内心或转折。
2. 视角焦点变化时，换段。
3. 动作结果出现时，换段。
4. 情绪方向变化时，换段。
5. 对话换人说话时，换段。
6. 关键信息或反转可以独立成短段。
7. 避免长段密密麻麻，也避免全篇碎成一行一段。

经验规则：手机端常规叙事段落宜短而清晰；说明段、设定段、心理段尤其要拆小，并且每段都要带动作、情绪或问题。

## 6. 段落长度服务节奏

| 场景状态 | 段落策略 | 风险 |
|---|---|---|
| 打斗、追逐、争吵、危机 | 短段、短句、动作结果清楚 | 过短会像提纲或短视频字幕 |
| 情绪确认、关系拉扯 | 中短段，动作和沉默要留空间 | 长心理解释会削弱心动或痛感 |
| 设定说明、调查推理 | 信息拆段，每段绑定问题或证据 | 一大段解释会像百科 |
| 日常轻松、群像喜剧 | 段落跟包袱节拍走，反应单独成段 | 全员吐槽碎段会吵 |
| 结算、余波、卷末 | 段落可略长，但每段要有状态变化 | 结算流水账会掉追读 |

## 7. 每一屏最好有微进展

手机读者不是每一屏都需要爆点，但每一屏最好能看到至少一种微进展：

- 一个动作推进。
- 一个新信息出现。
- 一个关系态度变化。
- 一个风险逼近。
- 一个情绪被触碰。
- 一个笑点或反应落地。
- 一个问题被缩小或扩大。
- 一个物件、证据、资源进入可用状态。

如果连续几屏都只有解释、赶路、寒暄、景物、复述或内心绕圈，读者会感觉“水”。

## 8. 信息密度：一段只解决一个读者问题

AI 容易在一个段落里同时解释世界观、人物背景、当前目标、旧伏笔、情绪总结和作者判断。手机端读者会被压住。

处理方式：

```text
先问读者当前最需要知道什么 -> 只回答这个问题 -> 用动作/对话/物件承载 -> 余下信息排队
```

信息投放顺序：

1. 当前看懂场景必需的信息。
2. 影响角色选择的信息。
3. 本章会回收的信息。
4. 后文才用的信息。
5. 作者自己觉得有趣但读者暂时不需要的信息。

第五类不进正文。

## 9. 转场：跳过无效过程，保留状态变化

转场不是把中间过程全写出来，而是让读者知道状态如何变化。

可跳过：

- 无冲突的赶路。
- 重复执行的训练。
- 不改变关系的寒暄。
- 不产生新信息的会议。
- 纯解释性的等待。
- 已经看过一次的流程。

需要保留：

- 转场中发生了选择、误会、代价、线索或关系变化。
- 转场本身展示背景规则或生活成本。
- 转场结束后的状态如果不交代会让读者迷路。

转场句要回答：时间过了多久、地点变到哪里、人物状态有什么变化、下一场压力是什么。

## 10. 章末不是只“卡答案”

有效章末要满足两个条件：本章有回报，下一章有压力。

常见章末类型：

| 类型 | 用法 | 风险 |
|---|---|---|
| 动作未完成 | 门被推开、刀落下、名单翻到某页 | 如果本章没回报，会像骗追读 |
| 信息揭露 | 证据、身份、规则、真相露出一角 | 不能每章都只露一角不回答 |
| 新选择 | 主角必须立刻做决定 | 决定要和本章变化有关 |
| 对手反制 | 主角刚赢，对手下一步出现 | 对手不能无脑复活式加压 |
| 情绪转折 | 关系态度、误会、告白、背叛改变 | 情绪要有前文积累 |
| 目标升级 | 小目标完成，大目标打开 | 升级要让读者看见收益和代价 |

断章公式：

```text
本章落袋 -> 后果反弹 -> 下一步必须行动
```

## 11. 章节字数和平台规则

章节字数要同时考虑平台规则、发布策略、读者习惯和剧情断点。

通用优先级：

```text
不明显违反平台规则 -> 尊重剧情断点 -> 保持更新节奏 -> 再考虑字数收益
```

平台规则、福利条件、上架要求和无线阅读偏好会变化。正式发布前必须重新确认目标平台最新规则。知识库只保留稳定原则：不要为卡字数牺牲剧情断点，也不要用超长章节或过短章节破坏读者节奏感。

## 12. 章节名和目录体验

章节名也是移动端阅读路径的一部分。

可用原则：

- 章节名可以适当提示本章爽点、任务、冲突或情绪。
- 不要提前剧透核心悬念、伏笔和反转答案。
- 不要用会误导读者预期的标题党。
- 不要让目录里连续出现无信息量标题，如“新的开始”“风波”“变故”。
- 如果平台或题材更适合数字章名，也要确保章节内部开头迅速给目标。

## 13. 对话段落：让谁在推动局面一眼可见

对话不是把台词切成短行就有节奏。

AI 检查：

- 每个说话人是否有不同目标。
- 台词是否带来新信息、施压、试探、转移、隐瞒、拒绝或让步。
- 关键反应是否有动作承接，而不是只写“他沉默了”。
- 对话结束时关系、信息或下一步行动是否变化。
- 两人以上场景是否避免“甲乙丙丁轮流发言”。

## 14. 内心独白和解释段

内心独白要服务选择，不要替读者总结。

可保留的内心：

- 角色在做取舍。
- 角色压住真实情绪。
- 角色误读局面。
- 角色发现自己不愿承认的欲望。
- 角色把旧经验用于当前判断。

应删除或压缩的内心：

- 作者替读者解释“他很痛苦”。
- 重复前文已经看懂的信息。
- 用抽象价值观总结人物。
- 在危机中大段回忆无关背景。

## 15. AI 写作硬性禁令

1. 禁止一章只有字数，没有主任务。
2. 禁止场景没有目标、阻力和转折。
3. 禁止连续多屏只有说明、回忆、赶路或寒暄。
4. 禁止正文出现超长密集段落，尤其是设定、心理和背景说明。
5. 禁止全篇一行一段，用空行假装节奏。
6. 禁止每章只卡答案，不给本章回报。
7. 禁止用章节名剧透核心悬念或制造错误预期。
8. 禁止为卡平台字数牺牲剧情断点和读者体验。
9. 禁止桥段只负责“把人从 A 挪到 B”，不产生任何状态变化。
10. 禁止用旁白总结替代动作、台词、物件和后果。

## 16. 章法与移动端阅读设计卡

```text
【章节编号/标题】
【本章主承诺】爽点 / 情绪 / 悬疑 / 关系 / 喜剧 / 经营 / 升级 / 其他
【本章主任务】
【本章必须给的回报】
【本章末尾牵引】
【场景列表】每场写目标、阻力、转折。
【每场微进展】动作 / 信息 / 关系 / 风险 / 情绪 / 笑点 / 资源 / 规则
【段落风险】超长段 / 过碎段 / 设定堆 / 内心绕圈 / 对话轮流发言
【转场处理】跳过 / 概括 / 写成场景；理由：
【章节名字】是否提示卖点但不剧透核心答案？
【移动端自检】连续几屏是否都有变化？是否有密集阅读疲劳？
【平台规则待确认】章节字数、更新判断、上架/福利要求是否需要查最新规则？
```

## 17. 审稿流程

1. 先写出本章主任务和本章回报。
2. 按场景标注目标、阻力、转折。
3. 找出连续无进展的段落和屏幕区域。
4. 标出超长段、全碎段、解释段、内心绕圈段。
5. 判断每段功能：动作、反应、信息、情绪、转场、结算、钩子。
6. 删除或压缩无功能段落。
7. 把必要信息改成动作、对话、物件、证据或后果。
8. 检查章末是否“本章落袋 -> 后果反弹 -> 下一步必须行动”。
9. 检查章节名是否提示期待但不误导或剧透。

## 18. 快速自查问题

- 本章主任务一句话能否说清？
- 本章有没有读者能拿到的回报？
- 每个场景是否都有目标、阻力和转折？
- 连续几屏是否都在解释、寒暄、赶路或内心总结？
- 是否有一段塞入太多信息？
- 段落长短是否服务当前情绪和动作速度？
- 章末是否只扣答案，没给本章落袋？
- 章节名是否帮助期待，而不是破坏悬念？
- 为了平台字数，有没有牺牲剧情断点？

## 19. 来源与迁移说明

- 起点社区《爽文的节奏是什么？》强调网文需要“定时锁”，到一定时间发生特定事情，开篇完成临场感后要尽快满足读者期待。本文将其迁移为“本章主任务”“每一屏微进展”和“读者主承诺推进”。
- 起点社区《如何更好地包装作品》指出排版会直接影响阅读体验，过长段落会形成密密麻麻的阅读压力；章节字数要兼顾平台规则、无线阅读、剧情断点和更新策略。本文将其迁移为“段落是手机阅读的呼吸单位”和“章节字数优先级”。
- Writer's Digest《The Art of the Paragraph》强调段落管理内容、调节节奏、生成情绪和声音。本文将其迁移为“段落功能”和“段落长度服务节奏”。
- NotMFA《What Is a Scene?》强调有效场景由目标、阻力和升级构成，场景不是地点或字数容器，而是改变故事状态的单位。本文将其迁移为“场景最小结构”。
- Writing Excuses 10.31 强调节奏要和读者承诺匹配，读者需要感到人物朝目标困难前进；场景应服务多个功能。本文将其迁移为“每一屏微进展”和“章法链路”。
- Writer's Digest《5 Tips for Writing a Fast-Paced Fantasy Novel》强调开篇和场景应承担多重功能，世界观随行动逐步揭示，必要时跳过无情绪价值的过程。本文将其迁移为“转场”和“信息密度”。

## 长篇训练、修炼与成长段落防水入口

当章节或小单元涉及训练、修炼、闭关、学习、备赛、技能升级、职业成长、系统任务或团队磨合时，需要调用 `docs/58-training-growth-anti-padding.md`。该专题负责判断训练段是否应该展开、压缩或跳过，并要求训练段具备目标、瓶颈、方法、代价、反馈、验证和下一阶段限制，防止训练流水账、闭关后直接无敌、数值升级无证明和对手原地等待。

## 高潮战、大场面与群体冲突调度入口

当章节或卷级单元涉及高潮战、决战、群战、团队战、公开审判、朝堂对峙、比赛决胜、灾难救援、发布会、直播爆点、群体谈判，或任何三条以上行动线同时推进的大场面时，需要调用 `docs/60-climax-battle-large-scene-coordination.md`。该专题负责确认目标、胜负条件、空间锚点、视角锚点、群体角色分工、关键转折、动作压缩、战后结算和台账回写，防止只堆人数、招式、爆炸、喊话和震惊。

## 信息战、舆论战与公开场合话语权攻防入口

当章节涉及证据链反杀、公开审判、庭审、朝堂对峙、寿宴摊牌、直播翻车、热搜风波、危机公关、发布会、项目验收、商战舆论、娱乐圈黑料、职场甩锅、宗门审判、弹幕评论区风向，或任何“胜负取决于谁被相信”的场景时，需要调用 `docs/61-information-public-opinion-discourse-battle.md`。该专题负责确认争点、关键受众、证据来源与边界、对手叙事、抢解释权方式、主角反制节奏、公开行动请求、落袋余波和合规风险，防止万能录音、主角独白、众人集体倒戈、热搜空转和现实舆论风险。

## 日常章、过渡章与低强度章节防水入口

当章节涉及日常、过渡、休整、赶路、聊天、吃饭、生活流、低冲突推进、战后余波、关系缓冲、情绪沉淀、信息整理、准备章、轻松章或主线暂时低烈度推进时，需要调用 `docs/62-everyday-transition-low-intensity-anti-padding.md`。该专题负责判断低强度章节是否值得独立成章，是否承接旧结果、产生微变化、交付小回报、保留人味、生成下一步并回写台账，防止日常流水账、过渡无增量、休整清零、赶路只写到了和聊天无目的。


## 与 63 号专题的衔接：移动端提醒要短、具体、可视化

移动端阅读中，伏笔提醒不宜依赖长段复盘。处理跨章伏笔时，应调用 `docs/63-foreshadowing-callback-reader-memory-management.md`，优先使用短句、物件、动作、称呼、环境或后果来唤醒读者记忆。

可执行标准：C1 轻回响控制在一个可视化细节或一句对话内；C3 再激活必须挂在当章冲突或选择上；C5 明确回顾只在跨卷、长间隔、复杂设定回收前使用，且应拆成移动端易读的小段。

## 补充：章节名细则以标题承诺层为准

本文件保留“章节名和目录体验”的移动端基础规则。凡涉及章名生成、目录页连续阅读体验、章名是否剧透、章名是否标题党、推荐标题与正文章名差异时，优先使用 `docs/67-chapter-title-toc-promise-management.md`。

执行口径：章节名不是正文外的装饰，而是读者进入本章前看到的第一层承诺。移动端章节名应短、清楚、有推进感；目录页连续标题应能让读者看到目标、冲突、阶段和情绪正在变化。

## 补充：段落焦点必须服从视角契约

移动端段落不仅要短，还要让读者知道“这一屏是谁在看”。当视角焦点、观察对象、心理反应或信息权限发生变化时，应优先换段，并参考 `docs/69-pov-narrative-distance-information-focus.md` 检查旁白是否越权。

审稿时重点看：一个手机屏里是否连续进入多人内心；环境描写是否有视角人物的观看路径；从动作切到心理是否有停顿；从角色感知切到作者说明是否造成 AI 式远景感。
