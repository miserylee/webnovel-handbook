# 104. 完本复盘、项目后评估与下一本经验迁移专项

状态：已确认

适用范围：完本后复盘、卷末复盘、项目暂停或腰斩复盘、阶段数据复盘、读者反馈复盘、编辑意见复盘、多 agent 协作复盘、下一本开书准备、同一作者系列化经验沉淀，以及任务中出现“这本书学到了什么”“下一本怎么避免同样问题”“哪些经验能迁移”“完本报告怎么写”“项目失败怎么复盘”等问题时使用。

核心结论：完本不是项目结束，而是下一本书的训练数据生成时刻。AI agent 不能只把完本复盘写成剧情总结、数据罗列或自我表扬；必须把目标假设、实际结果、读者行为、文本资产、失误原因、不可迁移偶然因素和下一本行动闸门拆开，形成可复用经验、禁用教训和待验证假设。

最小链路：原始目标 -> 实际结果 -> 关键证据 -> 偏差诊断 -> 有效资产 -> 失败模式 -> 可迁移规则 -> 不可迁移条件 -> 下一本闸门 -> 知识库回写。

---

## 1. 为什么需要独立专题

已有第 18 篇管理数据反馈，第 24 篇管理结局和完本风险，第 41、42 篇管理评论与反馈冲突，第 55 篇管理卷末结算，第 103 篇管理口碑资产。但 AI agent 仍然容易在完本后犯这些错误：

- 把复盘写成“剧情复述 + 辛苦总结”，没有可执行结论。
- 只看最终成绩，不看开篇、前十章、中段、付费、卷末和结局的阶段性偏差。
- 把偶然成功当成稳定规则，下一本盲目复制。
- 把失败全归咎于平台、题材或读者，不拆文本执行问题。
- 只记录负面问题，忽略真正有效的角色、关系、梗点和口碑资产。
- 把一部书的专属经验硬迁移到完全不同题材、平台或目标读者。
- 多 agent 协作中，不记录哪些提示、流程、台账和审稿机制有效。

本篇负责把完本和阶段复盘变成“下一本能用”的知识资产。

## 3. 来源吸收与迁移边界

### 3.1 After Action Review：从目标、事实、差异和学习开始

AAR 类方法通常要求团队回答：原本打算发生什么、实际发生什么、为什么有差异、下次如何改进。

迁移规则：网文复盘也不能只看感觉。必须先写清开书时的目标读者、主承诺、数据目标、更新计划和创作假设，再对照实际结果。

### 3.2 Atlassian retrospectives：复盘要形成行动项

Atlassian 的复盘资料强调团队回顾要讨论什么有效、什么无效，并形成可执行改进行动。

迁移规则：AI agent 的复盘输出必须有“下一本动作”，例如修改开书闸门、增加前三章测试、调整台账字段、取消某类低效审稿、提前做专业资料包，而不是只写“以后注意”。

### 3.3 Google SRE blameless postmortem：关注系统，不追责个人

Google SRE 的事后复盘文化强调无责复盘，关注系统改进和事实时间线。

迁移规则：小说项目复盘不要写成“作者不努力”“AI 太差”“读者不懂”。要拆系统问题：任务单不完整、目标读者不清、章节流水线缺审稿、台账没有回写、数据节点没有复盘、包装正文错配等。

### 3.4 Nielsen Norman Group：定量和定性证据结合

NN/g 关于分析用户体验数据的资料强调，行为数据和定性反馈应互相校验。

迁移规则：完本复盘不能只看评论，也不能只看数据。完读、追读、付费、催更、收藏、评论、评分、读者复述和编辑意见要放在同一张证据表里。

### 3.5 Jane Friedman 等作者经验：下一本书不是简单复刻上一本文案

作者出版和书籍营销经验常提醒：第二本书、下一本书会受前作读者、作者品牌、市场定位、发行节奏和读者期待影响，但不能机械复制上一部。

迁移规则：下一本书可以继承作者资产、目标读者认知和有效流程，但不能把上一部的角色关系、梗点、题材红利或平台时机当成稳定公式。

## 4. 概念定义

### 4.1 项目后评估

定义：在完本、停更、腰斩、卷末或阶段节点后，对项目目标、文本执行、数据反馈、读者口碑、流程质量和经验迁移进行系统复盘。

### 4.2 可迁移经验

定义：跨项目仍然成立的规则、流程、判断或资产类型。

例子：

- 前三章必须证明主承诺。
- 每 5 到 10 章维护口碑资产台账。
- 失败线必须有反弹抓手。
- 专业题材开书前要先建资料包。
- 多 agent 接力必须区分 canon、候选和待确认。

### 4.3 不可迁移条件

定义：只在特定平台、时机、读者群、角色组合、题材热度或更新状态下成立的经验。

例子：

- 某个梗当时火，不代表下一本还能用。
- 某个 CP 被喜欢，不代表同样关系结构可复制。
- 某次推荐位带来的数据，不等于题材本身稳定有效。
- 某个争议点制造评论，不等于口碑健康。

### 4.4 失败模式

定义：项目中反复出现、可被下一本提前预防的问题模式。

例子：开篇承诺强但第 6-10 章无落袋、中段支线膨胀、反派反制不足、付费章拖延答案、关系线误会硬拖、卷末只开新坑不结算。

### 4.5 下一本闸门

定义：下一本书进入立项、开篇、前十章或批量生产前必须满足的硬检查。

例子：没有目标读者和主承诺，不写第一章；没有前三章证明链，不进入第 4-5 章；没有资料包，不写高专业场景。

## 5. 复盘的五类证据

| 证据类型 | 例子 | 用途 |
| --- | --- | --- |
| 初始假设 | 目标读者、类型承诺、题材卖点、更新计划 | 判断项目是否偏离原目标 |
| 文本证据 | 前三章、前十章、付费节点、卷末、结局 | 判断执行是否兑现承诺 |
| 数据证据 | 阅读、收藏、追读、付费、完读、催更、评分 | 判断读者行为变化 |
| 反馈证据 | 评论、书评、编辑意见、试读反馈、社群讨论 | 判断读者为什么留下或离开 |
| 流程证据 | 任务单、台账、审稿记录、修订记录、agent 交接 | 判断协作系统哪里失效 |

原则：没有证据的复盘结论只能标为“候选”，不能写成已确认规则。

## 6. 六层复盘框架

### 6.1 定位层

检查：

- 目标读者是否清楚？
- 主承诺是否长期稳定？
- 包装吸来的读者和正文服务的读者是否一致？
- 题材热度、平台环境和竞品位置是否被正确判断？

输出：下一本目标读者和定位闸门。

### 6.2 开篇层

检查：

- 第一章是否让读者知道主角、问题和承诺？
- 前三章是否证明卖点？
- 前十章是否完成第一轮落袋？
- 哪一章开始明显掉读？

输出：下一本前三章生产规则和前十章验证清单。

### 6.3 长线层

检查：

- 中段是否有稳定冲突引擎？
- 支线是否服务主承诺？
- 反派和世界是否同步升级？
- 卷末是否结算而非只开新坑？

输出：下一本卷级结构和支线管理规则。

### 6.4 人物层

检查：

- 主角可追随性是否稳定？
- 哪些配角被读者记住？
- 关系线是资产还是负担？
- 人物弧线是否有行为证据？

输出：下一本人物卡、关系线和角色声音改进。

### 6.5 读者层

检查：

- 读者留下的理由是什么？
- 弃书触发器是什么？
- 正反馈资产是什么？
- 负口碑是否来自类型违约、毒点、付费被骗感或烂尾？

输出：下一本留存安全和口碑资产策略。

### 6.6 流程层

检查：

- 哪些 agent 流程有效？
- 哪些提示词导致模板化、复述剧情或设定污染？
- 台账是否真的维护了连续性？
- 审稿和修订是否过早润色、过晚发现结构问题？

输出：下一本协作流程、上下文包和审稿闸门。

## 7. 复盘时间点

| 时间点 | 目的 | 输出 |
| --- | --- | --- |
| 第 3 章后 | 检查承诺是否被证明 | 前三章验证记录 |
| 第 10 章后 | 检查第一轮落袋和追读方向 | 开篇小复盘 |
| 第 30 章后 | 检查长线引擎和中段风险 | 入轨复盘 |
| 每卷末 | 检查卷目标、回报、支线和读者情绪 | 卷末复盘 |
| 上架/付费后 | 检查付费价值和被骗感 | 付费节点复盘 |
| 完本后 | 检查全书兑现、口碑和下一本迁移 | 完本复盘报告 |
| 腰斩/暂停时 | 找出停止原因和可救资产 | 暂停/淘汰复盘 |

注意：完本后才复盘太晚。商业连载应做轻量阶段复盘，把问题尽早转成后续修正。

## 9. 可迁移经验分级

| 等级 | 定义 | 写入位置 |
| --- | --- | --- |
| T0 个案观察 | 单部作品出现，证据不足 | 项目复盘备注 |
| T1 候选经验 | 有明确证据，但未跨项目验证 | `notes/` 或候选规则 |
| T2 项目内已确认 | 在本项目多个阶段有效 | 项目文档和台账 |
| T3 跨项目可迁移 | 多项目、多题材或来源支持 | 知识库专题文档 |
| T4 高风险限制规则 | 一旦违反会造成严重损失 | 决策记录和闸门 |

原则：不要把 T0/T1 直接写成知识库硬规则。知识库吸收经验时，要标注已确认、候选或待确认。

## 10. 成功复盘也要找系统原因

成功项目同样要复盘，否则下一本会误学。

常见误判：

- 把平台推荐误当题材必火。
- 把单个角色爆红误当关系结构万能。
- 把读者容忍某个毒点误当毒点安全。
- 把更新稳定带来的数据误当文本质量完全过关。
- 把第一本的新鲜感误当作者品牌已经稳固。

成功复盘要问：

- 哪些成功来自可控执行？
- 哪些成功来自外部时机？
- 哪些成功来自目标读者空位？
- 哪些成功不可复制？
- 如果换题材、平台或更新节奏，这条经验还成立吗？

## 11. 失败复盘不追责，追系统

禁止复盘写法：

```text
这本失败是因为作者没坚持。
读者不懂慢热。
平台不给量。
AI 写得不行。
```

推荐写法：

```text
开书时目标读者设为“喜欢慢热群像的悬疑读者”，但前三章没有给出明确主问题，读者评论集中为“不知道主线”。第 6-10 章支线人物增加，主角目标未推进，导致追读下降。下一本若采用群像慢热，必须在第一章建立主问题，并在前十章给出第一轮答案或阶段反转。
```

原则：复盘要把失败转成可预防机制。

## 12. 下一本经验迁移流程

### 12.1 迁移前先筛选

每条经验必须问：

- 它来自哪个证据？
- 它依赖哪个题材、平台、读者或更新时间？
- 它是文本规律，还是运营偶然？
- 它能否用本知识库已有专题解释？
- 它是否会和下一本主承诺冲突？

### 12.2 迁移时只迁移功能

例如：

```text
上一本文案有效：主角在公开场合用专业能力反打质疑者。
可迁移功能：早期公开证明主角独特能力，并让见证者行动变化。
不可迁移内容：公开场合、专业领域、反打方式、见证者台词、具体桥段。
```

### 12.3 迁移后变成闸门

```text
下一本开篇闸门：前三章至少有一次公开或半公开能力证明，且证明后必须改变主角资源、关系或风险状态。
```

## 13. 多 agent 复盘分工

### 13.1 数据 agent

负责：整理阶段数据、时间线和异常点，不直接解释原因。

输出：数据证据表。

### 13.2 文本审稿 agent

负责：对照章节内容，判断承诺兑现、状态增量、毒点、节奏和人物问题。

输出：文本执行诊断。

### 13.3 评论分析 agent

负责：保留原话，聚类正负反馈，区分目标读者和非目标读者。

输出：反馈聚类表。

### 13.4 策划 agent

负责：把数据、文本和反馈合并，生成可迁移经验、不可迁移条件和下一本闸门。

输出：完本复盘报告。

### 13.5 知识库维护 agent

负责：只把稳定结论写入专题文档、来源清单和决策记录；候选经验先放 notes 或项目复盘。

输出：知识库更新建议。

## 14. 复盘中的常见错误

| 错误 | 后果 | 修法 |
| --- | --- | --- |
| 只写剧情总结 | 没有学习价值 | 改写为目标、证据、偏差、行动 |
| 只看最终数据 | 忽略掉点和阶段问题 | 分阶段复盘 |
| 只看评论 | 被高声量样本带偏 | 评论、数据、文本三证合一 |
| 只记录失败 | 丢掉可复用资产 | 同时记录有效角色、关系、梗点和流程 |
| 只归因外部 | 下一本无法改进 | 转成可控系统问题 |
| 盲目复制成功 | 下一本同质化或错配 | 标注不可迁移条件 |
| 复盘太晚 | 问题已滚雪球 | 第 3、10、30 章和卷末做轻复盘 |
| 把候选写成规则 | 知识库污染 | 用 T0-T4 分级 |

## 15. 复盘结论写法

### 15.1 无效结论

```text
下次要加强人物。
中段不能水。
读者喜欢爽点。
要多写名场面。
```

### 15.2 有效结论

```text
第 18-25 章追读下降，评论集中为“主角像旁观者”。文本回查显示这 8 章中主角主动选择只有 2 次，支线对话占比过高。下一本进入第 15 章后，每 3 章必须至少有一次主角主动改变资源、关系、信息或风险状态；若支线连续两章不改变主线状态，压缩或合并。
```

有效结论必须包含：位置、证据、原因、可控动作、下一本闸门。

---

## 16. 知识库回写规则

复盘后，只有三类内容适合写回知识库：

1. 已被多来源或多项目证明的稳定规则。
2. 对 AI agent 有明显执行价值的流程、模板和检查问题。
3. 一旦违反会造成高成本的风险闸门。

不应写回知识库的内容：

- 某部作品的完整剧情。
- 未验证的个人感受。
- 一次性平台活动结论。
- 单条读者评论。
- 具体竞品桥段。
- 未授权或可能侵权的内容。

候选经验可以先写入 `notes/` 或单本项目复盘，等有更多证据再升级。

## 18. 最小规则

时间很少时，至少执行：

1. 复盘先写原始目标，再写实际结果。
2. 没有证据的结论标为候选。
3. 同时复盘成功资产和失败模式。
4. 追读/付费/评论/文本四类证据至少取两类互相校验。
5. 不把剧情总结当复盘。
6. 不把偶然成功当稳定规律。
7. 不把单部作品专属经验直接写进知识库硬规则。
8. 每条有效结论必须生成下一本闸门。
9. 多 agent 复盘不追责个人，追流程和系统。
10. 复盘后只把稳定、可执行、高价值结论回写知识库。
