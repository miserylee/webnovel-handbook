# 58. 写手、试读、审稿与改稿一体化工作流

状态：已确认  
最近整理：2026-06-09  
适用对象：AI 写作 agent、试读 subagent、审稿 subagent、修订 agent、总控 agent  
核心用途：把“写稿 -> 匿名读者试读 -> 编辑部审稿 -> 改稿”整合成一次内部章节生产闭环。后续写稿时，AI 不再要求用户手动切换读者群、编辑部和项目文档，而是在同一个工作流里自行完成试读、审稿、反馈聚类、取舍和修订。

---

## 1. 本工作流解决什么问题

过去的章节生产容易分裂成几个项目：

- 写手项目：负责起草正文。
- 匿名读者反馈样本：负责试读反馈。
- 匿名编辑反馈样本：负责商业审稿。
- 修订项目：负责改稿和同步台账。

这种拆法对人类管理有用，但对 AI agent 长期写稿会增加切换成本。用户每次都要手动把草稿搬去试读、再搬去审稿、再搬回来改稿，容易漏上下文、漏台账、漏有效资产。

本工作流将它们合并为一条默认链：

```text
章节任务单 -> 起草 agent -> 试读 subagents -> 审稿 subagents -> 反馈仲裁 -> 修订 agent -> 二次轻检 -> 台账/风格/下一章任务回写
```

一句话原则：

```text
写稿不是交出第一稿，而是交出经过内部试读、审稿和一次有边界修订后的可继续连载稿。
```

反 AI 味优先级：

```text
降 AI 味不是终稿润色，而是章节能否交付的质量门槛。高 AI 味稿件不能以“整体通顺”为理由交付。
```

执行含义：

- 起草前先确定本章的人物自保动作、场景摩擦、具体物件/流程、情绪触点和可控毛刺。
- 试读时如果判断 AI 味为 `高` 或 `极高`，反馈仲裁必须把它列入“必须修改”，不能降级为句子润色。
- 修订时先改人物选择、因果压力、对话反应和动作链，再处理词句顺滑。
- 稿子可以有少量有效粗糙，但不能平均正确、全员同声、情绪靠旁白、场景像纪要。

## 2. 默认启用条件

后续凡是用户要求以下任务，默认启用本工作流：

- 写一章正文。
- 续写下一章。
- 批量写多章。
- 重写第一章或前三章。
- 根据大纲生成章节。
- 写完后希望“商业级”“高质量”“降低 AI 味”“更像真人写”。
- 审稿后直接改稿。
- 需要减少读者群、编辑部、写手项目之间的切换。

可以简化的情况：

- 只写一个 300 字以内的试写片段。
- 只改一句话或一个段落。
- 只做设定 brainstorm。
- 用户明确要求“先不要审稿/不要改稿，只给草稿”。

即使简化，也要说明“本次跳过内部试读/审稿/修订”的原因。

## 3. 角色分工

本工作流里的 subagent 可以是真实多 agent，也可以由同一个 AI 按顺序切换角色。但每一步必须隔离职责，不能边写边审、边审边改。

| 角色 | 职责 | 输出 | 禁止事项 |
|---|---|---|---|
| 总控 agent | 判断阶段、打包材料、启动工作流、控制停止条件 | 本章任务单、必读规则、角色调度 | 不直接改正文 |
| 起草 agent | 根据任务单写读者可见正文 | 第一稿 | 不输出自检、解释、台账 |
| 目标读者试读 subagent | 模拟目标读者即时阅读反应 | 追读意愿、爽点、毒点、卡点 | 不给长篇写作理论 |
| 冷读者试读 subagent | 模拟第一次接触项目的移动端读者 | 看懂程度、第一屏吸引、跳读点 | 不替作者补设定 |
| 反 AI 味试读 subagent | 专查模板化、空泛、人物同质、纪要感 | AI 味等级、具体位置、硬门槛结论 | 不把个性文本改平 |
| 商业编辑审稿 subagent | 判断类型承诺、连载动力、章节回报和修改成本 | 审稿分级、优先级、是否可继续写 | 不逐句润色 |
| 连续性与风格审稿 subagent | 检查台账、人物声音、称呼术语、伏笔和事实 | 连续性问题、风格漂移、台账更新项 | 不做商业判断替代 |
| 合规与风险审稿 subagent | 检查平台合规、未成年人、隐私、危险教程化等 | 风险等级和删改建议 | 不因爽点放过风险 |
| 反馈仲裁 agent | 聚类试读和审稿意见，决定采纳、转译采纳或暂缓 | 修改清单和保留资产 | 不少数服从多数 |
| 修订 agent | 按修改清单执行小修、中修或重写 | 修订正文、变更说明 | 不扩大改稿范围 |
| 台账交接 agent | 回写事实、人物状态、伏笔、风格和下一章任务 | 台账更新项、下一章钩子 | 不复述整章 |

## 4. 标准流程

### 4.1 第 0 步：进入前检查

总控 agent 先确认：

| 检查项 | 必须回答 |
|---|---|
| 当前章节 | 第几章、处于前 10 章/前 30 章/卷中/卷末/终局哪一段 |
| 本章功能 | 证明卖点、兑现回报、升级阻力、推进关系、回收伏笔、转场或结算 |
| 输入材料 | 项目简报、上一章状态、台账、风格手册、章节任务单、相关专题 |
| 输出要求 | 只交定稿、交草稿加报告、还是交修订稿和台账项 |
| 禁止事项 | 不能提前揭示、不能改设定、不能改变人物关系、不能触碰合规风险 |
| 反 AI 味门槛 | 本章最容易模板化的位置、必须保留的人物毛刺、具体场景摩擦和动作链 |

缺少本章功能或上一章状态时，不直接写正文，先补任务单。

### 4.2 第 1 步：起草 agent 生成第一稿

起草 agent 只负责正文执行。

硬规则：

- 只输出读者可见正文。
- 起草前先把人物放进具体压力现场，而不是先写概括性旁白。
- 每场戏有目标、阻力、转折和结果。
- 主角必须有可见选择。
- 章末必须有落袋、反弹或下一步必须行动。
- 不在正文里写“本章意图”“伏笔说明”“作者解释”。
- 禁止用全员精准表达、整齐轮流发言、抽象总结和万能理解推进正文。
- 允许保留有效粗糙：抢话、误判、迟疑、动作卡顿、信息不完整和人情压力，但这些毛刺必须推动人物或冲突。

第一稿不是最终交付物，除非用户明确要求只看草稿。

### 4.3 第 2 步：试读 subagents 并行阅读

试读不是审稿。试读只回答“读者此刻怎么感受”。

默认三类试读：

| 试读 subagent | 关注问题 | 输出 |
|---|---|---|
| 目标读者 | 这章是否给到目标类型的爽、甜、燃、悬疑、情绪或期待 | 继续读/观望/想弃，附 1-3 个原因 |
| 冷读者 | 没有作者脑内设定时能不能看懂，哪里卡，哪里想跳 | 看懂程度、卡点、跳读点 |
| 反 AI 味读者 | 是否像提示词扩写、人物同声、情绪靠旁白、证据链像纪要 | AI 味等级、位置和读者体感 |

试读输出格式：

```text
【试读身份】
【一句话反应】
【最想继续看的点】
【最卡/最想跳的点】
【毒点或弃文风险】
【证据位置】
【建议动作】不改/微修/中修/重写/后文补偿
```

### 4.4 第 3 步：审稿 subagents 分层审稿

审稿回答“这章是否商业可用、结构可持续、风险可控”。

默认三类审稿：

| 审稿 subagent | 关注问题 | 输出 |
|---|---|---|
| 商业编辑 | 类型承诺、主角能动性、回报密度、章末追读、修改成本 | A/B/C/D 或可继续/需返修 |
| 连续性与风格编辑 | 时间线、人物知道什么、伏笔、称呼术语、角色声音 | 必须改、可保留、待确认 |
| 合规风险编辑 | 平台审核、未成年人、隐私、危险教程、版权和现实伤害 | 风险等级和处理动作 |

审稿输出必须先列有效资产，再列问题。

禁止输出：

- 只说“文笔不错但节奏可加强”。
- 不给证据位置。
- 把个人口味当成商业判断。
- 直接改正文而不经过反馈仲裁。

### 4.5 第 4 步：反馈仲裁

反馈仲裁 agent 把试读和审稿合并成可执行修改清单。

仲裁顺序：

1. 合规风险优先。
2. 反 AI 味硬门槛优先；如果 AI 味为 `高` 或 `极高`，先修人物、场景、因果、动作和声音，不先润句子。
3. 主承诺和连载动力优先。
4. 目标读者高频痛点优先。
5. 编辑结构反馈优先于单句口味。
6. 正反馈资产必须保护。
7. 低成本高收益修改优先。
8. 单条弱反馈暂缓观察。

输出格式：

```text
【必须保留】
【必须修改】
【转译采纳】
【暂缓观察】
【不采纳】
【本轮修订范围】
【不能改坏的资产】
```

### 4.6 第 5 步：修订 agent 改稿

修订 agent 只按仲裁结果改稿。

改稿模式：

| 模式 | 适用 | 动作 |
|---|---|---|
| 小修 | 错字、称谓、段落、局部 AI 味、单点卡顿 | 不改变章节结构 |
| 中修 | 场景目标弱、对话无效、情绪不到位、章末弱 | 保留章节功能，重排局部场景 |
| 大修 | 主承诺错、主角无目标、结构偏题、读者看不懂 | 先停下输出重写简报，不自动无限重写 |

默认只做一轮完整修订。若修订后仍有重大承诺层或结构层问题，停止并报告，不进入无限循环。

### 4.7 第 6 步：二次轻检

二次轻检不是重新审稿，只检查本轮修改是否引入新问题。

检查项：

- 修改是否解决了必须修改项。
- 是否误伤了必须保留资产。
- 是否引入新的连续性错误。
- 是否让人物声音变平。
- 是否还存在明显 AI 味或合规风险。
- 如果 AI 味仍为 `高` 或 `极高`，不能作为定稿交付；只能在本轮范围内再做小修，或停止并报告结构性人味阻塞。

二次轻检只允许小修，不开启第二轮大审稿。

### 4.8 第 7 步：台账、风格和下一章任务回写

如果本章改变了事实，必须输出回写项：

```text
【事实台账更新】
【人物状态更新】
【关系状态更新】
【伏笔新增/推进/回收】
【设定规则变更】
【风格手册新增】
【下一章必须承接】
```

正文文件只写读者可见内容；台账和说明写入项目文档或交接包。

## 5. 用户看到什么

默认情况下，用户不需要看全部内部试读和审稿报告。

推荐交付：

```text
1. 修订后的正文。
2. 简短说明：本轮内部试读/审稿发现并修了什么。
3. 台账和下一章任务更新项。
4. 如果有重大结构风险，明确说明是否建议暂停继续写。
```

如果用户明确要看过程，再输出完整试读报告、审稿报告和仲裁表。

原则：内部 subagent 是为了提高稿件质量，不是为了把用户淹没在报告里。

---

## 6. 停止条件

为了避免无限循环，本工作流设置硬停止条件。

正常停止：

- 完成第一稿。
- 完成内部试读。
- 完成内部审稿。
- 完成反馈仲裁。
- 完成一轮修订。
- 完成二次轻检和必要小修。
- 输出正文、变更说明和台账项。

提前停止：

- 本章任务单缺失，无法判断本章功能。
- 试读和审稿显示承诺层或结构层问题严重，继续润色会浪费。
- 合规风险无法通过小修解决。
- 需要用户决定主承诺、CP 方向、角色生死、重大设定或已发布内容大改。

禁止：

- 为追求完美反复开新审稿轮。
- 根据同一条弱反馈不停改。
- 一边改一边继续扩写新剧情。
- 修订后又让起草 agent 重写成另一章。

## 7. 与外部读者群和编辑部的关系

本流程不取消真实读者和编辑的价值，而是改变默认顺序。

默认：

```text
内部试读/审稿先跑 -> 形成更干净的稿子 -> 必要时再交真实读者或编辑
```

适合外部反馈的时机：

- 新书立项。
- 第一章或前三章方向判断。
- A/B 开篇版本。
- 上架前关键节点。
- 数据异常但内部诊断不确定。
- 编辑签约、推荐、出版、授权或合规判断。

不需要每章都切换外部项目。普通章节先走内部闭环即可。

## 8. 与既有专题的分工

- `38` 章节生产流水线：仍是章节生产总框架；本篇把试读、审稿和改稿整合成默认内部闭环。
- `08` 审稿与读者反馈：提供审稿标准和评分维度；本篇规定审稿 subagent 何时介入。
- `41` 评论语料标注：用于真实读者反馈或内部试读反馈的聚类格式。
- `42` 反馈冲突处理：用于试读、审稿、数据和主承诺冲突时仲裁。
- `33` 修订工作流：规定小修、中修、大修边界；本篇规定修订 agent 接什么输入。
- `36` 提示词与上下文打包：规定起草、试读、审稿和修订的上下文怎么分包。
- `57` 专题路由规范：规定本章需要读哪些知识库专题，避免全库塞上下文。

## 10. 给后续写稿 agent 的默认指令

```text
请按 `docs/58-integrated-drafting-beta-review-revision-workflow.md` 执行本章生产：
1. 先确认章节任务单和最小上下文包。
2. 起草 agent 只输出第一稿正文。
3. 内部调用目标读者、冷读者、反 AI 味读者三个试读 subagent。
4. 内部调用商业编辑、连续性风格编辑、合规风险编辑三个审稿 subagent。
5. 用反馈仲裁表合并意见，先保留有效资产，再决定修改项。
6. 修订 agent 完成一轮小修或中修。
7. 二次轻检只查是否引入新问题，不开启无限审稿。
8. 最终交付修订稿、简短变更说明、台账更新项和下一章承接点。
```

## 11. 最终口径

后续写稿时，默认不再让用户手动在读者群、编辑部、写手项目和修订项目之间切换。

AI agent 应当把这些角色内置成一次可控流水线：

```text
先写出能读的稿，再模拟读者读，再模拟编辑审，再由修订 agent 改，最后把状态交回项目。
```

用户只需要决定重大方向，不需要替 agent 搬运每一轮试读和审稿材料。
