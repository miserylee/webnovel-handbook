# 58. 写手、试读、审稿与改稿一体化工作流

状态：已确认  
最近整理：2026-06-09  
适用对象：AI 写作 agent、试读 subagent、审稿 subagent、修订 agent、总控 agent  
核心用途：把“写稿 -> 匿名读者试读 -> 编辑部审稿 -> 改稿”整合成一次内部章节生产闭环。后续写稿时，AI 不再要求用户手动切换读者群、编辑部和项目文档，而是在同一个工作流里自行完成试读、审稿、反馈聚类、取舍和修订。

---

## 1. 本工作流解决什么问题

过去的章节生产容易分裂成几个项目：

- 写手项目：负责起草正文。
- 匿名读者反馈样本：负责试读反馈。
- 匿名编辑反馈样本：负责商业审稿。
- 修订项目：负责改稿和同步台账。

这种拆法对人类管理有用，但对 AI agent 长期写稿会增加切换成本。用户每次都要手动把草稿搬去试读、再搬去审稿、再搬回来改稿，容易漏上下文、漏台账、漏有效资产。

本工作流将它们合并为一条默认链：

```text
章节任务单 -> 起草 agent -> 试读 subagents -> 审稿 subagents -> 反馈仲裁 -> 修订 agent -> 二次轻检 -> 台账/风格/下一章任务回写
```

一句话原则：

```text
写稿任务可以交修订后的可继续连载稿；审稿任务默认只交判断和修改建议；当用户提供样章或明确 taste 时，任何修订都必须先保护作者味，再处理硬问题。
```

反 AI 味优先级：

```text
降 AI 味不是终稿润色，而是章节能否交付的质量门槛。高 AI 味稿件不能以“整体通顺”为理由交付。
```

执行含义：

- 起草前先确定本章的人物自保动作、场景摩擦、具体物件/流程、情绪触点和可控毛刺。
- 试读时如果判断 AI 味为 `高` 或 `极高`，反馈仲裁必须把它列入“必须修改”，不能降级为句子润色。
- 修订时先改人物选择、因果压力、对话反应和动作链，再处理词句顺滑。
- 稿子可以有少量有效粗糙，但不能平均正确、全员同声、情绪靠旁白、场景像纪要。

作者味保护优先级：

```text
当用户提供了自己的样章、重写稿或明确 taste 时，作者味是高优先级资产。内部试读和审稿不得把有辨识度的人味误判为“不够工整”。
```

执行含义：

- 总控 agent 在进入审稿前，必须先抽取本稿的“有效作者味”：口语毛边、生活细节、人物小动作、轻松节奏、糙梗、队伍气氛、旁白距离。
- 试读 subagent 可以指出章末弱、推进慢、信息少，但必须区分“松弛有味”和“散乱无效”。
- 商业编辑 subagent 不能只因不符合标准模板就建议大改；必须先说明会不会误伤作者味。
- 反馈仲裁必须把“不能改坏的味道”写入保留资产。
- 修订 agent 默认只修硬伤和高收益小问题；除非承诺层或结构层已经失败，不得整体重写成另一种文风。

一句话原则：

```text
有人味但不够锋利的稿，先磨刃；没有人味但很工整的稿，必须返修。
```

逻辑连续性优先级：

```text
明显逻辑漏洞、前后错位、突兀信息和无理由句子都是交付阻塞，不是普通润色问题。
```

执行含义：

- 每场戏必须能说清“上一场留下什么压力，本场因什么启动，本场结果如何推到下一场”。
- 每个新信息必须有来源：谁知道、从哪里知道、为什么现在说出或看见。
- 每个关键动作必须符合物理顺序：人在何处，物件在何处，谁能看见，谁能听见，动作如何完成。
- 每个突兀句子必须能解释功能：推进动作、暴露人物、改变信息、制造阻力、兑现回报、转场或埋钩。解释不了就删、移位或补因果。
- 逻辑修复优先于文风润色；不能把错位句子改漂亮后继续保留。

## 2. 默认启用条件

后续凡是用户要求以下任务，默认启用本工作流：

- 已有项目简报、台账、风格手册和章节任务单时，写一章正文。
- 续写下一章。
- 批量写多章。
- 重写第一章或前三章。
- 根据大纲生成章节。
- 写完后希望“商业级”“高质量”“降低 AI 味”“更像真人写”。
- 用户明确要求审稿后直接改稿。
- 需要减少读者群、编辑部、写手项目之间的切换。

### 2.1 写稿默认最小阅读包

本工作流启用时，不先全库泛读。总控 agent 先按 `docs/00-index.md` 判断任务阶段，再读取下列最小包：

- 通用写稿硬包：`58`、`06`、`11`、`37`、`07`、`32`。
- 对白、人物声音、人味或 AI 味明显相关时，必须追加 `59` 和 `04`。
- 逻辑漏洞、前后错位、无理由句子明显相关时，必须追加 `05`，必要时追加 `26` 或 `63`。
- 涉及真实行业、制度、地域、平台规则时，先检索 `docs/00-expanded-topic-catalog.md` 定位一个目标专题，再读取目标文档。
- `docs/01-source-inventory.md` 和 `docs/10-templates-and-checklists.md` 只按关键词检索，不整篇读取。

起草前必须额外写清三件事：

1. 本章最容易出现 AI 味的位置。
2. 本章 2-3 个关键对白分别是谁在争什么、遮什么、逼对方产生什么反应。
3. 本章必须留下的下一章欲望，不允许只写“局势更复杂了”。

可以简化的情况：

- 只写一个 300 字以内的试写片段。
- 只改一句话或一个段落。
- 只做设定 brainstorm。
- 用户明确要求

即使简化，也要说明“本次跳过内部试读/审稿/修订”的原因。

如果用户要求写第一章或新项目正文，但项目简报、人物卡、设定台账、前 10 章方向、第一卷细纲、第一章任务单、风格手册或初始台账缺失，本工作流不进入“起草 agent”。先输出开书准备缺口诊断或补齐对应项目文档。

## 3. 角色分工

本工作流里的 subagent 可以是真实多 agent，也可以由同一个 AI 按顺序切换角色。但每一步必须隔离职责，不能边写边审、边审边改。

| 角色 | 职责 | 输出 | 禁止事项 |
|---|---|---|---|
| 总控 agent | 判断阶段、打包材料、启动工作流、控制停止条件 | 本章任务单、必读规则、角色调度 | 不直接改正文 |
| 起草 agent | 根据任务单写读者可见正文 | 第一稿 | 不输出自检、解释、台账 |
| 目标读者试读 subagent | 模拟目标读者即时阅读反应 | 追读意愿、爽点、毒点、卡点 | 不给长篇写作理论 |
| 冷读者试读 subagent | 模拟第一次接触项目的移动端读者 | 看懂程度、第一屏吸引、跳读点 | 不替作者补设定 |
| 反 AI 味试读 subagent | 专查模板化、空泛、人物同质、纪要感 | AI 味等级、具体位置、硬门槛结论 | 不把个性文本改平 |
| 商业编辑审稿 subagent | 判断类型承诺、连载动力、章节回报和修改成本 | 审稿分级、优先级、是否可继续写 | 不逐句润色 |
| 连续性与风格审稿 subagent | 检查台账、人物声音、称呼术语、伏笔和事实 | 连续性问题、风格漂移、台账更新项 | 不做商业判断替代 |
| 合规与风险审稿 subagent | 检查平台合规、未成年人、隐私、危险教程化等 | 风险等级和删改建议 | 不因爽点放过风险 |
| 反馈仲裁 agent | 聚类试读和审稿意见，决定采纳、转译采纳或暂缓 | 修改清单和保留资产 | 不少数服从多数 |
| 修订 agent | 按修改清单执行小修、中修或重写 | 修订正文、变更说明 | 不扩大改稿范围 |
| 台账交接 agent | 回写事实、人物状态、伏笔、风格和下一章任务 | 台账更新项、下一章钩子 | 不复述整章 |

## 4. 标准流程

### 4.1 第 0 步：进入前检查

总控 agent 先确认：

| 检查项 | 必须回答 |
|---|---|
| 当前章节 | 第几章、处于前 10 章/前 30 章/卷中/卷末/终局哪一段 |
| 本章功能 | 证明卖点、兑现回报、升级阻力、推进关系、回收伏笔、转场或结算 |
| 输入材料 | 项目简报、上一章状态、台账、风格手册、章节任务单、相关专题 |
| 输出要求 | 只交定稿、交草稿加报告、还是交修订稿和台账项 |
| 禁止事项 | 不能提前揭示、不能改设定、不能改变人物关系、不能触碰合规风险 |
| 反 AI 味门槛 | 本章最容易模板化的位置、必须保留的人物毛刺、具体场景摩擦和动作链 |
| 逻辑连续性门槛 | 上一章/上一场如何承接，本章因果链，新信息来源，角色知道什么，关键动作物理顺序 |

缺少本章功能或上一章状态时，不直接写正文，先补任务单。

### 4.2 第 1 步：起草 agent 生成第一稿

起草 agent 只负责正文执行。

硬规则：

- 只输出读者可见正文。
- 起草前先把人物放进具体压力现场，而不是先写概括性旁白。
- 每场戏有目标、阻力、转折和结果。
- 主角必须有可见选择。
- 章末必须有落袋、反弹或下一步必须行动。
- 不在正文里写“本章意图”“伏笔说明”“作者解释”。
- 禁止用全员精准表达、整齐轮流发言、抽象总结和万能理解推进正文。
- 允许保留有效粗糙：抢话、误判、迟疑、动作卡顿、信息不完整和人情压力，但这些毛刺必须推动人物或冲突。
- 新信息不能无来源空降；新判断不能无证据冒出；新句子不能只因为“看起来有文采”而存在。
- 每一段至少承担一个功能：动作、信息、阻力、人物、情绪、回报、转场或钩子。没有功能的段落先删。

第一稿不是最终交付物，除非用户明确要求只看草稿。

### 4.3 第 2 步：试读 subagents 并行阅读

试读不是审稿。试读只回答“读者此刻怎么感受”。

默认三类试读：

| 试读 subagent | 关注问题 | 输出 |
|---|---|---|
| 目标读者 | 这章是否给到目标类型的爽、甜、燃、悬疑、情绪或期待 | 继续读/观望/想弃，附 1-3 个原因 |
| 冷读者 | 没有作者脑内设定时能不能看懂，哪里卡，哪里想跳 | 看懂程度、卡点、跳读点 |
| 反 AI 味读者 | 是否像提示词扩写、人物同声、情绪靠旁白、证据链像纪要 | AI 味等级、位置和读者体感 |
| 逻辑冷读者 | 是否出现前后错位、突兀信息、动作顺序错误、无理由句子 | 逻辑断点、缺失因果、必须补/删/移的位置 |

试读输出格式：

```text
【试读身份】
【一句话反应】
【最想继续看的点】
【最卡/最想跳的点】
【毒点或弃文风险】
【证据位置】
【建议动作】不改/微修/中修/重写/后文补偿
```

### 4.4 第 3 步：审稿 subagents 分层审稿

审稿回答“这章是否商业可用、结构可持续、风险可控”。

默认三类审稿：

| 审稿 subagent | 关注问题 | 输出 |
|---|---|---|
| 商业编辑 | 类型承诺、主角能动性、回报密度、章末追读、修改成本 | A/B/C/D 或可继续/需返修 |
| 连续性与风格编辑 | 时间线、人物知道什么、伏笔、称呼术语、角色声音 | 必须改、可保留、待确认 |
| 合规风险编辑 | 平台审核、未成年人、隐私、危险教程、版权和现实伤害 | 风险等级和处理动作 |

审稿输出必须先列有效资产，再列问题。

禁止输出：

- 只说“文笔不错但节奏可加强”。
- 不给证据位置。
- 把个人口味当成商业判断。
- 直接改正文而不经过反馈仲裁。

审稿反例：

| 错误审稿输出 | 问题 | 正确做法 |
|---|---|---|
| “整体不错，建议再增强代入感。” | 空泛，修订 agent 不知道改哪里 | 指出哪一场缺人物压力、生活入口或视角锚点 |
| “这里有点 AI 味，换几个词。” | 把结构性 AI 味误判为词语问题 | 判断是人物同声、情绪旁白、因果太顺还是场景像纪要，并要求先修场景和动作 |
| “逻辑有点跳，但不影响阅读。” | 放过交付阻塞 | 标出跳跃句，补信息来源、角色认知或前后因果；无法补就删或移位 |
| “这一段写得漂亮，保留。” | 只看句子美感，不看功能 | 检查该段是否推进动作、信息、人物、情绪、转场或钩子；无功能则删 |
| “主角可以更主动。” | 结论对但不可执行 | 写清主角在哪个决策点被动，并要求补一个选择、代价或反制动作 |
| “读者应该知道前因。” | 用作者脑补替代文本证据 | 列出文本中已经给出的信息；缺少的信息必须补在角色可见的位置 |
| “反派降智。” | 只贴标签，没有修复路径 | 指出反派缺少利益、信息或风险依据，并补一个合理阻碍动作 |
| “节奏慢，压缩。” | 可能误删有效情绪或伏笔 | 先判断慢在重复确认、无功能段落、转场拖沓还是回报延迟，再定删改范围 |

稿文命中反例：

| 坏稿片段 | 命中问题 | 审查动作 |
|---|---|---|
| “她忽然明白，三年前那场火也是长老安排的。” | 逻辑跳跃；缺推理链；角色直接知道答案 | 补证据来源、误导线索或试探过程 |
| “失踪十年的国师突然出现，手里拿着能证明清白的玉佩。” | 信息空降；新人物和关键证据同时救场 | 拆成预置、争夺、部分证明和解释权攻防 |
| “她知道第三页翻开，自己的秘密就完了。” | 角色知识越界；没有说明她为何知道第三页 | 补偷看、参与、签字、被告知等来源；否则改成恐惧反应 |
| “她冲出大殿，反手把门闩插上。” | 动作顺序/空间错误 | 检查门闩位置、人物站位、视线和动作可拍性 |
| “风像迟来的手，抚平所有沉默。众人继续看账册。” | 无功能漂亮句；意象不改变场景 | 删除，或让意象影响物件、声音、视线或人物反应 |
| “她终于明白，真正可怕的不是背叛，而是人心。” | AI 味情绪旁白；抽象升华替代行为 | 改为动作、沉默、选择、代价或关系后果 |
| “众人纷纷震惊：怎么会这样？” | 全员同声；群像合唱 | 按利益层拆反应：维稳、自保、抢解释权、切割、观察 |
| “系统显示全部真相，众人无话可说。” | 证据链像纪要；没有争夺 | 证据只给部分信息，增加解释权争夺和反制 |
| “真正的风暴才刚刚开始。” | 假章末钩子；只有预告无动作 | 改成可承接动作、物件、来人、缺页、反制或旧伤触发 |
| “剧情到了这里，她还是去了。” | 主角被剧情推走 | 补人物目标、压力、代价、关系逼迫或必须抢先行动的理由 |
| “长老把真正账册扔给她：你不是想查吗？” | 反派降智送证据 | 给反派陷阱、误导、交换条件、自信依据或反制计划 |
| “她赢了，所有人都后悔了。” | 爽点无见证；回报未落袋 | 写清赢的结果、见证者分层反应、对手代价和回报凭证 |

### 4.5 第 4 步：反馈仲裁

反馈仲裁 agent 把试读和审稿合并成可执行修改清单。

仲裁顺序：

1. 合规风险优先。
2. 反 AI 味硬门槛优先；如果 AI 味为 `高` 或 `极高`，先修人物、场景、因果、动作和声音，不先润句子。
3. 逻辑连续性硬门槛优先；如果存在明显逻辑漏洞、信息空降、行动顺序错误或无功能句，先修因果和位置。
4. 主承诺和连载动力优先。
5. 目标读者高频痛点优先。
6. 编辑结构反馈优先于单句口味。
7. 作者 taste、有效毛边和正反馈资产必须保护。
8. 低成本高收益修改优先。
9. 单条弱反馈暂缓观察。

输出格式：

```text
【必须保留】
【必须修改】
【转译采纳】
【暂缓观察】
【不采纳】
【本轮修订范围】
【不能改坏的资产】
```

### 4.6 第 5 步：修订 agent 改稿

修订 agent 只按仲裁结果改稿。

改稿模式：

| 模式 | 适用 | 动作 |
|---|---|---|
| 小修 | 错字、称谓、段落、局部 AI 味、单点卡顿 | 不改变章节结构 |
| 中修 | 场景目标弱、对话无效、情绪不到位、章末弱 | 保留章节功能，重排局部场景 |
| 大修 | 主承诺错、主角无目标、结构偏题、读者看不懂 | 先停下输出重写简报，不自动无限重写 |

默认只做一轮完整修订。若修订后仍有重大承诺层或结构层问题，停止并报告，不进入无限循环。

### 4.7 第 6 步：二次轻检

二次轻检不是重新审稿，只检查本轮修改是否引入新问题。

检查项：

- 修改是否解决了必须修改项。
- 是否误伤了必须保留资产。
- 是否引入新的连续性错误。
- 是否让人物声音变平。
- 是否还存在明显 AI 味或合规风险。
- 如果 AI 味仍为 `高` 或 `极高`，不能作为定稿交付；只能在本轮范围内再做小修，或停止并报告结构性人味阻塞。
- 是否还存在明显逻辑漏洞、前后错位、信息来源缺失、动作顺序错误或无功能句。
- 如果逻辑问题仍会让读者疑惑“这句哪来的/为什么现在发生”，不能作为定稿交付。

二次轻检只允许小修，不开启第二轮大审稿。

### 4.8 第 7 步：台账、风格和下一章任务回写

如果本章改变了事实，必须输出回写项：

```text
【事实台账更新】
【人物状态更新】
【关系状态更新】
【伏笔新增/推进/回收】
【设定规则变更】
【风格手册新增】
【下一章必须承接】
```

正文文件只写读者可见内容；台账和说明写入项目文档或交接包。

如果内部试读、审稿或修订发现原大纲、细纲、章节排表不再适配当前正文，不要为了服从旧计划硬写。反馈仲裁 agent 必须判断这是：

- 局部执行调整：只改本章场景、对白、钩子或回报。
- 章节任务调整：需要更新下一章任务单或后续 1-3 章排表。
- 单元结构调整：需要更新前 10 章拍表、卷纲或阶段目标。
- 方向级调整：影响主承诺、核心机制、主要关系或包装承诺，必须暂停并让用户确认。

动态调整不是随意漂移。修订 agent 可以改正文，但涉及细纲、排表、台账和风格手册的变化，必须写入回写项；涉及方向级调整时，不得无声继续写。

## 5. 用户看到什么

默认情况下，用户不需要看全部内部试读和审稿报告。

推荐交付按用户任务切换：

```text
1. 只审稿：先给有效资产和硬问题，不改正文。
2. 直接改稿：交修订后的正文，并简短说明本轮主要修了什么。
3. 起草定稿：交正文，必要时附少量台账和下一章任务更新项。
4. 如果有重大结构风险，明确说明是否建议暂停继续写。
```

如果用户明确要看过程，再输出完整试读报告、审稿报告和仲裁表。

原则：内部 subagent 是为了提高稿件质量，不是为了把用户淹没在报告里。

---

## 6. 停止条件

为了避免无限循环，本工作流设置硬停止条件。

正常停止：

- 完成第一稿。
- 完成内部试读。
- 完成内部审稿。
- 完成反馈仲裁。
- 完成一轮修订。
- 完成二次轻检和必要小修。
- 输出正文、变更说明和台账项。

提前停止：

- 本章任务单缺失，无法判断本章功能。
- 试读和审稿显示承诺层或结构层问题严重，继续润色会浪费。
- 合规风险无法通过小修解决。
- 需要用户决定主承诺、CP 方向、角色生死、重大设定或已发布内容大改。

禁止：

- 为追求完美反复开新审稿轮。
- 根据同一条弱反馈不停改。
- 一边改一边继续扩写新剧情。
- 修订后又让起草 agent 重写成另一章。

---

## 7. 与外部读者群和编辑部的关系

本流程不取消真实读者和编辑的价值，而是改变默认顺序。

默认：

```text
内部试读/审稿先跑 -> 形成更干净的稿子 -> 必要时再交真实读者或编辑
```

适合外部反馈的时机：

- 新书立项。
- 第一章或前三章方向判断。
- A/B 开篇版本。
- 上架前关键节点。
- 数据异常但内部诊断不确定。
- 编辑签约、推荐、出版、授权或合规判断。

不需要每章都切换外部项目。普通章节先走内部闭环即可。


## 11. 最终口径

后续写稿时，默认不再让用户手动在读者群、编辑部、写手项目和修订项目之间切换。

AI agent 应当把这些角色内置成一次可控流水线：

```text
先写出能读的稿，再模拟读者读，再模拟编辑审，再由修订 agent 改，最后把状态交回项目。
```

用户只需要决定重大方向，不需要替 agent 搬运每一轮试读和审稿材料。


## 句式节奏进入内部闭环

写手、试读、审稿和改稿都要处理句式节奏，不允许把“成片短句”留到最终润色阶段。

- 起草 agent：避免把提纲逐条切成短句；一段内要有动作链、观察、情绪遮掩和关系反馈。
- 试读 subagent：从读者体感判断是否“像 AI 在报点”，尤其关注连续短句、机械断行和情绪口号。
- 审稿 subagent：把明显成片短句列入 AI 味风险，不得用“清楚”“节奏快”直接放行。
- 修订 agent：先合并无功能短句，补足承接，再保留少量短句作为打击、转折或章末落点。
- 总控 agent：如果正文一整屏都短、平、齐，即使剧情逻辑没错，也不能作为可发布稿。

## 正稿事实核对与台账回写补充

一体化写作闭环在第 0 步进入前检查中，必须加入“正稿事实核对”。这一步优先级高于起草、试读和审稿。

### 正稿事实核对

起草或续写前，总控 agent 必须确认：

1. 当前人工正稿是哪几个文件。
2. 上一章真实章末落点是什么。
3. 当前章节要承接的是正文事实，不是大纲预期。
4. 哪些事件只是讨论过，尚未写进正稿。
5. 活跃包袱、主线支线状态和人物关系状态是否有台账记录。

如果无法确认正稿版本或上一章真实落点，不应直接起草。应先请求用户确认，或先建立/更新连载台账。

### 试读与审稿新增检查

内部试读和审稿除了看爽点、人味、AI 味，还必须检查：

- 是否把未发生事件当成已发生。
- 是否跳过上一章章末钩子。
- 是否让角色知道他们在正文中尚未获得的信息。
- 是否拿大纲安排压过当前章节节奏。
- 是否忽略作者改稿后的新版本。

### 修订后回写

修订 agent 完成正文后，台账交接 agent 必须输出或写入：

- 章节节奏更新。
- 关键节点推进更新。
- 包袱新增、推进、回收或废弃。
- 主线支线推进状态。
- 下一章必须承接点。
- 讨论过但仍未写入正文的内容。

如果用户后续再改稿，必须以用户改稿后的版本重新回写台账，不能沿用 AI 修订稿的节奏判断。
