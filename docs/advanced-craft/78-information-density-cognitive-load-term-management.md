# 78. 信息密度、认知负荷与专有名词投放专项

状态：已确认

适用对象：AI 写作 agent、大纲 agent、世界观 agent、审稿 agent、开篇 agent、修订 agent、风格手册 agent、多 agent 交接 agent。

适用任务：开篇设计、世界观投放、设定说明、专业题材、悬疑规则、规则怪谈、玄幻等级、组织势力、群像登场、长篇换地图、新卷启动、读者反馈为“看不懂”“设定太多”“人名太多”“名词太密”“信息过载”的稿件修订。

## 1. 本文解决什么问题

AI 写长篇网文时，常常把“资料完整”误当成“正文有效”。它会一次性塞入设定、人名、地名、组织名、等级名、道具名、系统规则、历史背景和专业术语。作者觉得世界完整，读者却在第一屏就失去方向。

典型问题：

- 第一章同时出现太多人名、组织名、地名和规则名。
- 角色还没行动，旁白先解释世界历史、等级体系和势力格局。
- 术语看似专业，但没有绑定动作、后果或选择。
- 悬疑/规则怪谈里规则太多，读者不知道当前要追哪个问题。
- 世界观名词很酷，但每个都需要解释，阅读成本高于爽点回报。
- 换地图或新卷启动时像重新开书，旧资产断线，新名词轰炸。
- AI 用括号、破折号、说明句反复补定义，正文像设定文档。
- 多 agent 接力时术语、称呼和专名不一致，读者记忆负担加倍。

本文把 cognitive load、working memory、plain language、concision、worldbuilding exposition、info dump 和中文网文开篇/设定经验，转化为 AI agent 可执行的信息负载管理规则。

## 2. 核心原则：信息必须按读者当前任务投放

最小链条：

```text
读者当前问题 -> 本场行动压力 -> 必要信息 -> 场景化承载 -> 即时用途 -> 后文回响
```

不要问“这个设定是不是重要”，先问：

```text
读者现在不知道它，会不会看不懂当前冲突、选择或回报？
```

如果不会，延后。

## 3. 三类信息负荷

借用认知负荷理论的思路，网文信息可以分成三类。

### 3.1 必要复杂度

读者必须理解，否则当前场景不成立。

示例：

- 比赛规则中决定胜负的一条限制。
- 系统任务的失败惩罚。
- 宗门名额为什么现在必须争。
- 证据链中本章要用的一环。

处理：可以保留，但要场景化、短句化、行动化。

### 3.2 无效负担

信息本身可能有趣，但此刻不服务当前场景。

示例：

- 本章用不到的十级等级体系。
- 出场人物的全家族谱。
- 组织历史、旧战争、地图百科。
- 过多同义称号和代号。

处理：删、后移、拆分、台账保存，不要挤进正文。

### 3.3 有效悬念负荷

读者暂时不完全理解，但愿意继续追，因为问题清楚、回报明确。

示例：

- 角色听到一个陌生代号，但马上带来危险反应。
- 规则只露一半，但主角必须立刻试探。
- 某个旧物没有解释，但人物反应说明它重要。

处理：可以保留，但必须给读者一个可追问题和近期回响。

## 4. 信息预算：一场戏不要同时教太多东西

AI 起草前，先给本场设信息预算。

推荐上限：

- 新重要人物：1 至 3 个。
- 新组织/势力：0 至 2 个。
- 新规则：1 个主规则，最多 1 个例外。
- 新术语：1 至 3 个，且必须绑定动作或后果。
- 新地图名：1 个主地点，其他可用普通称呼。
- 新伏笔：1 个主钩子，最多 1 个轻提示。

这些不是绝对硬杠，而是默认安全线。超过时必须有商业理由：强悬疑、群像大场面、规则怪谈入场、历史揭秘或卷级交接，并且要给读者明确锚点。

## 5. 专有名词投放规则

### 5.1 第一次出现：不要先解释全定义

错误方向：

```text
玄冥司，是大胤王朝负责监察修士、管理灵契、审判邪修的三司之一，创立于三百年前……
```

更稳方向：

```text
听见“玄冥司”三个字，茶楼里的人先收了声。
```

第一步先给功能感：它让谁害怕、谁受益、谁改变行动。

### 5.2 第二次出现：补一条可用规则

```text
玄冥司的人不抓凡人，除非案子牵到灵契。
```

只补当前场景要用的一条，不补百科。

### 5.3 第三次出现：让规则产生后果

```text
主角用“凡人不归玄冥司管”反卡对手，但也暴露自己懂灵契律。
```

名词真正进入读者记忆，不靠解释，而靠后果。

## 6. 信息锚点：读者需要抓手

新信息越多，越需要锚点。

常用锚点：

- 角色锚点：谁和这个信息绑定？
- 目标锚点：它影响主角要做什么？
- 风险锚点：不知道它会有什么危险？
- 情绪锚点：谁听见后害怕、愤怒、心虚或兴奋？
- 物件锚点：是否有一个具体物件承载信息？
- 空间锚点：它发生在什么可见位置？
- 结果锚点：本章它改变了什么？

没有锚点的信息，很容易变成读者的短时记忆负担。

## 7. 信息顺序：先让读者知道自己在追什么

读者可以暂时不知道答案，但不能不知道问题。

优先顺序：

```text
当前目标 -> 当前阻力 -> 必要规则 -> 角色选择 -> 后果 -> 更大设定
```

高危顺序：

```text
世界历史 -> 势力大全 -> 等级体系 -> 名词定义 -> 主角终于出场
```

尤其开篇和新卷启动，不要先把地图摊开。先让主角在地图里撞到问题。

## 8. 不同场景的信息密度策略

### 8.1 开篇

目标：让读者知道主角是谁、遇到什么问题、为什么要继续看。

信息策略：

- 少名词，多处境。
- 少背景，多冲突。
- 少解释，多动作后果。
- 第一屏优先给主角压力，不给世界百科。

### 8.2 玄幻/仙侠/奇幻设定

目标：让规则服务升级、资源、身份和胜负。

信息策略：

- 等级名不要一口气报全。
- 法宝/功法第一次出现先给效果和代价。
- 宗门/势力名先给行动功能：谁管、谁怕、谁能调资源。
- 规则例外要尽早回响，否则像空降。

### 8.3 都市/职场/专业题材

目标：让专业信息增加可信度，而不是变成资料展示。

信息策略：

- 术语最多少量，且绑定流程、工具、风险和人情。
- 外行人物可以提问，但不能写成作者讲课。
- 专业判断要带代价：误用术语、错流程、超权限会出事。

### 8.4 悬疑/规则怪谈

目标：让读者知道正在追哪个问题，并拥有可判断材料。

信息策略：

- 一次只突出一个核心异常或规则冲突。
- 真线索、红鲱鱼和背景信息要分层。
- 规则可以不全，但试探动作要清楚。
- 读者看不懂不是高级，能误读才是有效悬疑。

### 8.5 群像和多势力

目标：让读者先抓住功能，再记住名字。

信息策略：

- 先给功能称呼：穿红甲的、主位上的、管账的。
- 等角色产生动作或后果后，再给正式名。
- 同场新人物按利益分层，不要同时给全名和背景。
- 群像章节需要“谁最重要”的视角锚点。

### 8.6 新卷/新地图

目标：承接旧资产，打开新规则。

信息策略：

- 先带入旧目标、旧代价或旧关系债。
- 新地图先给第一场压力，不给导游词。
- 新组织名和规则名要分批出现。
- 新卷前五章逐步建立地图，不在第一章塞完。

## 9. 名词密度红线

高危信号：

- 一段里出现 3 个以上未解释新专名。
- 一个自然段同时引入人名、地名、组织名、规则名和历史事件。
- 同一个对象有多个称呼，且没有样式表。
- 术语第一次出现没有动作或后果。
- 解释文字比场景行动更长。
- 读者需要翻回去才能知道谁是谁。
- 作者用括号连续补定义。

修复动作：

- 用功能称呼临时代替正式名。
- 把新名词拆到不同场景。
- 删除本章用不到的背景。
- 第一次只给反应和功能，第二次补规则，第三次给后果。
- 写入风格手册或术语表，避免多 agent 改名。

## 10. 信息回响：新信息要在近期用一次

新名词、新规则、新组织、新人物进入正文后，最好在 1 至 3 章内至少产生一次回响。

回响可以是：

- 被角色使用。
- 被对手误用。
- 造成代价。
- 触发限制。
- 成为谈判筹码。
- 引发旁观者反应。
- 进入台账或风格手册。

如果新信息长时间不使用，读者会认为它是噪音。

## 11. AI 信息减负流程

第一步：列清单。

```text
本章新人物：
本章新地名：
本章新组织：
本章新规则：
本章新术语：
本章新伏笔：
```

第二步：判必要性。

```text
读者现在不知道它，会不会看不懂当前冲突/选择/回报？
```

第三步：判承载方式。

```text
它是通过动作、后果、反应、对话、物件，还是旁白定义出现？
```

第四步：判近期用途。

```text
它在本章或未来 1 至 3 章内会不会被用一次？
```

第五步：处理。

- 必要且有用途：保留并场景化。
- 必要但太重：拆分，先给功能后补定义。
- 非必要但未来有用：移入台账，后文再投放。
- 非必要且无用途：删除。

## 12. 读者反馈对照表

| 反馈 | 可能问题 | 优先修复 |
|---|---|---|
| 看不懂 | 当前问题不清，信息顺序错 | 先明确目标和阻力，再补必要规则 |
| 名词太多 | 新专名过密 | 降低新名词数量，用功能称呼过渡 |
| 设定太重 | 世界百科压过场景 | 删除本章用不到的背景，改成行动后果 |
| 太绕 | 线索、规则、人物关系同时展开 | 分层，一场只解决一个读者问题 |
| 记不住人 | 人物功能不清 | 先给功能动作，再给名字和背景 |
| 不知道爽在哪 | 规则没有转化成收益 | 把规则绑定胜负、资源、身份或代价 |
| 像说明书 | 旁白定义过多 | 改成对话冲突、物件、流程错误或后果验证 |
| 前面白讲了 | 新信息无回响 | 1 至 3 章内安排使用或删除 |

## 14. 最小执行规则

1. 新信息必须服务当前冲突、选择、回报或近期伏笔。
2. 读者可以暂时不知道答案，但不能不知道当前问题。
3. 第一次出现专名，优先给功能、反应或后果，不给百科定义。
4. 一场戏默认只引入少量关键新名词，超过预算必须有商业理由。
5. 术语必须绑定动作、流程、风险、代价或人物身份。
6. 本章用不到的背景进入台账，不进入正文。
7. 新信息最好在 1 至 3 章内回响一次，否则读者会当作噪音。
8. 多 agent 接力时，专名、称呼、术语和格式必须进入风格手册或样式表。
9. 修复“看不懂”时，先查信息顺序和当前问题，不要直接追加大段解释。
10. 有效复杂度可以保留，无效负担必须删除、延后或场景化。

## 15. 来源与吸收方式

外部来源主要提供四类启发：

- Cognitive Load Theory 相关资料：学习和理解会受到工作记忆限制影响，复杂信息需要减少无关负担，并按任务需要组织。
- Purdue OWL 关于 concision、clarity 和 plain style 的资料：清楚表达需要删除冗余、控制句子负担、让读者快速理解重点。
- Reedsy、Writer's Digest 关于 worldbuilding、exposition 和 info dump 的资料：世界观说明应随行动逐步出现，避免在故事停下时灌输背景。
- 起点社区、番茄作家专区和本知识库既有网文经验：开篇和章节信息投放必须服务读者留存，名词密度、设定解释和专业术语都要绑定场景压力和商业回报。

本文未照搬来源原文，而是将这些原则改写为中文商业网文 AI agent 可执行的信息负载系统：先确认读者当前问题，再给必要信息；先用功能、动作和后果建立名词，再逐步补规则；把无效负担移出正文，用台账和样式表保护长篇一致性。
