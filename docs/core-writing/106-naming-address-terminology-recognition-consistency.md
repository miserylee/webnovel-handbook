# 106. 命名、称谓、术语与可识别性一致性专项

状态：已确认

适用范围：新书设定、角色命名、群像登场、家族/宗门/公司/组织命名、地名、功法技能名、系统面板、专业术语、古代称谓、亲密关系称呼、外号梗、读者反馈“记不住人”“名字太像”“称呼乱”“术语太多”“设定名像乱码”、多 agent 接力中的改名和术语漂移。

核心结论：名字不是装饰，而是读者识别系统。AI agent 给人物、势力、地点、能力、术语和称谓命名时，必须同时考虑读者记忆负担、类型气质、文化/时代可信度、功能可见性、关系温度、口碑可复述性和长篇样式表一致性。能用功能称呼时，不急着命名；必须命名时，先让读者知道它的作用、关系和后果。

最小链路：命名必要性 -> 读者当前任务 -> 名字预算 -> 功能锚点 -> 音形差异 -> 称谓规则 -> 术语首次绑定动作/后果 -> 样式表回写 -> 后文再识别。

---

## 1. 为什么需要独立专题

已有第 37 篇管理风格手册和称呼术语样式表，第 78 篇管理信息密度和术语投放，第 87 篇管理角色登场和名字预算，第 100 篇管理正典事实。但命名问题仍值得独立成篇，因为它横跨多个系统：

- 角色名字太多，读者分不清谁是谁。
- 多个名字音形太近，移动端扫读时混淆。
- 同一个角色在不同章节被 AI 改名、改姓、改称谓。
- 古言/玄幻/修仙里称谓、辈分、官职、宗门职位乱用。
- 职业文和行业文堆术语，读者只觉得门槛高。
- 系统文、修仙文、无限流一次性抛出等级名、道具名、技能名、组织名。
- 为了显得原创，生成复杂生僻专名，反而降低可读性。
- 外号和亲昵称呼没有关系变化，只是作者硬发糖或硬搞笑。
- 竞品专名被改几个字就使用，引发融梗和原创风险。

本篇负责把“命名”从灵感行为改为可维护的识别工程。

## 3. 来源吸收与迁移边界

### 3.1 Reedsy / Writer's Digest：角色名要服务类型、文化和记忆

写作资料通常建议：角色名应符合人物背景、时代、文化、类型气质和读者记忆；避免过多相似名字；主要人物的名字应能帮助读者形成印象。

迁移规则：中文网文命名不应只追求好听、生僻或仙气。主角、反派、CP、导师、重要配角的名字要和身份、气质、关系功能、读者记忆标签相配。

### 3.2 PlainLanguage.gov / NN/g：熟悉词和低认知负担更易理解

Plain language 和可用性资料强调：尽量使用读者熟悉的词，避免不必要的行话；界面和文本应支持识别而不是强迫回忆。

迁移规则：专有名词、等级、机构、术语第一次出现时，不应让读者靠背词表理解。先给功能、动作、后果，再给正式名称。

### 3.3 SFWA / 编辑样式表资料：长篇需要 style sheet

编辑和写作资料强调，长篇作品需要样式表记录拼写、人物名、地点、特殊词、大小写、格式和一致性。

迁移规则：多 agent 写长篇时，人名、称谓、外号、地名、组织名、术语、等级名、系统字段和特殊格式必须进入样式表或项目台账。模型记忆不可靠。

### 3.4 起点/番茄和本知识库既有经验：移动端阅读需要名字预算

中文网文移动端阅读节奏快，章节内如果一次投放太多新名字、组织和术语，会直接增加跳读和弃读风险。

迁移规则：单场景新增需要记住的名字默认 1 到 3 个；非关键人物优先用功能称呼，等其产生状态变化或后文复用时再命名。

## 4. 概念定义

### 4.1 正式名

角色、地点、组织、功法、技能、道具、规则或术语在项目 canon 中登记的标准名称。

### 4.2 功能称呼

在读者还不需要记住正式名时，用作用或位置代替名称。

例子：管账的、红甲少年、坐主位的长老、项目负责人、拿钥匙的人、隔壁床的病人。

### 4.3 关系称呼

人物之间基于关系、权力、亲密度、情绪和场合使用的称谓。

例子：师父、师尊、老师、陈总、哥、阿姐、殿下、老谢、那位谢先生。

### 4.4 读者识别名

读者评论和记忆中实际使用的标签或外号。

例子：账本女主、红甲少年、倒霉师兄、那只乌鸦、会拆规则的男主。

### 4.5 样式表名

为了多 agent 一致性而记录的标准用法，包括正式名、别名、禁用写法、首次出现章节、关系称呼和变更条件。

## 5. 命名前的三个问题

AI agent 给任何东西命名前，先问：

1. 读者现在必须记住它吗？
2. 它现在的功能是什么？
3. 后文是否会复用、回响或改变状态？

如果答案是“不必须、不明确、不复用”，暂时不要正式命名。

可先写：

```text
门口那个穿灰色制服的女人没有报名，只把一张空表推到主角面前。
```

不要急着写：

```text
灰衣女子名叫沈璃霜，是天策司第三分署外勤审查组三等巡录。
```

## 6. 名字预算规则

| 场景 | 默认预算 | 说明 |
| --- | --- | --- |
| 单场景新增需要记住的人名 | 1-3 个 | 群像或会议也应分层命名 |
| 单章新增重要角色名 | 1-3 个 | 其他人用功能称呼 |
| 单章新增地名 | 1 个主地点 | 其他地点可用方位或功能 |
| 单章新增组织/势力名 | 1 个 | 必须绑定利益、资源或威胁 |
| 单章新增术语/规则名 | 1-3 个 | 必须绑定动作、风险或后果 |
| 单章新增技能/功法/道具名 | 1-2 个 | 必须马上产生效果或限制 |

超过预算时，必须有群像、战争、组织换卷或重大公开场景理由，并提供视角锚点。

## 7. 好名字的六个条件

### 7.1 可识别

读者扫一眼能和其他名字分开。

检查：

- 是否与已有重要人物同姓、同音、同字形？
- 是否连续出现多个“清、寒、月、霜、云、玄、夜、尘”类相近字？
- 是否有两个角色都叫“小X”“阿X”“老X”，且功能接近？

### 7.2 可承载

名字能承载人物功能、气质、社会位置或类型味，但不要直白到像标签。

风险：反派叫“恶绝天”、女主叫“白纯善”这类过度命名会显廉价，除非是喜剧或讽刺。

### 7.3 可读

移动端读者能顺畅读出，不被生僻字、长串外文、复杂符号和过密标点打断。

原则：生僻字、异体字、长复合名和特殊符号慎用。若必须用，给稳定读法和功能锚点。

### 7.4 可复用

名字、外号或称谓能在后文回响，而不是出现一次就消失。

检查：后文是否会在关系变化、公开评价、口碑传播或情绪回响中再次使用？

### 7.5 可维护

多 agent 不容易写错，样式表能清楚记录。

风险：同一角色有太多别名、尊称、乳名、外号、化名和身份名，会显得丰富，也会让接力写作崩盘。

### 7.6 可原创

不近似竞品专名、影视游戏 IP、现实名人或高识别梗。

涉及相似风险时，调用第 102 篇。

## 8. 不同对象的命名规则

### 8.1 主角名

主角名需要稳定、易记、可反复出现，不宜过度复杂。

检查：

- 是否和书名、主承诺、题材气质不冲突？
- 是否适合读者评论区称呼？
- 是否有自然简称、外号或关系称呼？
- 是否避免与同类爆款主角名过近？

### 8.2 重要配角名

配角名应和功能、阵营、关系和第一印象绑定。

例：

```text
不要只写：陆知遥出场。
推荐：陆知遥把会议室唯一一把备用椅推给主角，自己站到了门边。
```

读者先记住动作，再记名字。

### 8.3 反派/对手名

反派名不能只靠凶狠。更重要的是资源、策略和压迫方式。

检查：读者听到名字时，能否想起“他怎么威胁主角”，而不是只想起“坏人”。

### 8.4 组织/势力名

组织名先给功能，再给全称。

错误：

```text
玄曜天衡司、赤霄内府、九曜执律院同时登场。
```

推荐：

```text
先来的不是宗门的人，而是管刑罚的那一支。弟子们私下叫他们“黑令”。正式名字，等那块黑牌落到桌上时才出现：九曜执律院。
```

### 8.5 地名

地名要服务行动、资源、危险或情绪，不只是地图贴纸。

检查：

- 这个地点有什么功能？
- 到这里会改变信息速度、资源成本、权力关系或风险吗？
- 如果只是临时路过，是否可以用“驿站”“外城”“旧码头”代替正式名？

### 8.6 技能、功法、道具名

名称必须和效果、限制、代价绑定。

不要只写：

```text
主角使出九转玄冥碎星诀。
```

推荐：

```text
他没有硬接，先把灵力压进掌心第三道裂纹。那是“碎星”最危险的用法：能破阵，也会把自己的经脉震出暗伤。
```

读者先知道效果和代价，再记名字。

### 8.7 专业术语

术语必须绑定流程、工具、风险和后果。

如果术语只是炫专业，删掉或换成场景动作。

### 8.8 系统字段和面板名

系统文里，字段越稳定越好。

要求：

- 同一字段不随章节换名。
- 面板字段数量克制。
- 新字段出现要立刻改变选择或风险。
- 系统提示格式进入样式表。

## 9. 称谓矩阵

重要关系必须建立称谓矩阵：

```text
人物 A：
人物 B：
公开称呼：
私下称呼：
冲突时称呼：
亲密时称呼：
疏离时称呼：
身份场合称呼：
禁用称呼：
称呼变化条件：
首次变化章节：
```

### 9.1 称谓是关系温度计

称谓变化要有原因：关系推进、地位变化、背叛、和解、公开承认、私密保护、权力反转或情绪失控。

不要为了发糖突然改称呼，也不要为了虐突然乱改。

### 9.2 场合决定称谓

同一人物在不同场合可以不同称呼，但规则要稳定。

例：

- 朝堂：殿下。
- 私下：阿衡。
- 冲突：太子殿下。
- 心软：衡哥。
- 决裂：姓氏 + 职衔。

### 9.3 禁用称呼同样重要

有些称呼不能用，反而能形成张力。

例：旧爱不再叫乳名，师徒不敢直呼姓名，家族成员故意用官称制造距离。

## 10. 外号和读者梗

外号可以成为口碑资产，但必须来自正文事实。

有效外号来源：

- 角色反复行为。
- 公开事件。
- 关系互动。
- 社群误读。
- 对手羞辱被主角反转。
- 读者评论自发形成。

风险：

- 外号冒犯目标读者。
- 外号只靠作者硬塞。
- 外号复制竞品高识别梗。
- 外号过度消费，变成尬梗。

处理：读者自发外号可作为候选口碑资产，进入第 103 篇台账；正式回收前要检查是否符合人物和价值边界。

## 11. 名字漂移和改名处理

### 11.1 禁止静默改名

同一角色、组织、地点、技能或规则，不能因为 AI 续写时想起不同名字就静默改掉。

发现冲突时：

1. 查样式表和 canon。
2. 确定已确认正式名。
3. 把错误名标为废弃或误写。
4. 若必须剧情内改名，记录变更原因、时间、公开范围和后果。

### 11.2 剧情内改名必须有意义

改名可以成为剧情事件：继承身份、脱离家族、伪装潜入、关系重构、地位变化、公开昭雪。

但改名后需要：

- 正文解释读者为什么要更新记忆。
- 旧名在过渡期继续出现一段时间。
- 样式表记录旧名、现名、别名、使用场合。

## 12. 新名字登场流程

### 12.1 第一次出现

推荐顺序：功能 -> 压力 -> 动作 -> 名字。

```text
管账的人终于抬头。她没看主角，只把账册翻到空白那页：“签这里。”
她叫周映，外城所有租契都要从她手里过。
```

### 12.2 第二次出现

用上次锚点帮读者再识别。

```text
周映又在翻账册。主角这次先看她指尖停在哪一栏。
```

### 12.3 跨多章回归

不要只报名字。至少给一个旧动作、关系、称呼、物件或后果。

```text
周映把那本租契账册放到桌上，封皮比上次多了一道刀痕。
```

## 13. 术语首次出现规则

术语第一次出现，必须至少绑定以下一种：

- 动作：角色做了什么。
- 后果：它改变了什么。
- 风险：误用会怎样。
- 资源：谁能调动它。
- 身份：谁有资格使用。
- 情绪：为什么人物在意。

不要给百科定义。

错误：

```text
灵契是修士与本命灵器之间通过神魂绑定形成的契约体系。
```

推荐：

```text
主角刚碰到剑柄，掌心就被烫出一道细线。旁边的人脸色变了：“别碰，那是灵契。认主的东西，外人碰一下就要赔半条命。”
```

## 15. 审稿检查

### 15.1 命名可读性

- 本章新增需要记住的名字是否超过预算？
- 重要名字之间音形是否过近？
- 是否有生僻字、长串名、复杂专名影响扫读？
- 新名字是否先有功能、动作或关系？
- 是否存在只出现一次但被正式命名的路人？

### 15.2 称谓一致性

- 同一人物对同一对象的称呼是否符合关系阶段和场合？
- 称呼变化是否有剧情原因？
- 是否出现 AI 随机换称谓？
- 关系推进是否能通过称呼变化被读者感知？

### 15.3 术语和专名负担

- 新术语是否绑定动作、后果或风险？
- 专业术语是否只是装饰？
- 系统字段是否稳定？
- 组织名和地名是否有功能？
- 是否需要写入样式表？

### 15.4 原创性风险

- 专名是否近似竞品、影视游戏 IP 或现实机构？
- 是否只是把竞品名称改一个字？
- 是否存在整套命名体系相似？

有风险时，调用第 102 篇。

## 16. 常见错误与修法

| 错误 | 读者体感 | 修法 |
| --- | --- | --- |
| 一章十几个新名字 | 记不住、跳读 | 降到 1-3 个，其余用功能称呼 |
| 名字都很仙/都很酷 | 人物糊成一片 | 拉开音形、气质、身份和功能 |
| 只给名字不给行为 | 像名单 | 先给动作、压力或关系 |
| 称呼随机变化 | 关系线混乱 | 建称谓矩阵和变化条件 |
| 术语堆叠 | 看不懂、像设定表 | 每个术语绑定动作或后果 |
| 生僻字过多 | 移动端卡顿 | 重要名保持可读，生僻字慎用 |
| 地名像地图贴纸 | 世界空 | 地点必须改变资源、风险或关系 |
| 外号硬塞 | 尬梗 | 让外号来自事件和读者复述 |
| AI 静默改名 | 连续性崩 | 查样式表，错误名标废弃 |
| 改竞品专名 | 融梗风险 | 重建命名规则和机制体系 |

---

## 17. 多 agent 分工

### 17.1 策划 agent

负责：制定命名规则、名字预算、称谓矩阵、术语表和禁用近似名。

### 17.2 起草 agent

负责：按预算投放新名字；第一次出现先给功能和动作；新增正式名后标记回写。

### 17.3 审稿 agent

负责：检查名字混淆、称谓漂移、术语堆叠、样式表缺失和原创性风险。

### 17.4 连续性 agent

负责：维护样式表，处理改名、别名、废弃名和剧情内称谓变化。

### 17.5 包装 agent

负责：检查书名、角色名、势力名和卖点词是否容易被读者复述，是否有竞品近似风险。

## 18. 最小规则

时间很少时，至少执行：

1. 单场景新增需要记住的名字默认 1-3 个。
2. 不重要、不复用的人物优先不命名。
3. 新专名第一次出现先给功能、动作或后果。
4. 重要名字之间必须拉开音形和功能。
5. 同一对象只能有一个正式名，别名和称谓必须登记。
6. 称呼变化必须有关系、场合或情绪原因。
7. 系统字段、技能名、地名、组织名进入样式表。
8. 不用生僻复杂命名证明世界观高级。
9. 不把竞品专名改字后使用。
10. 多 agent 接力前必须读取并更新命名样式表。
