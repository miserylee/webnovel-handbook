# 100. 正典事实、不确定性、设定冲突与 AI 幻觉控制专项

状态：已确认

适用范围：单本小说项目初始化、长篇续写、多 agent 接力、章节起草、台账维护、设定补全、读者反馈“前后矛盾”、审稿发现“角色知道不该知道的事”、AI 续写编造旧情节、断更恢复、跨卷承接、设定重写、已发布正文修补、正式发布前连续性质检。

核心结论：长篇网文的连续性不是靠 AI 记性，而是靠正典事实治理。AI agent 不能把候选设定、推断、读者猜测、竞品桥段、临时脑洞或模型补全写成正文事实；每个关键事实都要有来源、状态、影响范围和冲突处理方式。

最小链路：事实需求 -> 来源查证 -> 状态标记 -> 影响分级 -> 冲突检测 -> 决策记录 -> 正文使用 -> 台账回写。

---

## 1. 这个专题解决什么问题

已有第 07 篇负责连续性与伏笔台账，第 36 篇负责上下文打包，第 38 篇负责多 agent 交接，第 33 篇负责修订和版本管理。本篇专门解决更底层的问题：哪些信息能被当成“故事事实”。

AI 长篇创作常见风险：

- 把候选设定写进正文，后续 agent 当成已确认事实。
- 忘记角色名字、称呼、年龄、伤势、资源、关系状态和技能限制。
- 推断一个角色“应该知道”某事，但正文从未让他知道。
- 为了解决当前场景，临时编造一个旧道具、旧承诺、旧规则或旧人情。
- 台账和正文冲突时，直接相信台账，导致覆盖正文 canon。
- 读者评论猜测被下游 agent 当成真实伏笔。
- 修订时把已发布章节中的事实改掉，却不处理后文连锁影响。
- 用“应该是”“可能已经”“大概可以”这种推断推动关键情节。

本专题要求：任何影响剧情、人物、规则、资源、时间线、信息权限和读者信任的事实，都必须先治理，再写进正文。

## 4. 基础概念

### 4.1 正典事实 Canon

已经被用户确认、项目正式文档确认，或已经进入读者可见正文并被后文承接的事实。

正典事实包括：人物姓名、称呼、年龄、身份、关系、伤势、资源、技能、时间线、世界规则、地理、组织、已发生事件、谁知道什么、谁欠谁什么。

### 4.2 候选 Candidate

AI 提出的可用方案、用户暂时感兴趣但未确认的想法、备选大纲、测试版本、未定人设、未选书名、未定关系方向。

候选可以进入项目文档的候选区，但不能被正文当作既定事实。

### 4.3 待确认 Pending

没有答案就会影响后续写作的关键问题。比如主角真实年龄、男主是否知道女主身份、某个技能是否能连续使用、某段关系是否已公开。

待确认不能靠 AI 脑补。低风险时可以用临时占位；高风险时必须停下或改写为不依赖该事实的场景。

### 4.4 推断 Inference

根据已有事实做出的合理判断，但正文或项目文档没有直接确认。

推断可以用于分析和候选方案，不能直接作为关键解法来源。

### 4.5 幻觉 Hallucination

AI 生成了看似合理、语气肯定、但没有来源支撑的事实。小说创作中的幻觉尤其危险，因为它会伪装成“合理补全”。

### 4.6 冲突 Conflict

两个来源对同一事实给出不同答案。冲突不等于立刻选一个；必须判断来源优先级、发布时间、是否已发布、影响范围和用户最新决定。

## 5. 事实状态标签

所有关键事实应使用以下状态标签：

| 标签 | 含义 | 是否可写入正文 |
| --- | --- | --- |
| 已确认 | 用户确认、正式文档确认或读者可见正文已发生 | 可以 |
| 候选 | 可用方案，尚未被用户或正式文档确认 | 不可当事实写入正文 |
| 待确认 | 必须确认，否则影响后文 | 高影响时暂停或绕开 |
| 推断 | 基于已有事实的合理猜测 | 不可作为关键结果来源 |
| 冲突 | 来源之间不一致 | 先裁决，后使用 |
| 废弃 | 曾经候选或旧设定，已明确不采用 | 不可使用 |
| 变更中 | 正在修订，后文影响未处理完 | 暂停依赖该事实写关键剧情 |

## 6. 来源优先级

处理单本小说项目时，建议按以下优先级判断事实：

1. 用户当前明确指令。
2. 已发布或读者可见正文。
3. 项目正式文档中的已确认设定。
4. 故事台账中已确认事实、人物状态、规则状态。
5. 当前卷/当前章任务单里的已确认目标。
6. 风格手册和术语表中的已确认用法。
7. 用户历史聊天中明确确认但尚未整理的决定。
8. 候选大纲、临时脑洞、读者猜测、AI 推断。

关键规则：正文 canon 通常高于台账；台账是正文事实的索引，不是正文的替代品。若台账与正文冲突，先标记冲突，不得静默覆盖正文。

## 7. 高风险事实类型

以下事实不能靠模型补全：

### 7.1 人物身份和关系

姓名、称呼、性别、年龄、亲属关系、师承、婚恋状态、是否公开、谁知道谁的秘密。

风险：一处写错会影响代入、关系线和后文冲突。

### 7.2 时间线

事件发生顺序、持续时间、倒计时、生日、伤势恢复、修炼周期、任务期限、案发时间。

风险：时间线错会导致悬疑、成长和代价全部失真。

### 7.3 资源和能力

钱、灵石、股份、职位、权限、技能冷却、系统点数、道具数量、欠债、奖励。

风险：资源凭空多出来等同作者作弊。

### 7.4 规则和限制

金手指规则、世界法则、组织制度、职业流程、比赛规则、法律/平台/合规规则。

风险：规则临时变化会破坏读者信任。

### 7.5 信息权限

谁知道真相、谁误会、谁见过证据、谁听到对话、谁能证明某事。

风险：信息权限错会让主角降智、反转失去公平性。

### 7.6 读者可见承诺

书名、简介、标签、章末钩子、旧伏笔、情绪债、关系承诺、付费前期待。

风险：承诺被遗忘会造成掉读和信任流失。

## 8. 写作前事实检查流程

每次起草关键章节前，agent 至少执行以下检查：

```text
本章依赖的关键事实：
每条事实来源：正文 / 项目文档 / 台账 / 用户当前指令 / 候选 / 推断
每条事实状态：已确认 / 候选 / 待确认 / 推断 / 冲突 / 废弃 / 变更中
是否影响关键结果：是 / 否
是否影响角色知识：是 / 否
是否影响规则或资源：是 / 否
是否需要用户确认：是 / 否
如果不能确认，是否可以改写为不依赖该事实：是 / 否
写后需要回写哪些状态变化：
```

如果关键结果依赖“候选/推断/待确认”事实，禁止直接写成确定剧情。

## 9. 不确定性处理规则

### 9.1 低影响不确定

例如路人姓名、临时店名、一次性场景细节。可以创建候选，并在写后记录。

### 9.2 中影响不确定

例如配角是否知道某个旧事件、某个道具是否还在主角手里。优先改写场景，让结果不依赖该事实；或写成角色主动查证。

### 9.3 高影响不确定

例如主角血统、核心规则、关系是否确认、关键证据来源、终局伏笔。必须暂停关键写法，标记待确认，或回到项目文档裁决。

### 9.4 已发布文本不确定

如果正文已发布，不能随意改旧事实。优先后文补偿、解释、转化或轻修；大改进入第 33 篇修订流程。

## 10. 设定冲突处理

冲突出现时按以下顺序处理：

1. 明确冲突项：哪两个来源冲突？
2. 标注来源类型和时间：正文、文档、台账、聊天、候选。
3. 判断是否已发布：已发布正文优先保护读者认知。
4. 判断影响范围：只影响一句话、一个场景、一条支线，还是全书规则。
5. 判断是否可解释为角色误解、信息差、谣言、版本差异或世界内规则例外。
6. 如果不能自然解释，进入决策记录。
7. 修订后更新台账、风格手册、设定文档和相关章任务单。

禁止：发现冲突后，直接选择“更顺手”的版本写下去。

## 11. Retcon 与补丁边界

Retcon 是对已建立事实的后设调整。商业连载可以使用，但必须谨慎。

### 11.1 可接受的 retcon

- 补足未明说的空白。
- 将角色误解、谣言或不完整信息重释。
- 解释边缘细节，不改变核心承诺。
- 通过后文付代价地改变理解。

### 11.2 高危 retcon

- 推翻已发布主线真相。
- 改核心规则而不付代价。
- 把读者已见证的选择说成没发生。
- 为了救当前剧情，临时新增关键资源。
- 把已确认关系改成“其实没有”。

### 11.3 Retcon 必须记录

任何 retcon 都应记录：旧事实、新事实、修改理由、影响章节、读者风险、补偿方式、是否需要作者话或后文解释。

## 12. AI 幻觉的常见形态

| 幻觉形态 | 表现 | 修复方式 |
| --- | --- | --- |
| 名字漂移 | 角色改名、称呼变了 | 查人物表和风格手册，统一称呼 |
| 资源幻觉 | 突然多出钱、道具、权限 | 查资源台账，补来源或改解法 |
| 关系幻觉 | 把未确认关系当公开事实 | 查关系状态和谁知道什么 |
| 时间幻觉 | 伤势恢复太快、期限错 | 查时间线和状态差额 |
| 规则幻觉 | 新规则临时救场 | 回到规则文档，补限制或改结果 |
| 旧情节幻觉 | 说前文发生过但其实没有 | 删除或改为当前场景建立 |
| 信息权限幻觉 | 角色知道不该知道的真相 | 改为发现过程、误判或信息差 |
| 平台数据幻觉 | 编造点击、追读、首订、规则 | 标记数据缺失，要求真实来源 |
| 外部事实幻觉 | 编造法律、医学、行业规则 | 联网查证或标待确认 |
| 读者反馈幻觉 | 编造评论趋势 | 只使用真实反馈样本 |

## 14. 正文使用规则

### 14.1 已确认事实

可以直接使用，但要遵守当前视角和信息权限。

### 14.2 候选设定

只能作为方案写在项目文档，或在正文中以不确定、传言、误会、角色猜测形式出现，不能变成旁白确定事实。

### 14.3 待确认事项

不要用旁白定论。可以写成角色查证、暂避、选择保守方案，或改写场景目标。

### 14.4 推断

可以用于设计阻力和备选，但不能作为关键反杀、揭谜、关系确认、规则生效的唯一依据。

### 14.5 冲突事实

必须裁决后再写。若为了剧情先写，必须选择不依赖冲突事实的中性表达。

---

## 15. 外部资料和现实规则

涉及现实平台、法律、医学、行业、历史、地理、职业流程、合规、版权、AI 标识、合同和数据时，默认具有时效性或高风险。

规则：

- 不编造平台规则、审核结果、授权证明、法律结论或后台数据。
- 不把旧公告当最新规则。
- 不把单个社区经验当通用事实。
- 不把 AI 生成资料当来源。
- 正式发布、签约、合规、法律、医学、金融等高风险用途必须重新查证。
- 无法查证时写“待确认”，不要用肯定语气。

## 16. 审稿检查清单

审稿时逐条检查：

- 本章关键结果是否依赖未确认事实？
- 是否有候选设定被写成旁白事实？
- 是否有角色知道了他不该知道的信息？
- 是否有旧道具、旧资源、旧规则突然出现？
- 是否有台账与正文冲突？
- 是否有已发布事实被静默改写？
- 是否有作者为了爽点临时补规则？
- 是否有读者猜测、评论、竞品桥段被写进本书 canon？
- 是否有外部平台、法律、行业、医学、数据被编造？
- 写后是否回写了人物状态、资源、规则、关系和伏笔变化？

通过标准：每个关键事实都有来源和状态，每个冲突都有处理记录，每个新增事实都有回写位置。

## 18. 最小可执行规则

AI agent 至少遵守以下规则：

- 候选不是事实，推断不是事实，读者猜测不是事实。
- 关键结果不能依赖未确认事实。
- 正文 canon 通常高于台账；台账冲突时先标记，不静默覆盖。
- 角色知识状态必须单独管理，不能因为读者知道就让角色知道。
- 资源、规则、时间线、关系状态和信息权限是高风险事实，不能靠模型补全。
- 外部平台、法律、医学、数据和合规规则必须查证；缺证据标待确认。
- 多 agent 交接必须分清已确认、候选、待确认、冲突、废弃和推断。
- 任何事实变更都要记录影响范围和回写位置。
