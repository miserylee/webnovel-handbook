# 163. 数字账号、密码隐私、网络安全、黑客桥段与电子数据可信度专项

适用范围：当 AI agent 需要设计、起草、审稿或修订现代都市、娱乐圈、公关舆论、商战、校园、职场、悬疑刑侦、记者线、律师线、赛博朋克、黑客角色、平台运营、账号盗用、数据泄露、隐私暴露、聊天记录、转账记录、后台日志、系统通知、AI/算法平台、电子证据、网络诈骗、勒索软件、社工钓鱼、数字身份和任何“网络行为改变剧情后果”的场景时，使用本文件。

核心目标：让数字情节不再是“黑客敲两下键盘就进入所有系统”“截图一出真相大白”“账号被盗只是临时误会”“隐私泄露无后果”，而是形成可追踪的叙事链：

`现实身份 -> 数字账号 -> 登录凭据 -> 设备会话 -> 权限范围 -> 数据资产 -> 平台规则 -> 日志痕迹 -> 风险扩散 -> 证据边界 -> 现实余波`

一句话规则：数字世界不是魔法后台，而是由身份绑定、凭据、设备、权限、日志、平台、法律边界和人性弱点共同组成的行动空间。

---

## 1. 本专题解决什么问题

AI 写数字情节常见坏味道：

1. 黑客万能：任何系统都能秒进，任何数据都能秒查，任何监控都能秒调。
2. 密码神话：知道生日、名字或一句台词就能破解所有账号。
3. 设备消失：账号、手机、电脑、云盘、SIM 卡、浏览器会话和登录设备之间没有区别。
4. 权限消失：实习生能看 CEO 邮箱，普通用户能调后台日志，警察能随口拿平台数据。
5. 截图万能：截图、录屏、聊天记录、转账记录一出现就自动等于真相。
6. 隐私无代价：开盒、人肉、泄露地址、曝光病历、公布聊天记录后，只服务爽点，不处理伤害。
7. 诈骗太弱：钓鱼、冒充客服、验证码、链接、社工只写成低级骗局，人物一降智就被骗。
8. 数据泄露无链条：泄露从哪来、谁能访问、怎么扩散、谁承担责任、谁修复，都没有。
9. 平台像空气：封号、实名、审核、申诉、风控、客服、日志、备份和合规部门都不存在。
10. 攻击教程化：为了写真实，把漏洞利用、绕过验证、窃取账号、恶意脚本和规避追踪写成可模仿步骤。

本专题把数字情节拆成“可信度、叙事功能、安全边界、证据后果”四层，让 AI agent 能写得像现实，却不变成现实攻击指南。

## 3. 安全边界：只写小说可信度，不写攻击教程

涉及网络安全、账号、隐私、黑客、漏洞、恶意软件、诈骗、数据泄露和电子取证时，AI agent 必须遵守以下边界：

1. 不输出真实漏洞利用步骤、代码、工具命令、配置流程、绕过方法或攻击路径。
2. 不写如何盗号、撞库、绕过多因素认证、规避风控、逃避追踪、清除日志或洗钱。
3. 不提供可复制的钓鱼话术、恶意链接制作、木马投放、勒索软件部署或数据窃取教程。
4. 不把开盒、人肉、隐私曝光、网络暴力和泄露未成年人信息写成无害爽点。
5. 写黑客角色时，用目标、权限、限制、风险、日志、后果和道德边界塑造人物，不用技术细节炫耀。
6. 写防御和恢复时，可以写概念性措施：冻结账号、改密、启用 MFA、通知平台、报警、保留证据、恢复备份、内部调查、法务介入。
7. 写现实法律、平台规则、个人信息、数据安全和电子证据时，必须查最新权威来源，并避免给出法律建议。

可写：概括性、叙事性的风险链和后果。

不可写：现实中可直接照做的攻击链和规避链。

## 4. 数字安全十线卡

设计数字桥段时，至少填写以下十线中的关键项。

| 线 | 要问什么 | 常见用途 | 常见坏味道 |
| --- | --- | --- | --- |
| 现实身份线 | 账号背后是谁，如何绑定现实身份 | 掉马、替身、冒名、实名制、平台封禁 | 账号等于本人，没有冒用和误认可能 |
| 凭据线 | 登录靠密码、验证码、设备、密钥、生物识别还是员工权限 | 盗号、恢复、风控、误登录 | 猜生日即可破解一切 |
| 设备线 | 数据在手机、电脑、云端、U 盘、服务器还是旧设备 | 丢手机、旧机取证、云同步、备份 | 账号和设备混为一谈 |
| 权限线 | 谁能看、谁能改、谁能导出、谁能删除 | 后台、公司、学校、医院、平台 | 普通人随便查后台 |
| 日志线 | 操作留下什么痕迹，谁能看到 | 登录地点、时间线、异常访问、内鬼 | 行动无痕，证据空降 |
| 数据线 | 数据是聊天、图片、转账、定位、订单、病历还是日志 | 调查、舆论、勒索、澄清 | 所有数据都同等可信 |
| 平台线 | 平台如何审核、封禁、申诉、协助调查、通知用户 | 热搜、直播事故、账号恢复 | 平台只在作者需要时出现 |
| 扩散线 | 数据如何被复制、转发、下载、截屏、二传、被删后留存 | 舆论爆炸、隐私伤害、谣言 | 删除等于消失，曝光无二次伤害 |
| 证据线 | 电子数据如何证明真实性、完整性、关联性 | 刑侦、律师、反杀、翻案 | 截图一锤定音 |
| 余波线 | 事件改变了钱、名声、关系、工作、法律风险还是心理安全 | 连载后果、卷末结算 | 修完账号就没事了 |

最小规则：任何数字情节至少要回答“谁有权限、留下什么痕迹、会伤害谁、能不能证明、后续谁买单”。

## 5. 账号不是人本身：现实身份、数字身份和冒用空间

### 5.1 三层身份

数字叙事中必须区分三层身份：

| 层级 | 内容 | 叙事用途 |
| --- | --- | --- |
| 现实身份 | 自然人、法人、组织、职业身份 | 法律责任、名誉、关系后果 |
| 平台身份 | 用户名、账号 ID、认证标识、粉丝数、等级 | 舆论影响、平台权限、商业价值 |
| 操作身份 | 谁实际拿着设备或凭据操作 | 冒名、内鬼、误会、取证争议 |

一个账号发出的内容，不一定等于账号主人本人操作。

但不能因此让所有证据失效。故事应让角色围绕登录时间、设备、地点、操作习惯、异常行为、见证人、平台记录和动机进行综合判断。

### 5.2 账号价值

账号不是空壳，常包含：

- 社交关系：好友、粉丝、群组、联系人。
- 经济价值：店铺、支付、广告、打赏、订单、游戏资产。
- 身份价值：实名认证、职业认证、蓝 V、学生账号、员工账号。
- 情绪价值：旧聊天、照片、纪念、创作存稿。
- 风险价值：隐私、把柄、合约、商业秘密、未公开作品。

写盗号、封号或账号恢复时，要写它具体损失什么，而不是只写“账号没了”。

### 5.3 冒用与误认

常见可用情节：

| 情节 | 正确写法 | 风险 |
| --- | --- | --- |
| 账号被盗后发言 | 先写异常登录、发言风格变化、联系人反应、平台处置 | 不要写盗号步骤 |
| 朋友代发消息 | 写设备共享、信任边界、口吻破绽、后续解释成本 | 不要让误会靠蠢维持 |
| 公司账号多人共用 | 写岗位交接、权限混乱、责任归属、审计困难 | 不要让“共用”变成万能漏洞 |
| 粉丝冒充本人 | 写认证差异、头像昵称仿冒、传播速度、澄清滞后 | 不要复制现实诈骗话术 |
| 旧手机号被回收 | 写绑定遗留、验证码风险、客服流程、证据补充 | 不要提供利用方法 |

检查问题：这个账号背后的现实人、平台身份和实际操作人是否被明确区分？

## 6. 凭据不是一个密码：认证因素与剧情压力

NIST、CISA、NCSC 等资料都强调，现代账号安全不应只靠一个密码。写作时不需要写技术细节，但需要知道凭据可以分层。

### 6.1 凭据类型

| 类型 | 小说可写成 | 剧情作用 |
| --- | --- | --- |
| 知识因素 | 密码、PIN、密保问题 | 易被遗忘、猜测、泄露、重复使用 |
| 持有因素 | 手机、硬件钥匙、验证码设备、工作卡 | 丢失、借用、被抢、失效、离职回收 |
| 生物/本地解锁 | 指纹、面容、设备本地解锁 | 强调设备在场、身体在场、胁迫风险 |
| 恢复渠道 | 邮箱、手机号、备用码、客服申诉 | 账号恢复、冒名、旧关系和证明材料 |
| 组织权限 | 员工账号、管理员权限、审批角色 | 内鬼、越权、交接、审计、权限回收 |

不要把“知道密码”写成控制全部世界。现代账号常有设备确认、验证码、多因素认证、风控拦截、异地登录提醒和申诉流程。

### 6.2 多因素认证的叙事意义

多因素认证不是一句“更安全”的背景，它能制造剧情：

- 角色知道密码，但没有手机，无法登录。
- 角色拿到旧设备，但设备需要本人解锁。
- 角色离职后密码还能用，但员工权限已被回收。
- 角色收到异地登录提醒，意识到有人动过账号。
- 攻击者能发消息，但不能转账或导出敏感数据。
- 平台要求二次验证，拖慢澄清或救援节奏。

检查问题：这次登录为什么成功或失败？凭据、设备和权限是否都成立？

## 7. 设备、会话与云端：数据在哪里

数字情节必须区分“账号里有”“设备上有”“云端有”“服务器有”“备份里有”。

### 7.1 五种数据位置

| 位置 | 可写内容 | 限制 |
| --- | --- | --- |
| 本地设备 | 照片、聊天缓存、下载文件、离线草稿 | 设备丢失、损坏、加密、解锁限制 |
| 云端账号 | 云盘、同步相册、邮箱、备份 | 登录权限、同步延迟、删除保留 |
| 平台服务器 | 用户注册、交易、日志、私信、内容审核记录 | 平台权限、法律流程、数据保留周期 |
| 公司/学校系统 | 邮箱、OA、考勤、门禁、代码库、客户数据 | 岗位权限、审计、合规、离职回收 |
| 第三方设备 | 朋友手机、网吧电脑、旧电脑、监控设备 | 可信度低、污染风险、隐私边界 |

### 7.2 常见剧情误区

| 误区 | 修复 |
| --- | --- |
| 手机砸了，所有证据消失 | 区分本地、云端、对方设备、平台服务器和转发副本 |
| 删除聊天等于没有发生 | 写删除范围、备份、截图、转发、平台记录、对方保存 |
| 换手机后完全失忆 | 写同步、账号恢复、旧设备、备份失败和缺失后果 |
| 找到 U 盘就找到全部真相 | 写文件来源、时间戳、篡改可能、谁能打开、为何保留 |
| 电脑高手能看见所有数据 | 写权限、加密、日志、网络隔离、合规风险 |

检查问题：关键数据到底在什么位置？角色凭什么能接触它？

## 8. 权限和后台：不是所有人都能看所有数据

数字系统最重要的可信度来自权限。

### 8.1 权限分层

| 层级 | 能做什么 | 剧情后果 |
| --- | --- | --- |
| 普通用户 | 查看自己的内容、修改资料、发帖、删除本人可见内容 | 受平台规则和账号安全限制 |
| 群/店铺/频道管理员 | 管成员、删帖、看部分运营数据、处理订单 | 权限有限，但足以制造冲突 |
| 员工账号 | 处理客服、订单、审核、工单、内部资料 | 有岗位边界和审计记录 |
| 高级管理员 | 配置权限、导出数据、处理安全事件 | 高风险，通常有审批和日志 |
| 执法/司法协助 | 通过法定程序调取、冻结、固定数据 | 有程序、时间和范围限制 |
| 非法入侵者 | 可能获得局部权限或泄露数据 | 必须写风险、痕迹和后果，不能教程化 |

### 8.2 权限改变就是剧情结果

每一次数字事件都可以改变权限：

- 主角从普通用户变成店铺管理员。
- 反派被踢出群管理，失去删帖权。
- 离职员工账号被回收，但旧设备还没交。
- 公司给临时工开了过高权限，埋下事故。
- 平台封禁账号，角色失去发声渠道。
- 安全事件后，所有导出数据需要审批，拖慢下一步调查。

检查问题：这场戏之后，谁的数字权限增加、减少、冻结或被监控？

## 9. 日志与痕迹：数字行为会留下影子，但影子不等于真相

### 9.1 可作为线索的数字痕迹

| 痕迹 | 可证明什么 | 不能单独证明什么 |
| --- | --- | --- |
| 登录时间 | 某账号在某时间有登录行为 | 操作者一定是账号本人 |
| 登录地点/IP | 大致网络来源或异常变化 | 精确现实位置和真实身份 |
| 设备信息 | 常用设备、浏览器、系统差异 | 设备持有人一定操作了账号 |
| 操作日志 | 谁点击、导出、删除、修改了什么 | 操作动机和完整上下文 |
| 聊天记录 | 某些内容曾被发送或接收 | 内容一定真实、完整、无剪辑 |
| 转账记录 | 金钱流动、时间、账户 | 背后法律性质和真实意图 |
| 定位/轨迹 | 设备或账号曾出现某范围 | 人本人一定在场 |
| 截图录屏 | 当时界面的可见内容 | 原始数据未被篡改 |
| 文件元数据 | 创建、修改、格式、来源线索 | 文件内容一定真实 |

### 9.2 日志的叙事用法

日志不该只在结尾空降。它可以承担过程戏：

1. 发现异常：异地登录、异常导出、半夜操作、登录失败次数暴增。
2. 缩小范围：只有三个人有权限，只有一台设备在场。
3. 制造误导：账号是主角的，但设备在别人手里。
4. 暴露内鬼：权限使用习惯不符合岗位。
5. 保护无辜者：记录证明主角当时没有操作权限。
6. 制造代价：为了拿日志，需要走法务、审批、平台协助或公开风险。

检查问题：电子痕迹是怎样被发现、验证、质疑和转化为行动的？

## 10. 电子证据：截图不是一锤定音

中国“两高一部”关于刑事电子数据的规定强调，电子数据需要围绕真实性、合法性、关联性进行审查；NIST 数字取证资料也强调取证活动要关注数据来源、完整性、保存和分析过程。写小说时不需要写专业流程，但要保留三件事：来源、完整性、关联性。

### 10.1 三问

| 问题 | 小说表达 |
| --- | --- |
| 来源是否清楚 | 这条数据从哪里来，谁拿到，原始载体或平台记录是否存在 |
| 是否完整可信 | 有没有截断、剪辑、伪造、拼接、篡改、上下文缺失 |
| 是否与案件相关 | 它证明的是事实、时间、身份、动机、关系，还是只证明曾出现过某段内容 |

### 10.2 电子证据的正确戏剧流程

推荐链条：

`出现数字线索 -> 初步怀疑 -> 固定或保存 -> 补充来源 -> 质疑真伪 -> 找到交叉证据 -> 形成行动 -> 承担程序或舆论后果`

错误链条：

`截图出现 -> 所有人相信 -> 反派认输 -> 主角胜利`

### 10.3 交叉验证

电子数据最好与其他证据互相支撑：

- 聊天记录 + 转账记录 + 见面时间。
- 登录日志 + 门禁记录 + 监控范围。
- 订单信息 + 快递轨迹 + 客服工单。
- 文件修改时间 + 邮件发送记录 + 会议纪要。
- 爆料截图 + 平台原始链接 + 当事人口供或证人证言。

检查问题：如果对手质疑截图伪造，主角还能拿出什么非同源证据？

## 11. 隐私和个人信息：不是剧情免费的燃料

个人信息包括能够识别或关联到自然人的各种信息。小说可以写隐私危机，但必须处理伤害和边界。

### 11.1 高风险信息

| 类型 | 示例 | 风险 |
| --- | --- | --- |
| 身份信息 | 姓名、身份证件、学籍、工作单位 | 冒用、骚扰、资格损害 |
| 联系方式 | 手机号、邮箱、住址、社交账号 | 骚扰、跟踪、开盒 |
| 财务信息 | 银行、支付、工资、债务 | 诈骗、勒索、关系羞辱 |
| 健康信息 | 病历、诊断、用药、残障信息 | 歧视、羞辱、二次伤害 |
| 未成年人信息 | 学校、班级、照片、监护关系 | 安全风险、伦理风险 |
| 位置轨迹 | 家、公司、常去地点、实时定位 | 跟踪、暴力风险 |
| 私密内容 | 私聊、照片、视频、日记 | 名誉伤害、关系崩塌 |
| 生物识别 | 人脸、指纹、声纹 | 高敏感度、难以更换 |

### 11.2 隐私泄露后的后果

不要让隐私泄露只服务爽点。至少选择 1-3 项后果落地：

- 现实安全：搬家、换号、报警、学校/公司安保介入。
- 情绪安全：惊恐、羞耻、愤怒、失眠、不敢出门。
- 关系后果：亲友误解、信任受损、边界重建。
- 职业后果：客户流失、停职调查、合同违约、岗位风险。
- 法律后果：维权、取证、诉讼、平台投诉。
- 舆论后果：二次传播、谣言变形、道歉无效、澄清滞后。
- 长篇台账：换账号、改住址、删除旧资料、限制公开活动。

检查问题：这个隐私泄露伤害了谁？谁从中获利？谁承担修复成本？

## 12. 数据泄露与平台事故：不是“资料被卖了”一句话

### 12.1 泄露链条

写数据泄露时，用下面链条：

`数据资产 -> 保存位置 -> 访问权限 -> 薄弱环节 -> 发现方式 -> 影响范围 -> 通知/封堵 -> 用户损害 -> 调查问责 -> 长期修复`

### 12.2 常见泄露源的小说化表达

| 来源 | 可写成 | 注意 |
| --- | --- | --- |
| 员工越权 | 客服导出名单、离职员工保留文件 | 写岗位权限、审计和责任 |
| 第三方合作 | 外包系统、营销工具、插件、供应商 | 写合同、接口、数据流向 |
| 配置错误 | 本该内部可见的数据被公开 | 不写具体配置方法 |
| 弱认证 | 旧账号、共享密码、未开 MFA | 写制度疏漏和人性便利 |
| 钓鱼社工 | 员工被冒充同事/客户诱导 | 不输出可复用话术 |
| 恶意软件/勒索 | 文件被加密、业务停摆、备份失效 | 不写部署方法，写恢复和代价 |
| 平台漏洞 | 用户数据异常暴露 | 只写宏观事故、补丁、通知、信任损失 |

### 12.3 泄露后的可见后果

- 用户收到诈骗电话。
- 主角被迫通知客户或粉丝。
- 公司停业整顿、平台下架功能、项目延期。
- 内部互相甩锅，安全团队和法务介入。
- 反派拿泄露数据做二次勒索。
- 主角失去合作方信任，必须公开透明补救。
- 受害者不接受“已经处理”的空话。

检查问题：数据泄露的影响范围、通知对象、修复动作和信任损失是否可见？

## 13. 钓鱼、诈骗和社工：人物不是降智，而是被压力和信任击中

网络诈骗和社工情节不应靠人物突然变蠢。更可信的写法是：角色在时间压力、身份权威、关系信任、恐惧、贪念、羞耻、疲劳或信息不对称下做出错误判断。

### 13.1 可信诱因

| 诱因 | 场景 |
| --- | --- |
| 时间压力 | 限时处理账号异常、合同快过期、孩子出事 |
| 权威压力 | 冒充平台客服、领导、学校、银行、执法人员 |
| 关系信任 | 好友账号被盗后借钱、同事发内部链接 |
| 恐惧羞耻 | 威胁曝光隐私、假称违法、制造羞辱感 |
| 利益诱惑 | 内部名额、退款、中奖、兼职、投资回报 |
| 疲劳状态 | 加班、失眠、赶稿、带娃、考试周 |
| 技术不熟 | 老人、未成年人、新员工、跨语言角色 |

### 13.2 安全写法

可写：角色看到异常、被逼判断、出现破绽、事后复盘、冻结账户、报警、平台申诉、关系修复。

不可写：完整诈骗话术、仿冒页面制作、验证码套取流程、绕过风控细节、洗钱路径。

### 13.3 诈骗后的关系后果

诈骗损失不只是钱，还会改变关系：

- 家人责怪受害者，制造二次伤害。
- 受害者羞于开口，错过黄金止损时间。
- 主角用共情和流程帮忙，建立信任。
- 反派利用受害者怕丢脸继续控制。
- 警方、平台、银行、学校或公司介入。

检查问题：受害者被骗是否来自可理解的压力，而不是作者强行降智？

## 14. 黑客角色：能力有边界，风险有回声

黑客角色可以很好看，但必须从“万能技术神”改成“有目标、有方法论、有资源限制、有法律和伦理压力的人”。

### 14.1 写黑客角色的六个维度

| 维度 | 要写什么 |
| --- | --- |
| 目标 | 是救人、调查、勒索、防御、揭露、修复，还是证明自己 |
| 专长 | 逆向、取证、防御、情报搜集、系统维护、脚本工具、社工识别等，概括即可 |
| 限制 | 时间、权限、设备、网络隔离、加密、日志、法务、道德底线 |
| 风险 | 留痕、违法、误伤、证据污染、反制、身份暴露 |
| 关系 | 与团队、雇主、受害者、平台、执法者的信任和冲突 |
| 代价 | 失去资格、被调查、被封号、背锅、心理负担、牺牲名声 |

### 14.2 不同类型的黑客角色

| 类型 | 可写爽点 | 边界 |
| --- | --- | --- |
| 防御型安全工程师 | 发现异常、止血、恢复服务、保护用户 | 不把防御写成非法攻击 |
| 取证型专家 | 还原时间线、发现篡改、保全证据 | 不写违法取证教程 |
| 白帽/研究员 | 负责任披露、补丁、行业声誉 | 不写漏洞利用细节 |
| 灰色情报人 | 知道地下消息、换取情报 | 写道德和法律风险 |
| 黑产反派 | 造成损失、胁迫、勒索 | 不美化，不教程化 |
| 普通高手 | 能看懂系统，不代表能秒入所有系统 | 控制能力范围 |

检查问题：这个黑客的能力是否有明确边界？每次越界是否留下风险和后果？

## 15. 数字调查：从线索到行动，而不是从搜索到答案

数字调查适合悬疑、娱乐圈、商战和职场线，但要避免“搜一下就全知道”。

### 15.1 数字调查五步

1. 问题定义：要证明什么？身份、时间、地点、动机、资金还是关系？
2. 数据来源：公开资料、当事人提供、平台记录、公司系统、电子设备、司法调取。
3. 权限边界：谁能合法获得？需要授权、审批、报警还是律师介入？
4. 交叉验证：至少用一个非同源证据验证关键结论。
5. 行动转化：证据用于澄清、谈判、报警、起诉、内部处分还是舆论反击。

### 15.2 公开资料和 OSINT 的小说边界

公开资料可以用于调查，但不能把现实人肉流程写成教程。小说中可以写：

- 主角发现公开时间线矛盾。
- 粉丝考古发现旧声明和现状冲突。
- 记者核对公开工商、裁判文书、招聘信息和采访记录。
- 律师团队整理公开舆论证据。
- 角色意识到自己过度公开生活轨迹，开始收缩隐私。

不要写：如何精确开盒真实住址、如何绕过隐私设置、如何批量爬取个人数据、如何骚扰目标。

检查问题：调查获得的信息是公开、授权、合法协助、内部权限，还是需要明确标记为违法风险？

## 16. 平台规则：封号、审核、实名、申诉和风控都是剧情资源

平台不是空气。现代数字桥段里，平台规则可以制造阻力和回报。

### 16.1 平台动作

| 动作 | 剧情作用 |
| --- | --- |
| 审核 | 让澄清延迟，让反派利用时间差 |
| 限流 | 真实内容发不出去，谣言先扩散 |
| 封号 | 角色失去发声渠道，也可能保护受害者 |
| 实名认证 | 提升可信度，也暴露隐私风险 |
| 账号申诉 | 需要身份证明、旧设备、绑定信息、等待时间 |
| 内容删除 | 降低扩散，也可能被质疑灭证 |
| 后台日志 | 帮助调查，但需要权限和程序 |
| 风控拦截 | 阻止异常交易，也可能误伤正常用户 |
| 平台公告 | 影响公众信任和舆论方向 |

### 16.2 平台的利益

平台也有目标：

- 控制舆论风险。
- 降低法律和监管风险。
- 保护用户安全。
- 保护商业合作。
- 避免承认系统漏洞。
- 在速度、准确和公关之间取舍。

检查问题：平台为什么这样处理？它的动作对主角、反派、受害者和公众分别有什么后果？

---

## 17. 职场数字安全：账号开通、权限回收和内部泄密

现代职场情节中，数字权限是岗位的一部分。

### 17.1 员工生命周期里的账号

| 阶段 | 数字事项 | 剧情风险 |
| --- | --- | --- |
| 入职 | 开邮箱、OA、门禁、代码库、客户系统 | 权限过大、培训不足 |
| 试用期 | 临时权限、导师审批、访问记录 | 背锅、误操作 |
| 转岗 | 权限变更、旧资料交接 | 旧权限未回收 |
| 晋升 | 管理后台、审批权、导出权 | 权力诱惑和审计风险 |
| 离职 | 账号停用、设备归还、数据交接 | 带走资料、删除证据、客户流失 |
| 外包/实习 | 最小权限、合同边界、系统隔离 | 责任归属模糊 |

### 17.2 内部泄密更有张力

比起万能黑客，内部泄密更适合职场、商战、娱乐圈：

- 有权限的人误发文件。
- 员工把客户名单拷给竞争对手。
- 外包团队保留生产数据。
- 离职员工还能登录旧系统。
- 高管让下属越权导出数据。
- 安全部门发现异常下载，但业务部门压着不报。

检查问题：这次泄密是外部攻击、内部越权、供应商事故、制度漏洞，还是几者叠加？

## 18. 校园和未成年人数字风险

涉及未成年人时必须提高安全阈值。

高风险场景：

- 班级群、校园论坛、匿名墙、表白墙。
- 学生照片、成绩、住址、家庭情况被曝光。
- 手机被同学拿走发消息。
- 账号被盗后发布侮辱内容。
- 老师、家长、学校要求查看手机。
- 网络欺凌从线上扩散到线下。
- 未成年人参与危险挑战、诈骗、隐私交换。

写法要求：

1. 保护未成年人隐私，不细写可识别信息。
2. 不把偷拍、传播私密内容、羞辱、网暴写成玩笑。
3. 让监护、学校、平台和必要的专业支持进入后果链。
4. 不让受害者承担全部责任。
5. 将数字伤害与线下安全、学业、家庭关系和心理压力连接。

检查问题：这段是否把未成年人隐私或伤害当成爽点、梗或恋爱助攻？

## 19. 赛博、科幻和系统文的迁移

架空世界可以有更强技术，但仍要有规则。

### 19.1 赛博世界的数字可信度

即使是赛博朋克，也要回答：

- 身份凭据是什么：芯片、虹膜、声纹、脑机接口、区块链身份、公司工牌。
- 权限如何授予：公司、政府、帮派、AI、平台、黑市。
- 数据如何储存：本地义体、云端、城市网格、公司服务器、离线黑匣子。
- 失效如何发生：断网、断电、权限冻结、身体损伤、系统升级、供应商停服。
- 证据如何可信：签名、日志、第三方见证、物理载体、多人记忆。
- 反抗代价是什么：被定位、被封禁、被追捕、失去医疗和交通权限。

### 19.2 系统文和无限流

系统日志、任务面板、积分记录和副本公告也属于数字/规则数据。

不能随便改，要有：

- 谁能看见。
- 是否可截图或转述。
- 是否可篡改。
- 是否会延迟。
- 是否能作为证据。
- 系统是否有审计或惩罚。
- 主角利用漏洞是否有代价。

检查问题：这个系统数据是规则事实、角色感知、幻觉诱导，还是反派伪造？

## 20. 类型文用法

### 20.1 都市娱乐圈

可用场景：

- 明星账号被盗后发出争议内容。
- 私生饭曝光住址和行程。
- 代拍、站姐、营销号、平台审核和公关团队抢时间。
- 旧聊天记录被断章取义。
- 经纪公司内部员工泄露未公开物料。

关键：写平台、粉圈、公司、公关、法务、当事人情绪和现实安全。

### 20.2 商战职场

可用场景：

- 客户数据泄露引发信任危机。
- 离职员工权限未回收，造成项目事故。
- 竞对拿到报价表，但来源不明。
- 安全部门和业务部门因是否公开事故冲突。
- 主角通过合规处置赢回合作方。

关键：写权限、流程、审计、供应商、通知和责任。

### 20.3 悬疑刑侦

可用场景：

- 登录日志与不在场证明冲突。
- 聊天记录被剪辑，原始设备失踪。
- 转账记录证明资金流，但动机另有隐情。
- 嫌疑人故意制造账号冒用假象。
- 警方依法固定电子数据，律师质疑关联性。

关键：电子证据必须进入真实性、完整性、关联性检查。

### 20.4 校园青春

可用场景：

- 匿名墙爆料引发班级撕裂。
- 学生账号被盗，名誉受损。
- 成绩截图被传播，家庭关系爆炸。
- 主角帮同学保存证据并寻求老师/家长/平台帮助。

关键：未成年人、隐私和网络欺凌必须联动安全边界。

### 20.5 赛博科幻

可用场景：

- 身份芯片被冻结，角色无法乘车、支付和就医。
- 公司删除员工数字履历，主角变成“系统里不存在的人”。
- AI 审核错误导致群体被限权。
- 主角夺回的不是文件，而是行动资格。

关键：数字权限要连接身体、城市、资源和生存。

## 21. 常见坏味道与修复

| 坏味道 | 读者体感 | 修复方式 |
| --- | --- | --- |
| 黑客秒进所有系统 | 太假，像魔法 | 限定目标系统、权限来源、时间成本、风险和后果 |
| 密码一猜就中 | 降智 | 增加多因素、设备、风控、恢复流程和误判空间 |
| 截图一锤定音 | 悬疑廉价 | 加原始来源、完整性争议、交叉证据和质证过程 |
| 平台无反应 | 现代感缺失 | 写审核、封禁、申诉、通知、日志、合规和公关 |
| 隐私泄露无伤害 | 冒犯、轻飘 | 写现实安全、情绪安全、职业损失和关系修复 |
| 数据泄露只当爆点 | 没有后果 | 写影响范围、用户通知、止血、问责和长期信任损失 |
| 诈骗靠受害者蠢 | 不尊重受害者 | 写压力、权威、疲劳、关系信任和羞耻机制 |
| 黑客角色无边界 | 爽但崩 | 给专长、限制、法律风险和道德选择 |
| 电子证据空降 | 作者作弊 | 在前文埋日志、设备、备份、平台规则和可攻击点 |
| 数字事故不进台账 | 长篇失忆 | 回写账号状态、权限变化、泄露范围和公开认知 |

---

## 22. 审稿检查清单

审稿时逐项检查：

1. 是否区分现实身份、平台账号和实际操作者。
2. 关键登录是否解释凭据、设备、会话和权限。
3. 账号被盗、封禁、恢复或冒用是否有具体损失和后果。
4. 角色是否能合法或合理接触关键数据。
5. 管理员、员工、平台、执法者的权限是否分层。
6. 电子痕迹是否有来源、时间、设备、日志或平台记录。
7. 截图、聊天记录、转账记录是否经过真伪和完整性检查。
8. 是否有非同源证据交叉验证关键结论。
9. 隐私泄露是否写出受害者安全、情绪、关系和现实成本。
10. 数据泄露是否写出影响范围、通知、止血、问责和修复。
11. 诈骗情节是否来自可理解压力，而不是人物突然降智。
12. 黑客角色是否有能力边界、风险和道德选择。
13. 是否避免输出攻击、盗号、钓鱼、恶意软件、绕过风控等可模仿步骤。
14. 是否处理平台审核、封禁、申诉、日志、客服和合规部门。
15. 职场数字权限是否与岗位、合同、离职和责任连接。
16. 涉及未成年人时是否提高隐私和伤害防护阈值。
17. 数字事件是否改变钱、名声、关系、权限、法律风险或下一步行动。
18. 是否把账号状态、权限变化、公开认知和证据边界写入台账。

## 25. 外部资料吸收点

本专题吸收外部资料时只保留小说可用的结构和风险边界：

- 中国《个人信息保护法》：个人信息处理有目的、范围、权益和义务边界；迁移为“隐私不是免费素材，泄露必须有伤害和责任”。
- 中国《数据安全法》《网络安全法》《网络数据安全管理条例》：数据安全、网络运营和网络数据处理有合规要求；迁移为“平台、公司和组织不能像无责任黑箱”。
- “两高一部”刑事电子数据规定：电子数据需要关注来源、真实性、完整性、合法性和关联性；迁移为“截图不是一锤定音，电子证据要能被质疑和交叉验证”。
- NIST Digital Identity Guidelines：认证涉及不同因素、身份生命周期和认证强度；迁移为“登录不是只靠一个密码”。
- CISA Secure Our World、StopRansomware：多因素认证、强密码、钓鱼识别、勒索软件和事件响应是普通组织与个人都会遇到的安全问题；迁移为“数字事故要有发现、止血、恢复和通知”。
- FTC identity theft/data breach 资料：身份盗用和数据泄露会造成现实经济和生活后果；迁移为“账号事故会延伸到钱、名声、工作和安全”。
- NCSC 2-step verification：重要账号的第二步验证会影响账号控制权；迁移为“缺设备、缺验证和账号恢复能自然制造剧情阻力”。
- NIST SP 800-86 数字取证资料：数字取证强调数据来源、保存、分析和事件响应；迁移为“电子痕迹必须有保存和解释过程”。
- OWASP Top 10：现实网络安全风险多来自访问控制、认证、配置、日志监控等系统性问题；迁移为“漏洞不是魔法单点，而是权限、设计和运维共同失效”。

## 26. 最小执行规则

如果 AI agent 只能记住五条：

1. 数字账号不等于现实本人，必须区分账号主人、实际操作者、设备和权限。
2. 登录不是只靠密码，关键情节要检查凭据、设备、MFA、恢复渠道和风控。
3. 截图、聊天记录、转账和日志只是线索，必须检查来源、完整性、关联性和交叉证据。
4. 隐私、盗号、数据泄露、诈骗和网暴必须有受害者后果，不得写成无害爽点。
5. 网络安全桥段只写叙事可信度、风险和后果，不写现实攻击、盗号、绕过验证或恶意软件教程。
