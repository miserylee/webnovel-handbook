## 1. 核心原则：评论是原始信号，不是修改命令

评论有价值，因为它记录了读者在某个阅读节点的即时反应。但评论也有噪音：情绪宣泄、玩梗、站队、非目标读者误入、平台风气、读者之间互相影响、少数极端意见、AI 生成式套话反馈，都可能混在一起。

AI agent 的任务不是“听评论改文”，而是完成四步转译：

1. 原文：读者到底说了什么。
2. 标签：这条评论属于哪类阅读体验。
3. 聚类：是否有多条评论指向同一问题。
4. 动作：这个问题是否需要改、改哪里、怎么观察。

执行原则：

- 先记录原文，再写解释，不能直接把评论改写成自己的判断。
- 一条评论可以拆成多个观察点，不能只贴一个总标签。
- 高频反馈优先，但低频高风险反馈不能忽略。
- 正反馈必须和负反馈同等沉淀，因为正反馈是作品资产。
- 读者提出的解决方案不一定采纳，但读者感受到的问题必须认真判断。
- 评论结论必须和章节文本、读者路径、后台数据、更新记录交叉验证。

风险：

- 如果只看单条评论，会被尖锐情绪牵着走。
- 如果只看评论数量，会忽略少数但致命的毒点。
- 如果只看负反馈，会误伤真正吸引核心读者的卖点。
- 如果把评论当作审稿报告，会让文本变成平均口味、失去类型锋芒。

## 2. 评论处理对象：什么需要进语料池

需要进入语料池：

- 平台章评、段评、书评、评分附文。
- 匿名读者试读反馈。
- 催更、弃文、跳章、补评、追问。
- 编辑反馈中带有读者体验判断的部分。
- 公测、盲评、A/B 开篇测试反馈。
- 读者明确复述的爽点、雷点、梗点、关系点。

不直接进入语料池，或只作背景记录：

- 纯表情、无内容刷屏。
- 明显广告、引战、辱骂、站外争执。
- 与作品无关的平台争论。
- 泄露个人隐私的信息。
- 无法定位作品或章节的二手转述。
- 明显模板化、无文本证据的 AI 式泛评。

处理要求：

- 评论原文只保留必要片段，不扩散读者个人信息。
- 如果评论涉及敏感身份、现实纠纷或平台违规，应先进入 `docs/20-content-compliance-risk-review.md`。
- 如果用户提供的是匿名读者反馈记录，默认只提炼观点，不保留可识别个人的昵称、头像、联系方式。

## 3. 标注单位：一条评论可以拆成多个观察点

读者评论常常混合多个问题。例如：

```text
女主这里突然圣母我有点接受不了，前面明明很清醒，而且这章铺垫太久了，最后还没打起来。
```

这不是一个标签，而是至少三个观察点：

- 人设连续性：女主行为和前文认知冲突。
- 节奏回报：本章铺垫过长。
- 章末承诺：读者期待的冲突没有发生。

标注时使用“观察点”作为最小单位：

- 一个观察点只对应一个主要体验。
- 每个观察点保留原文证据。
- 每个观察点必须标注章节或位置。
- 不确定位置时标为 `位置缺失：待确认`。

## 5. 标签体系：先统一语言，再讨论改法

### 5.1 阅读阶段标签

- `包装前置`：书名、封面、简介、标签、题材预期。
- `首章进入`：第一屏、主角出场、初始冲突、类型承诺。
- `前三章留存`：主角可追随性、第一轮爽点、第一关系钩。
- `前十章落袋`：阶段目标、短回报、第一轮承诺兑现。
- `当前剧情`：正在连载的副本、卷、支线或冲突。
- `中段疲劳`：重复桥段、支线膨胀、升级无差异。
- `付费节点`：付费前悬念、付费后兑现、订阅体验。
- `更新运营`：断更、爆更、更新时间、存稿质量、作者说明。
- `完结余波`：终局兑现、伏笔回收、番外、烂尾感。

### 5.2 问题层级标签

- `类型承诺错位`
- `主角能动性不足`
- `人物行为失真`
- `关系张力不足`
- `情绪回报不足`
- `爽点不落袋`
- `节奏慢或水`
- `信息过载或看不懂`
- `悬念不公平`
- `喜剧梗点失效`
- `世界观可信度不足`
- `章节可读性差`
- `文风或角色声音漂移`
- `AI味明显`
- `更新信任受损`
- `价值观或合规风险`

### 5.3 情绪倾向标签

- `喜欢`：读者明确表达继续看、爽、甜、笑、燃、带感。
- `困惑`：读者不知道发生了什么或为什么发生。
- `焦虑`：读者担心主线、关系或角色走向。
- `愤怒`：读者认为被欺骗、被恶心、被冒犯。
- `失望`：读者期待未兑现，但尚未完全退出。
- `退出`：读者明确说弃文、不追、跳章、取消收藏。
- `讨论`：读者围绕角色、谜团、CP、设定产生讨论。
- `催更`：读者明确表达下一章需求。

### 5.4 行为信号标签

- `催更`：说明章节末期待或关系钩有效。
- `复述爽点`：说明卖点可被记住。
- `复述梗点`：说明文本有传播语料。
- `站队争论`：可能说明关系张力有效，也可能说明价值观风险。
- `跳章`：说明中间段落或支线可能无增量。
- `弃文`：说明承诺长期未兑现或毒点过强。
- `二刷`：说明伏笔、情绪或梗点有回看价值。
- `求解释`：说明信息投放可能不足或读者追谜积极。

## 6. 证据强度：不是声音大就优先改

强证据：

- 多名目标读者在同一章节或同一问题上独立反馈。
- 评论反馈和数据掉点同向。
- 反馈能定位到具体章节、角色行动或阅读体验。
- 负反馈触及类型底线、合规风险或核心承诺。
- 正反馈反复复述同一个卖点、角色、梗或关系瞬间。

中证据：

- 反馈频率不高，但来自目标读者或付费读者。
- 反馈与文本诊断基本一致，但缺少数据支持。
- 评论集中在同类问题，但表达比较模糊。
- 反馈能指出问题感受，但不能定位具体位置。

弱证据：

- 单条情绪宣泄。
- 非目标读者的口味不匹配。
- 只给解决方案，不说明阅读问题。
- 与作品主承诺相冲突的个人偏好。
- 明显玩梗、跟风、引战或模板化反馈。

待确认：

- 反馈看起来严重，但缺少原文、章节、平台或时间窗口。
- 多条评论可能来自同一读者群体互相影响。
- 平台数据缺失，无法判断是否影响真实留存。

## 7. 聚类流程：从原文到主题

### 7.1 收集

先按时间、章节、平台收集评论。不要一开始就筛掉难听的评论，也不要只收集证明自己判断正确的评论。

### 7.2 清洗

清洗规则：

- 去掉重复刷屏和无内容表情。
- 删除个人隐私。
- 标出广告、引战、站外争执。
- 保留原文核心片段。
- 不把读者语言改写成作者喜欢听的语言。

### 7.3 拆分

把一条长评论拆成多个观察点。每个观察点只描述一个阅读体验。

### 7.4 标注

给每个观察点打阅读阶段、问题层级、情绪倾向、行为信号、证据强度。

### 7.5 归组

把相似观察点放在一起，形成初步主题。主题名称必须是问题句或发现句，不要只写“节奏”“人物”这种空标签。

好的主题名称：

- `第 12-14 章支线没有改变主线状态，目标读者开始说水。`
- `男主强势行为缺少补偿动作，甜宠读者开始感到油腻。`
- `读者喜欢女主反杀方式，说明机制限制带来的反制爽点有效。`

差的主题名称：

- `节奏问题`
- `男主问题`
- `有人喜欢有人不喜欢`

### 7.6 形成结论

每个聚类结论必须包含：

- 主题。
- 证据数量和来源。
- 代表性原文片段。
- 涉及章节。
- 可能病因。
- 是否与数据同向。
- 推荐动作。
- 观察窗口。

## 8. 评论到写作动作映射表

| 评论表达 | 可能指向 | 优先检查 | 常见动作 |
|---|---|---|---|
| “水”“没进展” | 章节状态增量不足 | 每章是否改变目标、资源、关系、信息、风险 | 压缩铺垫，增加本章可见进度 |
| “太慢” | 回报间隔过长 | 爽点、情绪、悬念、关系回报是否延迟 | 提前小回报，减少纯解释 |
| “看不懂” | 信息投放失败 | 关键目标、规则、动机是否场景化 | 补场景内提示，降低名词密度 |
| “崩人设” | 行动与已确认人设冲突 | 人物欲望、恐惧、底线、关系债 | 补动机或后果，必要时后文修复 |
| “主角没用” | 主角能动性不足 | 是否由系统、配角、巧合代替主角赢 | 增加主动选择和代价 |
| “不爽” | 承诺与回报不匹配 | 读者期待的赢法、落袋、打脸是否出现 | 调整回报类型或提前兑现 |
| “不甜”“没 CP 感” | 关系状态未变化 | 小事是否只对这两人特殊 | 联动情绪线专题补特殊性和关系进度 |
| “虐得累” | 情绪债无补偿 | 伤害后是否有理解、选择或回报 | 增加阶段补偿，不连续空虐 |
| “尬”“不好笑” | 喜剧机制失效 | 梗是否来自人物和情境 | 用人物癖好、错位和后果重做梗 |
| “像 AI” | 语言平滑、人物同声、解释过度 | 角色声音、叙述距离、具体摩擦 | 联动风格手册和降 AI 味指南 |
| “弃了” | 毒点或长期承诺失败 | 退出前 3-5 章发生了什么 | 做退场节点复盘，不立刻大改 |
| “催更” | 追读牵引有效 | 催更集中在哪个钩子后 | 保护有效钩子，做变体复制 |
| “这里好爽” | 有效卖点出现 | 爽点由什么触发 | 沉淀为可重复回报模型 |
| “这里笑死” | 有效梗点出现 | 谁的癖好、错位、反应制造笑点 | 保留人物化梗点，避免复制成口头禅 |

## 9. 冲突反馈处理：不是投票决定作品

读者反馈常常互相冲突：

- 有人嫌慢，有人觉得铺垫好。
- 有人觉得男主苏，有人觉得男主油。
- 有人喜欢女主狠，有人觉得女主冷血。
- 有人希望快点揭谜，有人喜欢慢慢猜。

处理顺序：

1. 先分读者类型：目标读者和非目标读者分开看。
2. 再分阶段：开篇读者、中段老读者、付费读者的反馈权重不同。
3. 再分承诺：反馈是否符合本书主承诺。
4. 再分后果：这个反馈如果不处理，会不会影响留存、付费、合规或核心资产。
5. 最后决定：保留锋芒、微调表达、后文补偿，还是进入重写流程。

结论格式：

```markdown
冲突点：
支持 A 的读者类型：
支持 B 的读者类型：
本书主承诺：
不能牺牲的资产：
可微调的表达：
本轮动作：
```

## 10. 正反馈资产库：喜欢什么比讨厌什么同样重要

每轮评论聚类必须输出正反馈资产。

正反馈资产类型：

- 角色资产：读者记住了谁，为什么记住。
- 关系资产：哪组互动让读者想追。
- 爽点资产：哪种赢法、反杀、落袋最有效。
- 情绪资产：哪种心疼、甜、燃、释然最有效。
- 梗点资产：哪句话、癖好、反差、场景可传播。
- 悬念资产：哪个问题让读者追问下一章。
- 风格资产：读者喜欢的旁白距离、语言质感、节奏。

使用方法：

- 修订时先列“不能误伤资产”。
- 续写时把有效资产做变体，不要机械复读。
- 包装时可以把高频正反馈转为简介、标签或章节名的承诺。
- 数据调参时，负反馈指向问题，正反馈决定哪些不能改平。

## 11. 评论反馈的红线

不能做：

- 不能伪造评论数量、评分、读完率或读者身份。
- 不能把单条恶评当成总体民意。
- 不能把读者解决方案直接当成改稿方案。
- 不能为了消除所有负反馈而牺牲类型承诺。
- 不能把非目标读者的不适直接等同于作品失败。
- 不能公开扩散读者群私人信息。
- 不能忽略合规、价值观、版权和现实伤害风险。
- 不能用“读者不懂”否定所有高频负反馈。

必须做：

- 标记证据强度。
- 保留代表性原文片段。
- 记录章节位置和时间窗口。
- 区分问题感受与改法建议。
- 同步正反馈资产。
- 给出观察窗口，而不是立刻宣布修改成功。

## 12. Agent 分工

评论收集 agent：

- 输入：评论、章评、书评、试读表、读者群记录。
- 输出：去隐私、去广告、保留必要原文的语料池。

标注 agent：

- 输入：语料池。
- 输出：观察点拆分、标签、证据强度。

聚类 agent：

- 输入：标注结果。
- 输出：主题聚类、代表性原文、章节位置、读者类型。

文本诊断 agent：

- 输入：聚类结论和章节文本。
- 输出：对应的创作问题和相关专题。

调参 agent：

- 输入：评论聚类、数据诊断、文本诊断。
- 输出：不改、微修、后文补偿、包装调整、重写或合规暂停建议。

复盘 agent：

- 输入：执行后的新评论和数据。
- 输出：有效、无效、副作用、继续观察或回滚。

## 15. 来源与迁移说明

本协议综合以下来源形成：

- 本知识库已有的审稿、读者反馈、数据诊断、连载调参、人味增强、修订流程和章节流水线规则。

- PLOS One 关于 Wattpad 数字社交阅读的研究：读者评论可以作为阅读反应和文本互动的信号，但需要结合文本位置和语境分析。
- Reedsy 关于 beta readers 的建议：反馈应服务读者体验和修订阶段，不能把早期反馈误用为校对或最终定稿意见。
- 定性研究与 UX 研究中的亲和图、主题聚类和偏差控制方法。

迁移到单本小说项目时，不要整段复制本专题。应只复制“评论标注字段”“聚类报告模板”“正反馈资产库”和已经确认的读者反馈结论，并把具体评论样本保存在项目自己的反馈记录或复盘文档中。


## 与 65 号专题的衔接：付费读者评论要单独标注

评论语料中如果出现“付费后水了”“这一章不值”“首订后质量掉”“上架前后节奏变了”“订阅亏了”“免费期更好看”等反馈，应联动 `docs/65-paid-conversion-subscription-retention.md`。

标注时需要区分评论者身份：目标读者、老读者、付费读者、非目标读者或未知。付费读者反馈不一定永远正确，但它直接关系读者信任，应优先判断问题发生在最后免费章、首个付费章、付费后 3-5 章、长期订阅价值还是更新运营。


## 与 66 号专题的衔接：评论整理之后才决定是否互动

处理评论、章评、书评、匿名读者反馈记录或评论区争议时，应联动 `docs/66-author-note-reader-community-interaction.md`，先判断评论属于可回复、需记录、需观察、需平台处理还是不宜回应。

评论聚类不是为了让作者逐条解释。对剧透猜测，不确认也不否认；对负面情绪，不急着对线；对明确 bug，可以感谢并修正；对读者群记录，默认匿名化并只提炼观点。任何高热互动都应写入数据复盘，避免把评论区热度误判为文本有效。
